	#AI编程 在多人协作的软件项目中，我们经常会遇到问题。除了代码报错，最主要的问题，我认为是风格不统一。
	
	我来举几个例子。比如在 Python 里，用 {} 构建字典通常效率更高，但 dict() 在某些场景下可读性更好。有的人喜欢用 """ 来写多行字符串，它很清晰，但用 () 包裹的字符串拼接在一些老代码里也很常见。
	
	还有很多。比如导入包的习惯，有的人喜欢 import os，有的人用 from os import path。导入包的顺序也很乱，有的人按字母排，有的人按系统包、第三方包和自己包的顺序排。
	
	字符串格式化也是。现在 Python 推荐用 f-string，但还是有很多人习惯用 .format() 方法，甚至用更早的 % 符号。
	
	还有变量命名。Python 官方推荐用下划线 snake_case，但总有人习惯用 camelCase 驼峰命名法，特别是从其他语言转过来的同事。
	
	列表推导式也是一个点。有的人喜欢用一行推导式生成新列表，有的人则更喜欢写一个完整的 for 循环，觉得那样更清楚。
	
	最后就是类型提示 Type Hints。有的人认真写，有的人完全不写，有的人只写函数签名，有的人连变量也写。
	
	这些问题单个看都不大，但它们堆在一起，代码库就会显得很乱。每个人阅读别人的代码时，都需要先切换一下大脑里的频道，这很耗费精力，维护起来也累。
	
	前端的 TypeScript 也一样。有的人可能还在用 `==` 而不是 ` === `。
	
	有的人定义了变量却从未使用。更重要的是类型，有的人图方便，到处用 any 类型，或者函数返回了 string 但声明的是 number。
	
	所以很多团队会用像 Black 这样的工具来强制统一代码格式，再用 Flake8 或者前端的 eslint 这样的工具来检查代码风格和捕获不好的写法。而 tsc noEmit 则能严格进行类型检查，确保类型安全。

	#笔记规则
	## 规则
	- 红色： 警示区。标记你**不理解、有疑问或认为有错的地方**。把红色的点当作你的学习目标。
	- 黄色： 我必须记住什么？只标记“事实”和“结论”。边界： **专有名词、核心定义、关键公式、作者的核心论点。**
	- 紫色： 作者是怎么说服我的？只标记结构和关系。边界： **段落的主题句、章节的过渡句、表示转折或因果的强逻辑词。**
	- 蓝色： 我以后能用上什么？只标记表达和文采。边界： **精彩的比喻、精妙的措辞、值得引用的观点。**你标记它，不是因为它核心，而是因为它漂亮。
	- 绿色： 补充区。标记重要的案例、数据、实验结果或次要的支撑细节。它们是对核心观点的有力佐证。
	## 核心原则
	重叠处理规则 (优先)： 结构 (紫) > 核心 (黄) > 欣赏 (蓝)。
