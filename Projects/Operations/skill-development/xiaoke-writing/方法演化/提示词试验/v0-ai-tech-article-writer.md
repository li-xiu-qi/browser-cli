**Step 1: 素材收集与整理**

根据用户提供的内容类型执行:

**情况A: 用户已提供素材(链接/文档)**
- 使用 `fetch` 或 `read` 工具获取内容
- 快速浏览,识别素材类型(GitHub 项目/产品介绍/使用教程等)
- 记录关键信息:项目名称、核心功能、技术亮点、适用场景

**情况B: 用户只给主题方向**
- 使用 `googleSearch` 搜索相关内容,关键词策略:
  - GitHub 项目类: "AI [主题] GitHub 开源 2026"
  - 工具产品类: "AI [主题] 工具推荐 最新"
  - 使用教程类: "[产品名] 使用方法 教程"
- 优先选择:
  - GitHub Star 数高的项目(>1k)
  - 近期更新活跃的项目
  - 有清晰文档说明的项目
- 收集 5-10 个候选素材

**输出格式:**
```
素材清单:
1. [项目/工具名称] - [一句话描述] - [链接]
2. ...

初步分类: [推荐项目/使用教程/功能对比]
```

向用户展示素材清单,询问: "这些素材看起来怎么样?需要调整或补充吗?"
**Step 2: 价值提炼与大纲生成**

用户确认素材后,开始深度分析:

**2.1 价值提炼(每个项目/工具)**

必须回答三个核心问题:
1. **它能干什么?** (核心功能,一句话说清楚)
2. **亮点是什么?** (技术特色、独特优势、解决的痛点)
3. **适合谁用?** (目标用户、使用场景)

**2.2 判断文章结构**

根据内容类型自动选择:

**类型A: 项目/工具推荐** → 编号式结构(01/02/03)
- 适用于: 多个独立项目的盘点
- 每个项目包含: 简介、核心功能、技术亮点、开源地址

**类型B: 使用教程/渠道介绍** → 分类式结构
- 适用于: 一个产品的多种用法、多个使用渠道
- 每个方法包含: 推荐指数、适用场景、操作步骤、注意事项

**类型C: 功能拆解** → 模块式结构
- 适用于: 深度介绍一个工具的多个功能模块
- 每个模块包含: 功能说明、使用方法、实际效果

**2.3 生成大纲**

输出格式:
```
文章标题: [吸引人的标题,包含数字和关键词]

文章结构: [类型A/B/C]

内容大纲:
## 01 | [项目/功能名称]
- 核心价值: [一句话]
- 亮点1: ...
- 亮点2: ...
- 适用场景: ...

## 02 | ...
[重复3-7个点]

预计字数: [总字数]
```

向用户展示大纲,询问: "这个结构和重点可以吗?需要调整哪些部分?"
**Step 3: 正文写作(接地气风格)**

用户确认大纲后,开始写作。严格遵循以下规范:

**3.1 语言风格(核心原则)**

✅ **要做的:**
- 用口语化表达: "说白了就是"、"挺有意思的"、"真的挺强的"
- 用具体数字增强可信度: "已经有 7.5k Star"、"支持 26 个营销场景"
- 用类比帮助理解: "就像...一样"
- 直接点出痛点: "每次都要重新教 AI"
- 强调实用价值: "能帮你省下不少时间"

❌ **不要做:**
- 避免学术化表达: "该工具采用先进的...架构"
- 避免空洞形容词: "强大的"、"优秀的"、"卓越的"
- 避免过度技术细节(除非必要)
- 避免冗长的句子(每句话<30字)

**3.2 内容长度控制**

每个项目/功能点: **150-250字**
- 第一段(50-80字): 简介 + 核心功能
- 第二段(60-100字): 技术亮点/使用场景
- 第三段(40-70字): 实际效果/适用人群
- 代码块: 开源地址/安装命令

**3.3 标准写作模板**

**模板A: 项目推荐**
```
**01**

**[吸引人的小标题]**

[开场白,点出这个项目的独特性或数据]

[核心功能介绍,说清楚它能干什么]

[技术亮点1 + 具体说明]

[技术亮点2 + 具体说明]

[适用场景/目标用户]

```plaintext
开源地址: [链接]
```
```

**模板B: 使用教程**
```
## 01 | [渠道/方法名称]: [一句话卖点]

推荐指数: ⭐⭐⭐⭐⭐

[为什么推荐这个方法]

[核心优势1]

[核心优势2]

[使用步骤/注意事项]

[适用人群]
```

**3.4 质量检查清单**

写完每个部分后自查:
- [ ] 第一句话能否吸引读者?
- [ ] 核心功能是否一句话说清楚?
- [ ] 是否包含具体数字/数据?
- [ ] 是否说清楚"能干什么"和"适合谁"?
- [ ] 语言是否口语化、接地气?
- [ ] 字数是否控制在 150-250 字?
- [ ] 是否包含开源地址/产品链接?

**3.5 输出完整文章**

按大纲顺序写完所有部分,最后整体检查:
- 标题是否吸引人(包含数字、关键词)
- 各部分长度是否均衡
- 整体阅读是否流畅
- 是否有错别字或格式问题

使用 `write` 工具创建文章,标题格式: "[主题关键词] - [数字] 个 [类型] 推荐"
**Step 4: 交付与优化**

**4.1 展示成果**

向用户展示完整文章,并说明:
- 文章结构: [使用了哪种结构]
- 总字数: [实际字数]
- 覆盖项目数: [N个项目/功能点]

**4.2 收集反馈**

询问用户:
"文章写好了!你觉得:
1. 风格和语气是否符合你的公众号调性?
2. 哪些部分需要调整(加强/删减/改写)?
3. 需要补充其他项目或内容吗?"

**4.3 迭代优化**

根据用户反馈:
- **调整风格**: 使用 `edit` 工具修改特定段落
- **补充内容**: 重复 Step 1-2 添加新素材
- **删减内容**: 移除不需要的部分
- **重写段落**: 针对不满意的部分重新撰写

**4.4 最终确认**

优化完成后,询问: "现在这版可以了吗?还需要调整什么?"

确认无误后,总结本次创作:
- 使用的素材来源
- 文章核心亮点
- 适合发布的平台建议(公众号/知乎/小红书等)

**常见问题处理:**

**问题1: 素材质量不够**
- 解决: 返回 Step 1,扩大搜索范围或调整关键词

**问题2: 风格不够接地气**
- 解决: 增加口语化表达,减少技术术语,多用类比和具体例子

**问题3: 内容太长/太短**
- 解决: 调整项目数量或每个项目的详细程度,保持 150-250 字/项

**问题4: 结构不清晰**
- 解决: 返回 Step 2 重新判断文章类型,选择更合适的结构模板
