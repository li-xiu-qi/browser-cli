**第一步：深度理解原文**

仔细阅读用户提供的技术文档（PDF、文本或链接）。在这个阶段，你需要：

1. 识别核心主题和关键概念
2. 找出作者想解决的核心问题
3. 梳理文档的知识层次（基础概念 → 方法论 → 实践案例）
4. 标记出最有价值的信息点（数据、案例、方法、洞察）

**判断标准：** 能用一句话概括"这篇文档在教读者什么"，并能列出3-5个核心知识点。
**第二步：构建逻辑骨架**

基于对原文的理解，设计文章的逻辑结构。遵循以下原则：

**⚠️ 关键警示：先判断文章类型，再决定结构重点**

在设计结构前，必须先回答：
- 原文是什么类型？（概念科普 / 方法指南 / 案例分析 / 工具教程）
- 读者的核心需求是什么？（理解概念 / 学会操作 / 获得灵感 / 解决问题）

**如果原文是「构建指南」或「操作教程」，必须确保：**
1. **技术细节是核心，不能弱化**
   - 文件结构、代码示例、配置规则必须完整保留
   - "怎么做"的权重 ≥ "是什么"和"为什么"
   - 读者看完应该能立即动手操作

2. **回到原文验证章节权重**
   - 查看原文目录，哪个章节最长？那就是最重要的部分
   - 如果原文用 6 页讲"如何构建"，你的文章也必须用相应篇幅覆盖
   - 不要因为"公众号要轻松"就删减核心技术内容

3. **避免被示例文章误导**
   - 如果用户提供了参考文章，注意它可能是"导读"而非"指南"
   - 导读型文章会弱化技术细节，但你要写的是指南型文章
   - 参考风格，但不要照搬结构

**叙述顺序选择：**
- 如果原文是方法论，用"问题 → 解决方案 → 实践指南"结构
- 如果原文是技术解析，用"是什么 → 为什么 → 怎么做"结构
- 如果原文是案例研究，用"背景 → 挑战 → 方案 → 效果"结构
- **如果原文是构建指南，用"概念介绍（20%） → 核心构建步骤（60%） → 进阶技巧（20%）"结构**

**副标题设计：**
- 每个副标题要能独立传递价值（读者看标题就知道这部分讲什么）
- 副标题之间要有明显的递进关系（从浅到深、从理论到实践、从问题到方案）
- 使用具体的、有画面感的表达，避免空泛词汇
- **对于操作类内容，副标题要明确体现"步骤"或"组成部分"**

**示例对比：**

针对概念科普：
❌ "Skill 的基本概念" → ✅ "Skill 到底是什么？让重复的事情不重复做"

针对构建指南：
❌ "写好 description，让 Skill 自动出现" → ✅ "构建 Skill 的核心：从 frontmatter 到完整指令"
❌ "Skill 的设计" → ✅ "一个文件夹，三层加载：Skill 的聪明设计"

**自我检查清单（必做）：**
- [ ] 如果这是操作指南，我是否完整覆盖了"怎么做"？
- [ ] 读者看完能立即动手吗？还是只理解了概念？
- [ ] 我是否因为"易读性"而牺牲了必要的技术细节？
- [ ] 我的结构权重是否匹配原文的章节权重？

**输出：** 列出3-6个副标题，形成文章骨架。每个副标题下简要标注要覆盖的核心内容（1-2句话）。
**第三步：内容填充与人话转译**

在确定的逻辑骨架下，填充具体内容。重点关注：

**人话表达原则：**
1. **用日常场景类比技术概念**
   - 选择普罗大众都能理解的事物（厨房、食谱、地图、工具箱等）
   - 类比后立即回到技术本质，不过度依赖比喻
   - 示例："MCP 是专业厨房，Skill 是食谱。但这个比喻只是辅助理解，真正重要的是..."

2. **拆解复杂概念**
   - 一次只解释一个概念
   - 用"什么意思？""为什么？""怎么做？"这样的口语化过渡
   - 先给结论，再解释原因

3. **避免AI腔调**
   - 不用"赋能""生态""闭环"等buzzword
   - 不用"首先...其次...最后..."这种机械式结构（除非确实是步骤说明）
   - 多用短句，保持节奏感

**⚠️ 技术内容处理原则（针对指南类文章）：**
1. **代码示例必须完整且带注释**
   - 保留原文的代码块，不要简化
   - 用"好例子 vs 坏例子"对比增强理解
   - 在代码后加上"这段代码做了什么"的白话解释

2. **文件结构/配置规则必须可视化**
   - 用代码块展示文件树结构
   - 用表格整理规则清单
   - 关键规则用 ✅ ❌ 符号对比

3. **操作步骤必须分步骤编号**
   - 第一步、第二步、第三步明确标注
   - 每步说明"做什么""为什么""预期结果"
   - 常见错误和解决方案放在每步之后

**内容取舍标准：**
- 保留：直接支撑核心论点的内容、有具体数据/案例的内容、能立即应用的方法
- 删除：重复的概念解释、过于细节的技术参数、与主线无关的延伸讨论
- 压缩：背景信息（一两句话带过）、次要案例（只保留最典型的）
- **特别注意：对于指南类文章，"能立即应用的方法"是核心，不能为了易读性而删减**

**格式要求：**
- 每个段落3-5句话，不超过150字
- 适当使用加粗强调关键术语或核心观点
- 代码示例保留，但要加上"好的例子 vs 不好的例子"对比
- 用"---"分隔不同主题部分
**第四步：开头与结尾设计**

**开头（100-200字）：**
- 用轻松的口吻引入话题（"大家好呀""今天聊聊"）
- 快速说明文章来源和价值（"刷到了 XX 的官方指南""核心干货给你梳理出来"）
- 可选：加一句轻松的调侃或场景化描述（"留着春节无聊的时候看"）

**结尾（可选，50-100字）：**
- 总结核心要点（2-3句话）
- 给出行动建议（"试试看""下次可以这样做"）
- 或者用开放式问题引发思考

**注意：** 不要用"总而言之""综上所述"这种生硬的结尾，保持自然收束。
**第五步：质量检查与优化**

生成初稿后，进行以下检查：

**逻辑检查：**
- 只看副标题，能否理解文章的完整逻辑？
- 每个部分之间是否有自然的承接？
- 是否有跳跃或断层？

**表达检查：**
- 是否有AI腔调的词汇？（替换掉）
- 比喻是否日常化？是否过度使用？
- 是否有过长的段落？（拆分）

**价值检查：**
- 读者读完能获得什么？（知识、方法、工具）
- 是否有可以立即应用的内容？
- 是否删除了所有冗余信息？

**⚠️ 针对指南类文章的额外检查：**
- **可操作性检查**：读者看完能立即动手吗？缺少哪些关键信息？
- **技术完整性检查**：文件结构、配置规则、代码示例是否完整？
- **权重匹配检查**："怎么做"的篇幅是否足够？是否被"是什么""为什么"占用太多空间？

**最终输出：** 一篇3000-5000字的公众号文章，markdown格式，可直接复制发布。
