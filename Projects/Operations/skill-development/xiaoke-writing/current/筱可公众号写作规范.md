# 筱可公众号写作规范

这个文件是 `xiaoke-writing` 当前稳定的写稿入口。

别的 AI 或后续接手的人，如果要沿用这套私号写作方法论写稿、改稿或复盘，先读这份规范，再决定是否继续翻更细的规则。

## 一眼入口

- 要写稿或改稿，先读这份文档
- 如果当前文章工作区里已经有 `反馈调整记录.md`，改稿前先读它

## 发布链路最小口径

如果这次任务的终点不是“改出一版稿”，而是“把稿子送进草稿箱”，默认记住下面 6 条：

1. 发布面默认优先取 `输出-终稿.md`，如果还没有终稿，就取最新的 `输出-vN.md`
2. 标题、小标题骨架、封面图和正文卡文案，最好在发布前就锁住，不要边上传边改结构
3. 图片文件名先正规化，`alt` 只写图片真正想表达的判断，不写“正文卡”“封面图”“文末卡”这类类型词
4. 正文中的外部链接默认使用标准 Markdown 格式 `[文本](URL)`，wenyan 默认会转换成文末脚注。**不再需要手动编写文末"参考来源"章节**，wenyan 会自动整理正文里出现过的链接。如果某个链接需要读者在正文里直接复制，发布时加 `--no-footnote`，并在链接文本里直接包含完整 URL
5. 发布命令默认走 Wenyan Server 模式和 repo-local CLI，不默认用全局 `wenyan`
6. 以返回 `Media ID` 作为进入公众号草稿箱的成功标志

如果已经进入发布阶段，直接跳 [Wenyan服务器发布SOP](../../wechat-writing/current/Wenyan服务器发布SOP.md)，不要只停在这份总规范里。

## 发布主稿 frontmatter 规范

进入发布阶段前，发布主稿（`输出-终稿.md` 或最新 `输出-vN.md`）必须包含以下 frontmatter：

```yaml
---
title: 文章标题
cover: ./assets/封面图.png
digest: 50-100 字文章摘要
author: 筱可AI
share_text: 分享文章时使用的话术，1-2 句话
---
```

字段说明：
- `title`：文章标题，必填
- `cover`：封面图路径。如果不写，Wenyan 会默认拿正文第一张图当封面
- `digest`：文章摘要，用于公众号列表展示（50-100字为宜）
- `author`：作者名，本项目默认统一写 `筱可AI`
- `share_text`：分享文章时使用的话术，1-2 句话，方便后续发朋友圈/社群时直接复制

**封面图位置**：默认把封面图放在正文标题下方、正文第一段之前，让读者进入文章后第一眼看到封面。写法示例：

```markdown
# 文章标题
![封面](./assets/封面图.png)

正文第一段...
```

**图片文件名规范**：`assets/` 目录下的图片文件名避免使用空格，统一用短横线 `-` 连接，例如 `glm-coding-plan.png` 而不是 `glm coding plan.png`。

## 这个项目默认怎么写

- 这是私号项目，对应个人账号、个人品牌和长期私有表达资产
- 开头优先从场景、问题、误判切入，不要一上来堆抽象框架词
- 技术文章要写出机制和例子
- 观点文章要给明确判断
- 方法文章要落到动作、边界或工作流
- 默认先把话说清楚，再追求“像文章”
- 如果是私号思考文、判断型文章、对谈重编排稿，默认优先参考 `写作规范-我的视角-v2.md`
- 如果是对比/汇总/整理类文章且作者有真实认知变化，同样优先参考 `写作规范-我的视角-v2.md`
- 如果这篇思考类文章的价值，一部分来自作者真实认知变化，开头优先从“我原来怎么想 -> 这次意识到什么 -> 判断怎么变了”切入
- 这种“我的视角”是为了补信任感和代入感，不是为了把全文写成情绪流水账
- 思考类文章默认先锁标题和小标题骨架，再启动配图
- `输出.md` 默认视为第 `1` 版初稿，后续版本从 `输出-v2.md` 开始递增
- 如果某一轮已经生成 `输出-vN.md`，后面再改标题、小标题、开头或正文结构，默认继续升新版本，不回写旧版，方便复盘
- `输出-终稿.md` 只有真正定稿时才生成；没有定稿时，不要提前建这个文件
- 如果工作区里已经有 `反馈调整记录.md`，本轮改稿前先回看它，再继续看最新主稿
- 一旦一篇稿子进入第二轮及以上协作，尤其开始连续收到用户对事实、标题、表达、结构的反馈，创建并持续更新 `反馈调整记录.md`
- `反馈调整记录.md` 不是素材池，也不是版本稿；它只记录这轮被指出的问题、为什么有问题、已经怎么改、后面复盘看什么，并作为后续复盘的第一入口

## 结构与排版检查

写公众号文章时，默认检查这几项：

- 不要把整篇写成提纲式分点
- 也不要把整篇压成纯段落
- 分点和段落混排，要让版面密度自然
- 思考类文章的小标题优先写成判断推进，不要写成目录标签
- 如果原始材料是对谈、播客、演讲重编排稿，小标题顺序优先写成“为什么值得看 -> 变化发生在哪 -> 读者接下来该补什么”，具体看 [思考类文章小标题方法](../methods/标题与小标题/思考类文章小标题方法.md)
- 不要大量使用 `不是……而是……`
- 不要写成“最抓人的一句话”“最值得看的”这类评委腔表达；需要表达感受时，把主体写出来
- 如果作者视角本身就是价值来源，不要再退回无主体的客观总评句，例如“最有价值的地方是”“难得就在这里”“把这些判断摊开讲了一遍”
- 不要用“漂亮的反转句”“后背发紧”“反向塑形”这类怪词去代替真正的判断
- 不要堆表情符号
- 不要用蹩脚比喻硬撑气氛
- 不要拿破折号代替真正的逻辑连接
- 重点要靠句子和结构，不要靠装饰性标点

## 目录落点

- 新的想法、灵感、观察，先放 `02_选题库/待选/`
- 确定进入生产后，再开 `03_工作区/[文章标题]/`
- 单篇文章依赖的链接、导出资料和截图，放该文章自己的 `参考资料/`
- 单篇文章在协作过程中收到的密集反馈，放该文章自己的 `反馈调整记录.md`
- 多篇文章共用但还没沉淀成长期资产的资料，放 `07_共享参考资料/`
- 提炼后的观点、案例、金句，放 `08_素材收藏/`
- 项目级可复用视觉资产，放 `09_视觉素材库/`
- 单篇文章专用视觉、专题封面图、临时主题参考图，先放对应文章工作区的 `assets/`

## 视觉使用

- 开头卡用来立主判断和阅读预期
- 正文卡用来托住一句核心判断
- 文末卡只放文末，不插进正文中段
- 图片 `alt` 默认只写图像真正想表达的判断，不写“正文卡”“封面图”“文末卡”这类类型词
- 专题图先按文章工作区管理，确认跨多篇长期复用后，再升级到项目级素材库

### 思考类文章默认怎么配图

- 如果是思考类、判断类、关系分析类文章，默认优先走：
  - 开头 `1` 张 Gemini 意境图
  - 正文 `1 到 2` 张正文卡
  - 结尾 `1` 张固定文末卡
- 这种文章里，Gemini 更适合负责“开头定气质”，不再默认负责多个正文图位
- 正文更适合用卡片托住判断，而不是反复插画
- 只有正文里某个关系真的必须靠画面来成立时，才额外加 Gemini 正文图

### 思考类文章默认怎么发布

- 如果文章属于思考类、方法论、系统分析长文，默认发布主题不是 Wenyan 原始默认样式
- 这类文章默认优先套 `xiaoke-default`
- 发布时优先用 `wenyan-cli-xiaoke-customizations` 这条 CLI，而不是只用通用 CLI
- 具体命令和路径以 [Wenyan服务器发布SOP](../../wechat-writing/current/Wenyan服务器发布SOP.md) 为准

### 对比汇总类文章默认怎么发布

- 如果文章属于对比评测、厂商汇总、数据整理、选型参考类文章，默认优先套 `xiaoke-comparison`
- `xiaoke-comparison` 相比 `xiaoke-default` 更强调信息密度和表格可读性，色调偏科技蓝，装饰感更弱
- 发布命令和路径同样以 [Wenyan服务器发布SOP](../../wechat-writing/current/Wenyan服务器发布SOP.md) 为准

### 文末默认怎么收

- 如果文章正文已经收口完成，文末默认不要只剩一张文末卡
- 更稳的做法是：先补一小段互动 CTA，再接默认文末卡
- 当前项目里复用最多的默认 CTA 是：

```text
以上，既然看到这里了，整理不易，如果觉得不错，随手点个赞、在看、转发三连吧，如果想第一时间收到推送，也可以给我个星标⭐～谢谢你看我的文章，下次再见。
```

- 默认位置：正文结论之后，文末卡之前
- 如果当前文章气质特别克制，可以把这句压成更短版本，但不要整篇没有任何互动收尾

### 教程类文章不要套这个结构

- 如果是教程、操作说明、产品上手、配置排错类文章，默认不要套思考类的配图结构
- 教程类优先考虑：
  - 截图
  - 步骤图
  - 结构说明图
  - 必要时再补卡片

## 发布前检查

- 当前主稿是否明确
- 标题是否已经定下来
- 如果标题还拿不准，是否已经回看 [公众号标题方法论](../methods/标题与小标题/公众号标题方法论.md)
- 如果有多个标题版本，是否已经回看 [公众号标题选择规范](../methods/标题与小标题/公众号标题选择规范.md)
- 如果工作区里已经有 `反馈调整记录.md`，这轮修改和剩余待看点是否已经同步进去
- 图片路径是否可用
- 参考资料是否放对目录
- 是否需要补开头卡、正文卡或文末卡
- 是否触发敏感内容脱敏

如果目标是进公众号草稿箱，默认直接看：

- [Wenyan服务器发布SOP](../../wechat-writing/current/Wenyan服务器发布SOP.md)

## 维护规则

- 如果这份规范被更新了，要顺手同步 `SKILL.md`、`README.md`、项目侧兼容入口，以及必要的 `AGENTS.md`
- 不要把项目真相只留在某个模型的 memory 里
- 目标是让其他 AI 只看仓库内文档，也能知道当前有效口径
- 这同样适用于“反馈如何记录、后面如何复盘”这类协作约定，不要只留在单篇文章里临时出现
- 视角与信任感这类规律，如果已经稳定成高频写法，就回流到 `versions/` 主线或专题补充线；还不稳定的，再继续留在项目侧 `项目期写作沉淀/`
