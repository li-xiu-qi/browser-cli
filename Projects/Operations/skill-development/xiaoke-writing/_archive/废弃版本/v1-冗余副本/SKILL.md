# 筱可写作方法论

三层核心问题：
- **不要替读者想** —— 对象定义权还给读者
- **不要让一个顺的解释抢跑到事实前面** —— 分开已知与推测
- **不要用怪词撑判断** —— 字面能通但读起来发空的表达

---

## 写作顺序

不要反过来：

1. **先判断价值** —— 这篇文章到底卖什么
2. **再设计节奏** —— 怎么让读者继续往下读
3. **最后收口表达和排版** —— 硬检查与格式规范

---

## 第一步：价值判断（最重要）

动笔前回答 4 个问题：

| 问题 | 选项 |
|-----|------|
| **这篇文章最强的价值是什么？** | 教读者怎么做 / 帮读者理解一件事 / 展示一个案例为什么成立 / 提供一份可收藏的资源清单 |
| **最适合写给谁？** | 新手 / 普通团队 / 开发者 / 管理者 / 对某类问题有明确需求的人 |
| **最适合在哪个场景被打开？** | 想安装上手 / 想理解新概念 / 想判断值不值得跟 / 想知道别人怎么做成的 / 想快速拿走一套方法 |
| **读者看完后最应该带走什么？** | 一个方法 / 一个判断 / 一个步骤清单 / 一个案例框架 / 一套资源入口 |

**一句话测试**：能不能用一句话说清，这篇文章最值得卖给读者的是什么？

示例：
- ❌ "介绍 Anthropic Skills 博客"
- ✅ "普通团队怎么让 Agent 真正进入工作"

---

## 第二步：文章类型与骨架

### 类型 A：教程指南
特征：核心在"怎么做"，看完能直接动手

骨架：
```
是什么 -> 为什么 -> 怎么做
概念介绍(20%) -> 核心步骤(60%) -> 进阶技巧(20%)
```

### 类型 B：案例方案
特征：核心在"这套东西为什么成立"，拿走的是方案理解

骨架：
```
背景 -> 问题 -> 方案 -> 为什么有效 -> 怎么借鉴
结果冲击 -> 常规做法不够 -> 系统结构 -> 落地启发
```

### 类型 C：对比/汇总
特征：核心在"横向比较后给出判断"

骨架：
```
为什么要比 -> 维度设计 -> 逐个对比 -> 整体判断 -> 适用建议
```

## 第三步：表达硬检查

### 检查 1：不要替读者想

**禁止句式**：
- "你可能觉得..."
- "读者也许会问..."
- "大家第一反应是..."

**正确做法**：直接给事实，让读者自己判断。

示例：
- ❌ "你可能觉得这个功能很简单，但其实..."
- ✅ "这个功能表面只有一行配置，实际涉及三个系统的状态同步。"

### 检查 2：不要让模型先补一个可解释对象

**问题**：模型在输出时，会先"脑补"一个对象，然后围绕它展开，但这个对象在原文里并不存在。

**信号词**："这"、"它"、"其"、"该" 后面跟着的其实是模型自己补的。

示例：
- ❌ "GPT-4 发布了新功能，**其**优化了推理能力。" （"其"指的是什么？功能还是模型？）
- ✅ "GPT-4 发布了新功能，**这项功能**优化了推理能力。"

**规则**：
- 用"这""它"前，先确认前文有没有明确对应
- 如果对应模糊，换成具体名词

### 检查 3：怪词识别与替换

**怪词特征**：字面能通，读起来却发空发飘。

| 怪词类型 | 示例 | 替换方向 |
|---------|------|---------|
| 抽象大词 | "能力"、"逻辑"、"维度"、"层面"、"视角" | 换成具体动作或结果 |
| 空泛修饰 | "有效"、"合理"、"科学"、"充分" | 说明为什么有效、怎么合理 |
| 假性力度 | "最抓人的"、"本质上是"、"核心在于" | 换成具体判断或删掉 |

示例：
- ❌ "这个设计的**能力**很强"
- ✅ "这个设计能在一秒内处理 1000 条记录"

- ❌ "**本质上**是状态管理问题"
- ✅ "表面看是渲染问题，实际是状态没同步" （或干脆删掉"本质上"）

### 检查 4：机械对比句

**禁止**：
- "不是 A，而是 B"（除非你真的在纠正常见误解）
- "与其说 A，不如说 B"
- 没有明确对象的"最..."

示例：
- ❌ "这不是一个工具问题，而是一个思维问题"
- ✅ "工具只能解决执行环节，真正的瓶颈在判断标准"

### 检查 5：评委腔 / 空心对比句

**问题**：用"最抓人的""值得关注的"这种评委视角，但没说清楚对谁、在什么场景下成立。

**规则**：
- 删掉所有"最..."，换成具体对谁、在什么场景下
- 删掉"值得..."，直接说为什么重要

示例：
- ❌ "最抓人的是第三点"
- ✅ "如果你只有 5 分钟，直接看第三点：它决定了前两点能不能落地"

### 检查 6：分点列表控制

- 超过 10 个列表项 → 考虑合并或转成段落
- 列表里套列表 → 拆成多个小列表，每个配一句话过渡

### 检查 7：表情符号

- 全文不超过 2 个 Emoji
- 不要连续多个 Emoji

### 检查 8：AI 味信号词

**高频 AI 味词汇**：
- "让我们"、"不妨"、"试想一下"
- "值得注意的是"、"需要强调的是"
- "不难发现"、"显而易见"

**处理**：全部删掉或换成直接陈述。

## 第四步：格式与排版

### 标题层级
- 控制在 3 级以内
- 不要跳级（# 下面是 ###）

### 段落长度
- 移动端优先，每段尽量短
- 超过 3 行的段落考虑拆分

### 引用块
- 仅用于：金句摘录、外部定义、重要强调
- 不要用于：普通内容缩进、多行短句排版

### 代码块
- 仅用于：真实代码、配置、命令行
- 不要用于：普通文本排版

### 加粗
- 控制在 5 处/千字以内
- 不要连续加粗

## 完整工作流

```
起稿前：
    回答 4 个价值判断问题
    确定文章类型和骨架

写作中：
    按骨架填充内容
    每段自检：有没有替读者想？有没有怪词？

初稿后：
    逐条过表达硬检查
    修正所有 P0 级问题

定稿前：
    格式与排版检查
    标题层级、段落长度、加粗数量
```
