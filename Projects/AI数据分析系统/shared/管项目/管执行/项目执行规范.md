# AIASys 项目执行规范

> 项目执行阶段的标准动作指南

---

## 一、执行管理六步法概览

项目管理执行阶段的六个关键步骤：

| 步骤 | 核心动作 | 频率 | 输出物 |
|------|----------|------|--------|
| 一、开启动会 | 对齐目标、分工、机制 | 项目启动时 | 会议纪要 |
| 二、跟进度 | 同步进展、识别阻塞 | 每日/每周 | 进度更新 |
| 三、控质量 | 检查交付物、把控标准 | 关键节点 | 验收报告 |
| 四、处变更 | 评估影响、决策调整 | 变更发生时 | 变更记录 |
| 五、同老板 | 同步风险、申请资源 | 每周/风险时 | 周报/专项汇报 |
| 六、管团队 | 激励士气、解决冲突 | 持续 | 团队状态评估 |

## 三、步骤二：跟进度

### 日常跟踪机制

**每日（开发团队内部）**：
- 站会：昨天完成/今天计划/阻塞问题（15分钟）
- Git commit 提交代码
- implementation-status 更新进度

**每周五（向A汇报）**：
- 发送周报（基于执行状态文档）
- 包含：进度、风险、下周计划、需要支持

### 进度可视化

| 工具 | 用途 | 查看权限 |
|------|------|----------|
| Git commit 历史 | 代码产出 | 全团队 |
| implementation-status | 功能状态 | 全团队 |
| 周报 | 整体进展 | A + 全团队 |

### 阻塞升级规则

| 阻塞时间 | 处理动作 | 升级对象 |
|----------|----------|----------|
| < 4小时 | 自主解决 | - |
| 4小时-1天 | 组内求助 | 团队内部 |
| 1-2天 | 项目负责人介入 | 筱可 |
| > 2天 | 上升A决策 | A |

## 四、步骤三：控质量

### 质量检查节点

每个里程碑必须进行质量检查，详见 [[质量检查清单]]

### 代码质量规范

**后端（Python）**：
```bash
# 提交前必须执行
python -m black --line-length 79 .
python -m isort --line-length 79 --profile black .
python -m mypy --ignore-missing-imports app/
```

**前端（TypeScript/React）**：
```bash
# 提交前必须执行
npx tsc --noEmit
npm run lint
```

### PR 规范

- 每个 PR 必须有描述：做了什么、为什么、如何测试
- 每个 PR 必须关联 issue/任务
- 每个 PR 必须经过 code review 才能合并

## 五、步骤四：处变更

### 变更流程

```
收到变更请求
    ↓
评估影响（工时/排期/其他功能）
    ↓
提供选项给A决策
    ↓
A确认后更新计划
    ↓
同步全团队执行
```

### 变更记录模板

|------|----------|--------|------|-------|----------|

## 六、步骤五：同老板

### 周报模板

```markdown
## AIASys 项目周报（第X周）

### 本周完成
- [ ] 里程碑X：完成度X%
- [ ] 功能A：已完成

### 当前风险
- 风险1：状态（正常/关注/已触发）
- 风险2：状态

### 下周计划
- 里程碑Y：目标完成度
- 功能B：启动开发

### 需要支持
- [ ] 需要A决策/支持的事项
```

### 何时必须找A

- 触发风险预案时
- 需要额外资源时
- deadline需要调整时
- 需求范围变更时

## 七、步骤六：管团队

### 团队健康检查

每周评估：
- 工作负荷是否均衡？
- 是否有人员冲突？
- 士气状态如何？

### 激励机制

- 里程碑达成：团队庆祝
- 个人突出贡献：周报中表扬
- 连续加班：申请调休/团队聚餐
