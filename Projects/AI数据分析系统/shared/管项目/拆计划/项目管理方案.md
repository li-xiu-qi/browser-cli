# AIASys 智能制造分析平台项目管理方案

**项目负责人**：筱可  
**批准人**：A  
**项目级别**：B级多人项目（2-5人协作）  
**技术架构**：Host-Worker 双 Agent 架构，React + FastAPI + Jupyter  
**方案目标**：对齐业务价值、明确技术目标、申请资源、建立机制

## 一、项目背景

### 1.1 项目起源

**项目定位**：
AIASys 是面向工业制造的智能数据分析平台，采用 Host-Worker 双 Agent 架构。

**核心功能**：
- **设备数据分析** — 振动分析、频谱分析、趋势分析、异常检测
- **智能故障诊断** — 故障检测、分类、严重性评估、维护建议  
- **参数预测优化** — 时间序列预测、参数辨识预测、参数优化
- **智能对话** — 基于大语言模型的自然语言交互
- **文件管理** — 支持 CSV、XLSX、PDF、DOCX 等多种格式
- **Notebook 回放** — 分析过程可追溯、可复现

**支持设备**：轧机、电机、减速器、轴承

**技术栈**：
| 层级 | 技术 |
|------|------|
| 后端 | Python 3.10+, FastAPI, smolagents, Jupyter, Celery, uv |
| 前端 | React 19, TypeScript 5.9, Vite 7, Tailwind CSS 4 |
| UI 组件 | shadcn/ui, Radix UI, Lucide React |
| 数据库 | Supabase (PostgreSQL) |
| 消息队列 | Redis + Celery |
| 容器化 | Docker, Docker Compose |

## 四、项目管理方案

### 4.1 拆事 —— 确认节点

| 阶段 | 时间 | 确认内容 | 输出物 | 关键人 |
|------|------|----------|--------|--------|
| 1 | W1 | 需求与方案确认 | PRD + 技术方案 | 批准者 + 项目负责人 |
| 2 | W2-3 | 后端架构完成 | 接口文档 + 核心模块 | 项目负责人 |
| 3 | W3-4 | 前端框架确认 | 原型 + 页面框架 | 前端 + 项目负责人 |
| 4 | W5-6 | 核心功能联调 | 可运行版本 | 全团队 |
| 5 | W7 | 内测与优化 | 测试报告 | 项目负责人 |
| 6 | W8 | 终版交付 | 上线版本 | 批准者 |

> 核心原则：确认完A再确认B，避免返工。

### 4.2 拆人 —— RASIC分工

| 任务 | R（负责人） | A（批准人） | S（支持方） |
|------|-------------|-------------|-------------|
| **整体项目** | 项目负责人 | 批准者 | 前端实习生、算法实习生 |
| **后端开发** | 项目负责人 | 项目负责人 | - |
| **前端开发** | 前端实习生 | 项目负责人 | 项目负责人（backup） |
| **算法开发** | 项目负责人 | 项目负责人 | 算法实习生 |
| **项目管理** | 项目负责人 | 批准者 | - |
| **需求确认** | 项目负责人 | 批准者 | - |

**关键说明**：
- R有且仅有一个，确保责任清晰
- A在关键节点（需求确认、终版交付）必须参与
- 若前端实习生无法继续，项目负责人作为backup接手

### 4.3 拆时间 —— 排期与Buffer

**总周期**：8周（含30% Buffer应对意外）

| 周次 | 关键任务 | 里程碑 | Buffer预留 |
|------|----------|--------|------------|
| W1 | 需求确认 + 技术方案 | 方案确认 | 2天 |
| W2-3 | 后端核心开发 | 接口完成 | 3天 |
| W3-4 | 前端开发 | 页面框架 | 3天 |
| W5-6 | 联调测试 | 可运行版本 | 4天 |
| W7 | 内测优化 | 测试通过 | 2天 |
| W8 | 交付上线 | 项目完成 | 2天 |

> 若前端断档，启用Plan B：削减前端功能，优先保证核心功能可用。

## 六、沟通机制

| 形式 | 频率 | 内容 | 形式 | 支撑工具 |
|------|------|------|------|----------|
| **周报** | 每周五 | 进度、风险、下周计划 | 飞书文档 | 执行状态跟踪文档 |
| **例会** | 每周一 | 对齐优先级、解决问题 | 15-30分钟 | 执行状态跟踪文档 |
| **紧急沟通** | 随时 | 重大风险、需要决策 | 即时 | - |

**执行状态跟踪机制**：

项目已建立 **implementation-status** 功能实现跟踪体系，实时记录：
- 各模块（前端/后端）功能开发状态（⬜ 未开始 /  进行中 /  已完成）
- 当前阻塞问题与责任人
- 风险状态与预案触发条件
- Changelog 变更摘要与验收报告（completion-records）分离管理

该文档作为周报和例会的数据源，确保信息透明、可追溯、减少沟通成本。

### 6.2 技术驱动的双轨跟踪

除手动更新的执行状态文档外，项目采用 **Git 工作流自动化跟踪** 机制，通过代码提交（commit）和合并请求（PR）实时反映开发进度：

| 跟踪维度 | 数据来源 | 反映指标 | 更新频率 |
|----------|----------|----------|----------|
| **代码活跃度** | Git commit 记录 | 各模块开发投入、个人产出 | 实时 |
| **功能完成度** | PR 合并状态 | 功能开发完成、待 review、已上线 | 实时 |
| **协作效率** | PR 评论与反馈周期 | 代码 review 速度、阻塞情况 | 实时 |
| **质量指标** | PR 大小、冲突频率 | 任务拆解合理性、协作摩擦 | 每周汇总 |

**Git 工作流规范**：

```
功能开发 → 创建分支 → 提交 commit → 发起 PR → Code Review → 合并主干
              ↑___________________________________________↓
                          进度自动同步到执行状态文档
```

**Commit 规范（用于进度识别）**：
- `feat(前端): 完成登录页面UI` — 可关联到前端模块进度
- `fix(后端): 修复API响应延迟` — 可关联到后端模块问题
- `docs(算法): 更新模型训练说明` — 可关联到文档进度

**PR 与任务关联**：
- 每个 PR 对应一个具体功能点或任务
- PR 标题格式：`[模块] 任务描述`（如 `[前端] 实现数据可视化图表`）
- PR 描述中注明关联的执行状态文档条目

**进度可视化**：
- 通过 Git 看板（如 GitHub/GitLab Projects）直观展示任务状态
- 自动生成每日/每周代码贡献统计，补充执行状态文档

**优势**：
- 减少手动汇报负担，代码即进度
- 客观数据支撑，避免"我觉得进度正常"的主观偏差
- 及时发现阻塞（如某人长时间无 commit、PR 久未 review）

---

### 6.3 双轨跟踪体系

| 方式 | 适用场景 | 更新频率 | 责任人 |
|------|----------|----------|--------|
| **执行状态文档** | 里程碑达成、风险变化、资源调整 | 事件驱动 + 每周汇总 | 项目负责人 |
| **Git 工作流** | 日常开发进度、代码产出、功能完成 | 实时自动 | 全体开发 |

**整合方式**：
- Git 数据作为执行状态文档的输入源之一
- 周报结合两者：Git 统计展示产出，执行状态文档展示业务进展
- 例会基于 Git 看板快速过进度，聚焦阻塞问题
