## 第二步：拆里程碑

### 里程碑总览

| 阶段 | 里程碑 | 交付成果 | 验收人 | 时间 |
|------|--------|----------|--------|------|
| 1 | 需求确认 | PRD + 技术方案 | A + 项目负责人 | W1 |
| 2 | 后端完成 | API + 文档 | 项目负责人 | W3 |
| 3 | 前端完成 | 页面框架 | 项目负责人 | W4 |
| 4 | 功能联调 | 可运行版本 | 项目负责人 + A | W6 |
| 5 | 内测通过 | 测试报告 | 项目负责人 | W7 |
| 6 | 终版交付 | 上线版本 | A | W8 |

### 里程碑定义原则

- **必经之路**：不做完就无法进入下一阶段
- **有交付成果**：不是"做了什么"，而是"交付了什么"
- **可检查可评审**：能明确判断是否完成
- **前后依赖**：有清晰的先后顺序

## 第四步：拆角色（RASIC分工）

详见 [[项目管理方案#RASIC分工]]

### 核心分工

| 任务 | R（负责人） | A（批准人） | S（支持方） |
|------|-------------|-------------|-------------|
| 整体项目 | 项目负责人 | A | 前端、算法 |
| 后端开发 | 项目负责人 | - | - |
| 前端开发 | 前端实习生 | 项目负责人 | 项目负责人（backup） |
| 算法开发 | 项目负责人 | - | 算法实习生 |

### 分工原则

- **R 有且仅有一个**：避免多头领导
- **A 在关键节点**：需求确认、终版交付必须 A 参与
- **S 投入时间**：不是只给建议，是一起干活

## 第五步：拆排期

### 总排期

**周期**：8周（含 30% Buffer）

| 周次 | 里程碑 | 关键任务 | Buffer |
|------|--------|----------|--------|
| W1 | 需求确认 | PRD、技术方案、原型 | 2天 |
| W2-3 | 后端完成 | API开发、单元测试 | 3天 |
| W3-4 | 前端完成 | 页面框架、组件库 | 3天 |
| W5-6 | 功能联调 | 集成测试、Bug修复 | 4天 |
| W7 | 内测通过 | 内测、优化、文档 | 2天 |
| W8 | 终版交付 | 上线、验收、交接 | 2天 |

### 关键路径

```
需求确认 → 后端开发 → 功能联调 → 内测 → 交付
              ↓
           前端开发 → 功能联调
```

**关键路径时长**：__________ 天（决定最短完成时间）
