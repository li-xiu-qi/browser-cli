# 扩展技术：分库分表（Sharding）详解

**技术类型**: 数据库水平扩展  
**适用场景**: 单表数据量超过千万级  
**难度**: ⭐⭐⭐⭐ 高

### 1.2 一句话解释

> **分库分表 = 把一张大表拆成多张小的子表，分散到多个数据库，按规则路由查询**

## 二、分片策略

### 2.1 常用分片算法

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     分片算法对比                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   1. Hash 取模分片（最常用）                                                  │
│   ─────────────────────────                                                   │
│                                                                              │
│   算法：shard = hash(shard_key) % shard_count                                │
│                                                                              │
│   示例：tenant_id = "abc123"                                                 │
│         hash("abc123") = 1234567                                             │
│         1234567 % 4 = 3                                                      │
│         → 路由到 shard-3                                                     │
│                                                                              │
│   优点：                                                                     │
│   • 数据分布均匀                                                             │
│   • 简单高效                                                                 │
│                                                                              │
│   缺点：                                                                     │
│   • 扩容麻烦（需要迁移数据）                                                  │
│   • 范围查询需要扫描所有分片                                                  │
│                                                                              │
│──────────────────────────────────────────────────────────────────────────────│
│                                                                              │
│   2. 范围分片                                                                │
│   ────────────                                                               │
│                                                                              │
│   算法：按时间或ID范围                                                       │
│                                                                              │
│   示例：document 表按时间                                                     │
│         2023-01 ~ 2023-03 → shard-0                                          │
│         2023-04 ~ 2023-06 → shard-1                                          │
│         2023-07 ~ 2023-09 → shard-2                                          │
│                                                                              │
│   优点：                                                                     │
│   • 天然支持范围查询                                                         │
│   • 扩容简单（新增时间范围即可）                                              │
│                                                                              │
│   缺点：                                                                     │
│   • 数据可能不均匀（热点问题）                                                │
│   • 旧数据不再写入，分片利用率降低                                            │
│                                                                              │
│──────────────────────────────────────────────────────────────────────────────│
│                                                                              │
│   3. 一致性 Hash（推荐用于缓存）                                              │
│   ─────────────────────────                                                   │
│                                                                              │
│   算法：将 key 和节点都映射到一个环上                                         │
│                                                                              │
│   图示：                                                                     │
│          0°                                                                  │
│          │                                                                   │
│    270° ─┼─ 90°     node-A (0°)                                             │
│          │           node-B (120°)                                           │
│         180°         node-C (240°)                                           │
│                                                                              │
│   key 落到环上，顺时针找最近的 node                                           │
│                                                                              │
│   优点：                                                                     │
│   • 扩容时只影响相邻节点（迁移数据少）                                        │
│   • 虚拟节点解决数据倾斜                                                      │
│                                                                              │
│   缺点：                                                                     │
│   • 实现复杂                                                                 │
│   • 元数据管理成本高                                                          │
│                                                                              │
│──────────────────────────────────────────────────────────────────────────────│
│                                                                              │
│   4. 分片表（ lookup 表）                                                     │
│   ───────────────────                                                         │
│                                                                              │
│   算法：用一个表记录每个 key 对应的分片                                        │
│                                                                              │
│   示例：                                                                     │
│   ┌───────────────┬──────────┐                                               │
│   │  tenant_id    │  shard   │                                               │
│   ├───────────────┼──────────┤                                               │
│   │  tenant_a     │  shard-0 │                                               │
│   │  tenant_b     │  shard-1 │                                               │
│   │  tenant_c     │  shard-2 │                                               │
│   └───────────────┴──────────┘                                               │
│                                                                              │
│   优点：                                                                     │
│   • 灵活，可以任意调整                                                       │
│   • 支持动态迁移                                                             │
│                                                                              │
│   缺点：                                                                     │
│   • 需要查表，性能稍差                                                       │
│   • 元数据表本身可能成为瓶颈                                                  │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 RAGFlow 推荐策略

```python
# 对于 RAGFlow，推荐 Hash + 分片键 tenant_id

# 原因：
# 1. RAGFlow 是多租户系统，天然按 tenant 隔离
# 2. 查询基本都带 tenant_id，路由简单
# 3. 租户数据分布相对均匀

# 分片键选择：
# 主键：tenant_id（必选）
# 辅助：user_id（可选，用于用户相关表）

# 路由示例：
class ShardingRouter:
    SHARD_COUNT = 16  # 初始16个分片
    
    @staticmethod
    def get_shard(tenant_id: str) -> int:
        """获取分片编号"""
        # 对 tenant_id 取 hash
        hash_val = hashlib.md5(tenant_id.encode()).hexdigest()
        # 取前8位转整数
        int_val = int(hash_val[:8], 16)
        # 取模
        return int_val % ShardingRouter.SHARD_COUNT
    
    @staticmethod
    def get_db_config(tenant_id: str) -> dict:
        """获取数据库配置"""
        shard = ShardingRouter.get_shard(tenant_id)
        
        # 分片映射到物理库（4个物理库，每个4个逻辑分片）
        db_index = shard // 4
        
        return {
            'host': f'mysql-shard-{db_index}',
            'database': f'ragflow_shard_{shard}',
            'shard': shard
        }
```

```yaml
    broadcastTables:
      - llm_config
      - system_config

    shardingAlgorithms:
      db_hash_mod:
        type: INLINE
        props:
```

### 3.2 Python 代码层实现

```python
# api/db/sharding_router.py

import hashlib
from peewee import MySQLDatabase

class ShardingManager:
    """分片管理器"""
    
    # 分片配置
    SHARD_DB_COUNT = 4       # 4 个物理库
    SHARD_TABLE_COUNT = 16   # 16 个逻辑表
    
    # 数据源缓存
    _db_pool = {}
    
    @staticmethod
    def get_shard_index(tenant_id: str) -> int:
        """根据 tenant_id 计算分片索引"""
        hash_val = hashlib.md5(tenant_id.encode()).hexdigest()
        int_val = int(hash_val[:8], 16)
        return int_val % ShardingManager.SHARD_TABLE_COUNT
    
    @staticmethod
    def get_db_index(shard_index: int) -> int:
        """根据分片索引获取数据库索引"""
        return shard_index % ShardingManager.SHARD_DB_COUNT
    
    @staticmethod
    def get_db(tenant_id: str) -> MySQLDatabase:
        """获取租户对应的数据库连接"""
        shard = ShardingManager.get_shard_index(tenant_id)
        db_index = ShardingManager.get_db_index(shard)
        
        cache_key = f'db_{db_index}'
        
        if cache_key not in ShardingManager._db_pool:
            # 创建新连接
            db = MySQLDatabase(
                f'ragflow_shard_{db_index}',
                host=f'mysql-shard-{db_index}',
                port=3306,
                user='root',
                password='password'
            )
            ShardingManager._db_pool[cache_key] = db
        
        return ShardingManager._db_pool[cache_key]
    
    @staticmethod
    def get_table_name(base_name: str, tenant_id: str) -> str:
        """获取分片表名"""
        shard = ShardingManager.get_shard_index(tenant_id)
        return f'{base_name}_{shard}'


# Model 层改造
class ShardedDocument:
    """分片文档模型"""
    
    @staticmethod
    def _get_model(tenant_id: str):
        """动态获取 Model 类"""
        table_name = ShardingManager.get_table_name('document', tenant_id)
        db = ShardingManager.get_db(tenant_id)
        
        # 动态创建 Model 类
        class Meta:
            database = db
            table_name = table_name
        
        attrs = {
            'id': CharField(max_length=32, primary_key=True),
            'tenant_id': CharField(max_length=32, index=True),
            'name': CharField(max_length=255),
            'Meta': Meta
        }
        
        return type(f'Document_{table_name}', (Model,), attrs)
    
    @classmethod
    def create(cls, tenant_id: str, **kwargs):
        """创建文档"""
        Model = cls._get_model(tenant_id)
        return Model.create(tenant_id=tenant_id, **kwargs)
    
    @classmethod
    def get_by_id(cls, tenant_id: str, doc_id: str):
        """查询文档"""
        Model = cls._get_model(tenant_id)
        return Model.select().where(
            (Model.id == doc_id) & 
            (Model.tenant_id == tenant_id)
        ).first()
    
    @classmethod
    def list_by_tenant(cls, tenant_id: str, page=1, size=20):
        """列表查询"""
        Model = cls._get_model(tenant_id)
        return Model.select().where(
            Model.tenant_id == tenant_id
        ).paginate(page, size)
```

```python
        """
        # 按分片分组租户
        shard_groups = {}
        for tid in tenant_ids:
            shard = ShardingManager.get_shard_index(tid)
            shard_groups.setdefault(shard, []).append(tid)
        
        # 并行查询各分片
        tasks = []
        for shard, tids in shard_groups.items():
            task = CrossShardQueryHandler._count_shard(shard, tids)
            tasks.append(task)
        
        results = await asyncio.gather(*tasks)
```

### 4.1 不停机迁移

```python
# migration/dual_write_migration.py

"""
不停机迁移方案：双写 + 增量同步
"""

class DualWriteMigration:
    """双写迁移服务"""
    
    def __init__(self):
        self.old_db = MySQLDatabase('ragflow_old')
        self.new_shards = [
            MySQLDatabase(f'ragflow_shard_{i}')
            for i in range(4)
        ]
        self.migration_status = {}
    
    async def start_migration(self, tenant_id: str):
        """
        开始迁移单个租户
        
        阶段1：双写
        阶段2：全量迁移

        # 阶段4：切换读（灰度）
        await self._switch_read(tenant_id, shard)
        
        # 阶段5：停止双写
        await self._disable_dual_write(tenant_id, shard)
        
        print(f'Tenant {tenant_id} migrated to shard {shard}')
```

## 五、优缺点

### 5.1 优点

| 优点 | 说明 |
|------|------|
| **水平扩展** | 加机器即可增加存储和性能 |
| **性能提升** | 单表数据量减少，查询更快 |
| **隔离性** | 租户数据分散，互不影响 |
| **并发提升** | 多个数据库并行处理 |

### 5.2 缺点与挑战

| 缺点 | 解决方案 |
|------|----------|
| **跨分片查询困难** | 避免跨分片，或异步聚合 |
| **分布式事务复杂** | 使用 Saga 模式，最终一致 |
| **数据迁移痛苦** | 双写方案，灰度迁移 |
| **运维复杂度** | 使用 ShardingSphere 简化 |
| **主键生成** | 使用雪花算法 |
