# 扩展技术：数据库连接池详解

**技术类型**: 数据库连接优化  
**适用场景**: 高并发数据库访问  
**难度**: ⭐⭐ 中等

### 1.2 一句话解释

> **连接池 = 预先创建好数据库连接，用的时候借，用完归还，避免频繁创建连接的开销**

## 二、为什么需要连接池？

### 2.1 创建连接的成本

```
创建一次数据库连接的耗时分解：

1. TCP 三次握手建立连接                    ~10-30ms
   SYN ──► SYN+ACK ──► ACK

2. TLS/SSL 握手（如果使用 SSL）             ~20-50ms

3. MySQL 协议握手                          ~5-10ms
   - 服务端发送握手初始化包
   - 客户端发送认证信息
   - 服务端验证并发送 OK 包

4. 应用层准备（线程栈分配等）               ~5-10ms

总计：~50-100ms

执行一条简单 SQL：~5-10ms

结论：创建连接比执行 SQL 慢 5-10 倍！
```

### 2.2 RAGFlow 场景

```python

    # 有连接池：+1ms
    
    return response

# 总计：
# 无连接池：200ms+（4次连接）
# 有连接池：10ms+（复用连接）
```

## 三、连接池核心参数

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     连接池参数详解                                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   核心参数                                                                   │
│   ──────────                                                               │
│                                                                              │
│   1. 最小连接数（min_connections / min_idle）                                │
│      ──────────────────────────────────────                                  │
│      含义：连接池保持的最小空闲连接数                                         │
│      建议：5-10 个                                                           │
│      作用：低峰期也保持一定连接，避免频繁创建                                 │
│                                                                              │
│   2. 最大连接数（max_connections / max_size）                                │
│      ──────────────────────────────────────                                  │
│      含义：连接池允许的最大连接数                                             │
│      建议：20-50 个（根据并发量调整）                                         │
│      注意：不能超过 MySQL 的 max_connections                                  │
│            每个连接占用内存（~10MB），太多会耗光内存                           │
│                                                                              │
│   3. 连接超时（connection_timeout）                                          │
│      ──────────────────────────────                                          │
│      含义：从连接池获取连接的最大等待时间                                     │
│      建议：30 秒                                                             │
│      场景：连接池满了，新请求等待可用连接                                     │
│                                                                              │
│   4. 空闲超时（idle_timeout / max_idle_time）                                │
│      ──────────────────────────────────────                                  │
│      含义：连接在池中最长空闲时间，超时关闭                                   │
│      建议：600 秒（10分钟）                                                   │
│      作用：回收长期不用的连接，释放资源                                       │
│                                                                              │
│   5. 连接最大生命周期（max_lifetime）                                        │
│      ────────────────────────────────                                        │
│      含义：连接的最长使用时间，强制刷新                                       │
│      建议：1800 秒（30分钟）                                                  │
│      作用：防止连接因长期存活而失效（如防火墙切断）                            │
│                                                                              │
│   6. 健康检查（health_check_interval）                                       │
│      ────────────────────────────────                                        │
│      含义：检查连接是否可用的间隔                                             │
│      建议：30 秒                                                             │
│      作用：及时剔除失效连接                                                   │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

# 使用连接池
async def query_with_pool(pool, sql, params=None):
    """使用连接池执行查询"""
    # 从池中获取连接
    async with pool.acquire() as conn:
        # 创建游标
        async with conn.cursor() as cur:
            await cur.execute(sql, params)
            result = await cur.fetchall()
            return result
    # 连接自动归还到池中

### 4.4 连接池监控

```python
# common/db_monitor.py

import asyncio
import logging
from datetime import datetime

class ConnectionPoolMonitor:
    """连接池监控器"""
    
    def __init__(self, db_pool, check_interval=30):
        self.db_pool = db_pool
        self.check_interval = check_interval
        self.running = False
    
    async def start_monitoring(self):
        """开始监控"""
        self.running = True
        while self.running:
            await self._check_pool_health()
            await asyncio.sleep(self.check_interval)
    
    async def _check_pool_health(self):
        """检查连接池健康状态"""
        try:
            status = await self._get_pool_status()
            
            # 记录指标
            logging.info(f"DB Pool Status: {status}")
            
            # 告警检查
            if status['usage_ratio'] > 0.8:
                logging.warning(
                    f"DB Pool usage high: {status['usage_ratio']:.2%}. "
                    f"Consider increasing max_connections."
                )
            
            if status['wait_queue'] > 10:
                logging.error(
                    f"DB Pool wait queue long: {status['wait_queue']}. "
                    f"Requests are being blocked!"
                )
            
            if status['available'] == 0:
                logging.error(
                    "DB Pool exhausted! No available connections."
                )
        
        except Exception as e:
            logging.error(f"Failed to check pool health: {e}")
    
    async def _get_pool_status(self):
        """获取连接池状态"""
        max_conn = self.db_pool.max_connections
        available = len(self.db_pool._connections)
        in_use = len(self.db_pool._in_use) if hasattr(self.db_pool, '_in_use') else 0
        wait_queue = len(self.db_pool._waiters) if hasattr(self.db_pool, '_waiters') else 0
        
        return {
            'timestamp': datetime.now().isoformat(),
            'max_connections': max_conn,
            'available': available,
            'in_use': in_use,
            'wait_queue': wait_queue,
            'usage_ratio': in_use / max_conn if max_conn > 0 else 0,
        }
    
    def stop(self):
        """停止监控"""
        self.running = False


# 集成到应用
async def init_monitoring():
    """初始化连接池监控"""
    from api.db.db_models import db
    
    monitor = ConnectionPoolMonitor(db, check_interval=30)
    asyncio.create_task(monitor.start_monitoring())
    
    return monitor
```

### 5.1 连接数计算

"""
如何确定合适的连接池大小？

公式：
连接数 = ((核心数 × 2) + 有效磁盘数)

示例：
- 服务器：4 核 CPU
- 数据库磁盘：SSD 1 块
- 连接数 = (4 × 2) + 1 = 9

实际建议：
- Web 应用：20-50 连接
- 批处理：5-10 连接
- 微服务：10-20 连接（每个服务）

注意事项：
1. 连接数不是越多越好，太多会增加数据库负担
2. 多个服务共用数据库时，分配好各自配额
3. 监控实际使用情况，动态调整
"""

## 六、常见问题

### 6.1 连接池耗尽

```
症状：应用报错 "Too many connections" 或获取连接超时

原因：
1. 连接池配置太小（max_connections 太小）
2. 连接泄漏（使用后没有正确归还）
3. 慢 SQL 占用连接太久

解决方案：
1. 增加 max_connections（但不要超过 MySQL 限制）
2. 检查代码确保连接正确关闭（使用 with 语句）
3. 优化慢 SQL，减少连接占用时间
4. 使用连接超时自动回收
```

### 6.2 连接失效

```
症状：报错 "MySQL server has gone away" 或连接超时

原因：
1. 连接长期空闲被防火墙切断
2. MySQL 的 wait_timeout 设置太短
3. 网络不稳定

解决方案：
1. 设置 pool_recycle < MySQL wait_timeout
2. 启用 pool_pre_ping（使用前 ping 一下检查）
3. 设置合理的 max_lifetime 定期刷新连接
```
