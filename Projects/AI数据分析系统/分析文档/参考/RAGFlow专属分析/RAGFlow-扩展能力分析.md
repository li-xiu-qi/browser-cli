# RAGFlow 扩展能力分析

**分析日期**: 2026-02-05  
**核心问题**: 模块化单体架构的扩展能力究竟如何？瓶颈在哪？

---

## 一、一句话结论

> **RAGFlow 的扩展能力是「有限水平扩展」，不是「无限弹性扩展」**：
> -  可以水平扩展 Task Executor（文档解析Worker）
> -  Web 层只能整体扩展，不能细粒度扩展单个模块
> -  受限于共享数据库（MySQL/ES）的写入能力
> -  适合「中小规模」，不适合「超大规模SaaS」

## 四、扩展方案

### 4.1 当前架构下的最大扩展

```yaml
# K8s 部署 - 最大扩展配置

# 1. Web 层：多副本 + 负载均衡
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ragflow-web
spec:
  replicas: 10  # 最多10-20个，再多数据库扛不住
  template:
    spec:
      containers:
        - name: web
          image: ragflow:latest
          command: ["/bin/bash", "--disable-taskexecutor"]
          env:
            - name: DB_POOL_SIZE
              value: "20"  # 限制连接池
---

# 2. Task Executor：大规模Worker
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ragflow-executor
spec:
  replicas: 50  # 可以更多，只受Redis和GPU限制
  template:
    spec:
      containers:
        - name: executor
          image: ragflow:latest
          command: ["/bin/bash", "--disable-webserver"]
          resources:
            limits:
              nvidia.com/gpu: 1  # GPU加速
---

# 3. 数据库：读写分离
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql-master
spec:
  replicas: 1  # 只能1主
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql-slave
spec:
  replicas: 3  # 3从，分担读压力
---

# 4. ES：集群模式
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: es-cluster
spec:
  replicas: 7  # 3主 + 4数据节点
  volumeClaimTemplates:
    - spec:
        resources:
          requests:
            storage: 1Ti  # 大存储
```

## 五、生产建议

### 5.1 不同规模的选择

| 规模 | 推荐方案 | 配置 |
|------|----------|------|
| **< 100用户** | 单机Docker | 4C16G，单机MySQL |
| **100-1000用户** | K8s标准部署 | Web 3副本 + Executor 5副本 |
| **1000-5000用户** | K8s扩展部署 | Web 10副本 + Executor 20副本 + 读写分离 |
| **> 5000用户** | 需要改造 | 分库分表 或 微服务拆分 |
| **> 5万用户** | 必须改造 | 微服务架构重构 |

## 七、一句话总结

> **RAGFlow 的分布式扩展能力确实有限**：
> - Task Executor 扩展性好（Worker模式）
> - Web 层只能整体扩展，无法细粒度扩容
> - 数据库是最大瓶颈（单库单表）
> - **适合中小规模（<5000用户），大规模使用需要架构改造（分库分表或微服务拆分）**
