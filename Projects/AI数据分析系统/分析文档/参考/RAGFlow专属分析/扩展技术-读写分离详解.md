# 扩展技术：数据库读写分离详解

**技术类型**: 数据库性能优化  
**适用场景**: 读多写少的应用  
**难度**: ⭐⭐ 中等

---

## 一、什么是读写分离？

### 1.1 概念解释

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        读写分离原理                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │

│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │                        MySQL 单机                                   │   │
│   │                                                                     │   │
│   │   读请求 ──┐                                                        │   │
│   │   读请求 ──┼──► 查询 ──► 返回结果                                    │   │
│   │   写请求 ──┤                                                        │   │

│   │                                                                     │   │

│                                                                              │

│   │                                                                     │   │

│   │                  │    │  (Master)   │                              │   │
│   │                  │    │  • 处理写    │                              │   │
│   │                  │    │  • 同步数据  │───┐                          │   │
│   │                  │    └─────────────┘   │                          │   │
│   │                  │                       │ 复制                     │   │
│   │                  │    ┌─────────────┐   │ (binlog)                 │   │
│   │   读请求 ────────┼──► │  MySQL 从库  │◄──┘                          │   │
│   │   读请求 ────────┼──► │  (Slave)    │                              │   │
│   │   读请求 ────────┤    │  • 只处理读  │                              │   │
│   │   读请求 ────────┤    └─────────────┘                              │   │
│   │                  │                                                 │   │
│   │                  │    ┌─────────────┐                              │   │
│   │   读请求 ────────┴──► │  MySQL 从库  │                              │   │
│   │                       │  (Slave)    │                              │   │
│   │                       └─────────────┘                              │   │
│   │                                                                     │   │
│   │   优势：读写分离，分担压力                                            │   │
│   │         读能力线性扩展（加从库）                                       │   │
│   │         从库可以定时备份不影响主库                                     │   │
│   │                                                                     │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 一句话解释

> **读写分离 = 写操作去主库，读操作去从库，通过复制保持数据一致**

## 二、为什么需要读写分离？

### 2.1 典型问题场景

```python
# 场景：RAGFlow 的文档查询场景

# 用户上传文档（写操作）
INSERT INTO document (id, name, content) VALUES ('doc1', 'xxx.pdf', '...');
# 耗时：~10ms

# 其他用户同时查询文档列表（读操作）
SELECT * FROM document WHERE tenant_id = 'xxx' ORDER BY create_time DESC;
# 耗时：~50ms（数据量大时需要扫描很多行）

# 问题：
# 1. 读操作慢，阻塞了写操作
# 2. 大量用户同时查询，数据库 CPU 100%
# 3. 单台机器连接数耗尽（默认 151）
```

### 2.2 性能提升数据

| 指标 | 单机 | 1主2从 | 提升 |
|------|------|--------|------|
| 读QPS | ~1000 | ~3000 | 3x |
| 写QPS | ~500 | ~500 | 不变 |
| 连接数 | 151 | 151 x 3 | 3x |
| 延迟P99 | ~100ms | ~30ms | 3x |

## 三、实现原理

### 3.1 MySQL 主从复制

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     MySQL 主从复制原理                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   Master（主库）                                    Slave（从库）           │
│   ──────────────                                    ─────────────           │
│                                                                              │
│   ┌─────────────┐                                  ┌─────────────┐         │
│   │  用户写入   │                                  │             │         │
│   │  INSERT/   │                                  │             │         │
│   │  UPDATE    │                                  │             │         │
│   └──────┬──────┘                                  │             │         │
│          │                                         │             │         │
│          ▼                                         │             │         │
│   ┌─────────────┐      ┌─────────────┐            │             │         │
│   │  binlog     │─────►│  Dump Thread │─────────────►│ I/O Thread │         │
│   │  (二进制日志)│      │  (发送日志)  │    网络       │ (接收日志) │         │
│   └─────────────┘      └─────────────┘            └──────┬──────┘         │
│          ▲                                               │                │
│          │                                               ▼                │
│          │                                        ┌─────────────┐         │
│          │                                        │  relay log  │         │
│          │                                        │  (中继日志) │         │
│          │                                        └──────┬──────┘         │
│          │                                               │                │
│          │                                               ▼                │
│          │                                        ┌─────────────┐         │
│          └────────────────────────────────────────│ SQL Thread  │         │
│                                                   │ (重放SQL)   │         │
│                                                   └──────┬──────┘         │
│                                                          │                │
│                                                          ▼                │
│                                                   ┌─────────────┐         │
│                                                   │  从库数据   │         │
│                                                   └─────────────┘         │
│                                                                              │
│   复制方式：                                                                 │
│   • 异步复制（默认）：Master 不等待 Slave 确认，性能最好，可能丢数据          │
│   • 半同步复制：Master 等待至少一个 Slave 确认，数据安全，性能稍差            │
│   • 同步复制：Master 等待所有 Slave 确认，最安全，性能最差                    │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 数据一致性延迟

```python
# 注意：主从复制有延迟（通常毫秒级，但可能到秒级）

# 主库写入
INSERT INTO document (id, name) VALUES ('doc1', 'xxx.pdf');
# 立即返回成功

# 立刻去从库查询（可能查不到）
SELECT * FROM document WHERE id = 'doc1';
# 返回：Empty set (0.00 sec)  ← 还没同步过来！

# 等待 100ms 后查询
SELECT * FROM document WHERE id = 'doc1';
# 返回：1 row in set (0.00 sec)  ← 同步完成

# 解决方案：
# 1. 对一致性要求高的读，强制走主库（强制读主）
# 2. 写入后等待一定时间再读（sleep）
# 3. 使用半同步复制减少延迟
```

## 四、代码实现

### 4.1 RAGFlow 改造方案

```python
# api/db/db_router.py - 数据库路由模块

import random
from peewee import MySQLDatabase
from contextvars import ContextVar

# 上下文变量：标记当前是否强制读主
db_context = ContextVar('db_context', default='slave')


class MasterSlaveRouter:
    """读写分离路由器"""
    
    def __init__(self, master_config, slave_configs):
        """
        Args:
            master_config: 主库配置
            slave_configs: 从库配置列表
        """
        # 创建主库连接
        self.master = MySQLDatabase(
            master_config['database'],
            host=master_config['host'],
            port=master_config['port'],
            user=master_config['user'],
            password=master_config['password'],
            max_connections=100
        )
        
        # 创建从库连接池
        self.slaves = []
        for config in slave_configs:
            slave = MySQLDatabase(
                config['database'],
                host=config['host'],
                port=config['port'],
                user=config['user'],
                password=config['password'],
                max_connections=50
            )
            self.slaves.append(slave)
    
    def get_db(self, operation='read'):
        """
        获取数据库连接
        
        Args:
            operation: 'read' 或 'write'
        
        Returns:
            Database 连接
        """
        # 写操作必须走主库
        if operation in ['write', 'insert', 'update', 'delete']:
            return self.master
        
        # 检查上下文：是否强制读主
        if db_context.get() == 'master':
            return self.master
        
        # 读操作：负载均衡到从库
        if self.slaves:
            return random.choice(self.slaves)
        
        # 没有从库， fallback 到主库
        return self.master


# 全局路由器实例
router = MasterSlaveRouter(
    master_config={
        'host': 'mysql-master',
        'port': 3306,
        'database': 'rag_flow',
        'user': 'root',
        'password': 'password'
    },
    slave_configs=[
        {'host': 'mysql-slave-1', 'port': 3306, ...},
        {'host': 'mysql-slave-2', 'port': 3306, ...},
    ]
)


class ForceMasterContext:
    """强制读主上下文管理器"""
    
    def __enter__(self):
        self.token = db_context.set('master')
        return self
    
    def __exit__(self, exc_type, exc_val, exc_tb):
        db_context.reset(self.token)


def force_master():
    """装饰器：强制使用主库"""
    def decorator(func):
        def wrapper(*args, **kwargs):
            with ForceMasterContext():
                return func(*args, **kwargs)
        return wrapper
    return decorator
```

### 4.2 Service 层改造

```python
# api/db/services/document_service.py

from api.db.db_router import router, force_master
from api.db.db_models import Document

class DocumentService:
    @classmethod
    def create(cls, **kwargs):
        """创建文档 - 写操作，走主库"""
        db = router.get_db('write')
        
        # Peewee 切换数据库
        with db:
            return Document.create(**kwargs)
    
    @classmethod
    def get_by_id(cls, doc_id):
        """查询文档 - 读操作，走从库"""
        db = router.get_db('read')
        
        with db:
            return Document.select().where(Document.id == doc_id).first()
    
    @classmethod
    def update(cls, doc_id, **kwargs):
        """更新文档 - 写操作，走主库"""
        db = router.get_db('write')
        
        with db:
            return Document.update(**kwargs).where(Document.id == doc_id).execute()
    
    @classmethod
    @force_master()
    def get_by_id_immediate(cls, doc_id):
        """
        强制读主场景：
        - 刚创建文档，需要立刻读取
        - 对一致性要求极高的查询
        """
        return Document.select().where(Document.id == doc_id).first()


# 使用示例
class DocumentApp:
    async def upload(self):
        # 1. 写操作 → 主库
        doc = DocumentService.create(
            name='xxx.pdf',
            tenant_id='tenant_123'
        )
        
        # 2. 需要立刻读取 → 强制读主（因为从库可能还没同步）
        fresh_doc = DocumentService.get_by_id_immediate(doc.id)
        
        return {'doc_id': doc.id}
    
    async def list(self, tenant_id):
        # 3. 普通查询 → 从库
        docs = DocumentService.list_by_tenant(tenant_id)
        return docs
```

### 4.3 使用中间件自动路由

```python
# api/middleware/db_router_middleware.py

from api.db.db_router import router

class DBRouterMiddleware:
    """自动数据库路由中间件"""
    
    async def process_request(self, request):
        """请求开始时，根据 HTTP 方法选择数据库"""
        method = request.method
        
        if method in ['POST', 'PUT', 'DELETE', 'PATCH']:
            # 写操作标记
            request.db = router.get_db('write')
        else:
            # 读操作标记
            request.db = router.get_db('read')
        
        return request


# 在应用中使用
app = Quart(__name__)
app.register_middleware(DBRouterMiddleware())
```

## 五、实际配置

### 5.1 MySQL 主从配置

```bash
# 1. 主库配置（my.cnf）
[mysqld]
server-id = 1                    # 唯一ID
log-bin = mysql-bin              # 开启 binlog
binlog-format = ROW              # 行级复制
binlog-row-image = FULL          # 记录完整行数据

# 创建复制用户
CREATE USER 'repl'@'%' IDENTIFIED BY 'password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';
FLUSH PRIVILEGES;

# 2. 从库配置（my.cnf）
[mysqld]
server-id = 2                    # 唯一ID（与主库不同）
read-only = 1                    # 只读（超级用户除外）
relay-log = mysql-relay-bin      # 中继日志

# 3. 从库启动复制
CHANGE MASTER TO
    MASTER_HOST='mysql-master',
    MASTER_USER='repl',
    MASTER_PASSWORD='password',
    MASTER_LOG_FILE='mysql-bin.000001',
    MASTER_LOG_POS=0;

START SLAVE;

# 4. 查看复制状态
SHOW SLAVE STATUS\G
# 检查：Slave_IO_Running: Yes
#       Slave_SQL_Running: Yes
#       Seconds_Behind_Master: 0
```

## 六、优缺点

### 6.1 优点

| 优点 | 说明 |
|------|------|
| **读性能线性扩展** | 加从库即可提升读能力 |
| **读写不互相阻塞** | 读不影响写，写不影响读 |
| **提高可用性** | 主库挂了，从库可临时接管读 |
| **数据备份** | 从库可用于定时备份 |
| **分担压力** | 报表查询走从库，不影响业务 |

### 6.2 缺点

| 缺点 | 说明 | 解决方案 |
|------|------|----------|
| **数据延迟** | 主从复制有毫秒级延迟 | 强制读主、半同步复制 |
| **写无法扩展** | 只有主库能写 | 分库分表（下一阶段）|
| **架构复杂** | 需要维护多个实例 | 使用云数据库 RDS |
| **故障切换复杂** | 主库挂了需要切主 | 使用 MHA/ Orchestrator |
