# 何时使用队列？——从 DB-GPT / RAGFlow 看架构设计

> 从 DB-GPT 和 RAGFlow 的项目分析中，提炼何时使用队列、何时不用队列的通用原则与架构启发。

---

## 一、核心结论（一句话）

```
实时交互 → 不用队列
后台处理 → 使用队列
```

---

## 二、决策树：何时使用队列？

![决策树](assets/when-to-use-queue-decision-tree.svg)

### 2.1 决策流程

```
任务分析
    ↓
用户需要实时响应？
    ├── 是 → 任务 < 10秒？
    │           ├── 是 → 不用队列（同步调用）
    │           └── 否 → 系统负载突增？
    │                       ├── 是 → 使用队列
    │                       └── 否 → 不用队列
    └── 否 → 需要失败重试？
                ├── 是 → 使用队列（异步处理）
                └── 否 → 系统负载突增？
                            ├── 是 → 使用队列
                            └── 否 → 不用队列
```

### 2.2 决策口诀

| 场景 | 是否用队列 | 口诀 |
|------|-----------|------|
| 实时 + 短时 + 同步 |  不用 | 快就完了 |
| 异步 + 长时 + 可靠 |  使用 | 稳字当头 |

## 三、队列使用矩阵

![队列使用矩阵](assets/queue-usage-matrix.svg)

### 3.1 实时交互场景（不用队列）

| 功能 | 项目 | 为什么不用？ |
|------|------|-------------|
| **Chat 对话** | DB-GPT / RAGFlow | 用户需要立即看到回答 |
| **实时搜索** | 通用 | 查询必须立即返回 |
| **API 查询** | 通用 | 同步调用，超时敏感 |

**核心原因（重要）**：

```
资源消耗分布：
• WebServer: 轻量（CPU 5%，内存 100MB）→ 不是瓶颈，可水平扩展
• Worker: 重量级（GPU 100%，显存 70GB）→ 是瓶颈

队列的作用：
• 不能增加 GPU 算力（Worker 处理能力固定）
• 只能延迟处理时间（增加用户等待）
• 对实时 Chat 没有帮助，反而降低体验

高并发解决方案：
• 限流保护 Worker
• 水平扩展 WebServer（低成本）
• 扩容 Worker（加 GPU，高成本）
```

**共性**：
- 执行时间 < 10秒
- 用户体验要求实时
- **资源消耗在 Worker，不在 WebServer**
- 同步调用更简单

### 3.2 后台处理场景（使用队列）

| 功能 | 项目 | 为什么使用？ |
|------|------|-------------|
| **文档处理** | RAGFlow / DB-GPT | PDF 解析耗时几分钟 |
| **知识库构建** | DB-GPT | 需要削峰填谷 |
| **批量任务** | 通用 | 资源消耗大 |

**共性**：
- 执行时间 > 10秒
- 可以后台异步处理
- 需要失败重试
- 需要进度追踪

## 四、从项目中提炼的架构模式

![架构模式对比](assets/architecture-patterns-comparison.svg)

### 4.1 模式1：同步直接调用

```
客户端 → 网关 → 服务层 → Worker
                (无状态扩展)
```

**适用**：实时交互
- DB-GPT Chat
- RAGFlow Chat
- ChatGPT

**特点**：
-  低延迟
-  实现简单
-  水平扩展

### 4.2 模式2：异步任务队列

```
客户端 → 服务层 → 队列 → 消费者
                    ↓
                   存储（进度追踪）
```

**适用**：后台处理
- RAGFlow 文档处理
- DB-GPT 知识库
- 邮件系统

**特点**：
-  削峰填谷
-  失败重试
-  解耦系统

### 4.3 模式3：混合架构（推荐）

```
        ┌→ Chat API (同步)
客户端 → 网关
        └→ Doc API (异步队列)
```

**适用**：完整 AI 平台
- DB-GPT (完整版)
- RAGFlow
- 企业级系统

**特点**：
-  按需选择
-  最佳实践
-  灵活组合

## 五、DB-GPT vs RAGFlow 对比总结

| 维度 | DB-GPT | RAGFlow | 共性 |
|------|--------|---------|------|
| **Chat** |  无队列 |  无队列 | 都选择同步 |
| **文档处理** |  可选队列 |  必须队列 | 都用队列 |
| **模型服务** | 自托管 Worker | 调用外部 API | 同步调用 |
| **核心定位** | 大模型服务 | 文档智能 | 各有侧重 |

### 关键发现

1. **Chat 都不用队列（不是因为不能扛高并发）**
   - 实时性要求
   - 用户体验优先
   - **资源消耗在 Worker（GPU），不在 WebServer**
   - **队列不能增加 GPU 算力，只能增加延迟**
   - 高并发通过限流 + 扩容 Worker 解决

2. **文档处理都用队列**
   - 耗时任务（几分钟）
   - 用户可等待
   - 削峰填谷
   - 可靠性要求

3. **瓶颈不同，扩展方向不同**
   - DB-GPT: 瓶颈是 GPU Worker → 扩容 Worker（加 GPU）
   - RAGFlow: 瓶颈是 CPU Task Executor → 扩容 Executor
   - WebServer: 不是瓶颈 → 水平扩展（低成本）

## 六、架构启发

![架构启发](assets/lessons-learned.svg)

### 6.1 启发1：队列不是万能的

```
误区：队列能提高性能
真相：队列只能削峰填谷，不能提高吞吐量

Chat 场景的特殊性：
• WebServer: 轻量，可水平扩展到 1000+ 台（成本低）
• Worker: GPU 是瓶颈，处理能力固定（成本高）
• 队列不能增加 GPU 算力，只能让用户排队等待
• 实时 Chat 中，排队 = 体验极差

案例（1000 用户同时请求，10 个 Worker）：
• Worker 处理能力 = 100 QPS
• 不用队列: 100 请求处理，900 请求立即失败（或等待几秒）
• 用队列: 100 请求处理，900 请求排队（等几分钟）
• 结果：吞吐量一样，队列反而增加延迟

正确的高并发方案：
• 限流保护 Worker
• 水平扩展 WebServer（处理连接）
• 扩容 Worker（加 GPU，真正的解决方案）
```

### 6.2 启发2：瓶颈决定扩展方向

```
DB-GPT:
  瓶颈：GPU Worker
  扩展：增加 Worker 数量

RAGFlow:
  瓶颈：CPU Task Executor
  扩展：增加 Executor 进程

教训：先找到瓶颈，再决定如何优化
```

### 6.3 启发3：用户体验优先

```
设计原则：
• 用户等待的 → 同步，无队列
• 用户不等待的 → 异步，用队列

案例：
• Chat：用户盯着屏幕 → 不用队列
• 知识库：用户可离开 → 使用队列
```

### 6.4 启发4：混合架构是趋势

```
现代 AI 平台架构：
┌─────────────────────────────────────┐
│           混合架构                   │
├─────────────────────────────────────┤
│  实时功能        后台功能            │
│  ────────        ────────            │
│  Chat           知识库构建           │
│  搜索           文档处理             │
│  API            批量任务             │
│  ────────        ────────            │
│  同步调用        异步队列            │
└─────────────────────────────────────┘
```

## 七、Chat 高并发问题澄清

### 7.1 常见误解

**误解**: "Chat 不用队列 = 不能处理高并发"

**正解**:
```
Chat 不用队列，不是因为不能扛高并发，而是因为：

1. 资源消耗分布不均
   • WebServer: 轻量（CPU 几 ms，内存 100MB）
   • Worker: 重量级（GPU 100%，显存 70GB）
   • WebServer 可以水平扩展到 1000+ 台

2. 队列不能解决 GPU 瓶颈
   • 队列只能延迟处理
   • 不能增加 Worker 处理能力
   • 对实时 Chat 反而降低体验

3. 高并发的真正解决方案
   • 限流：保护 Worker
   • 扩容 WebServer：处理更多连接（低成本）
   • 扩容 Worker：加 GPU（真正的瓶颈，高成本）
```

### 7.2 资源消耗对比

| 组件 | 单机并发能力 | 资源消耗 | 扩容成本 |
|------|-------------|---------|---------|
| **WebServer** | 10,000+ 连接 | CPU 5%，内存 100MB | 低（云服务器） |
| **Worker** | 10-20 请求 | GPU 100%，显存 70GB | 极高（A100 GPU） |

**结论**: WebServer 不是瓶颈，Worker 才是。

## 八、最佳实践建议

### 8.1 判断 checklist

在决定是否使用队列前，问自己：

| 问题 | 如果"是" | 建议 |
|------|---------|------|
| 用户需要实时看到结果？ | 是 |  不用队列 |
| 任务执行时间 < 10秒？ | 是 |  不用队列 |
| 任务可能失败需要重试？ | 是 |  使用队列 |
| 系统负载可能突增？ | 是 |  使用队列 |
| 需要进度追踪？ | 是 |  使用队列 |
| 任务耗时长（> 1分钟）？ | 是 |  使用队列 |

### 8.2 技术选型建议

| 场景 | 推荐方案 | 说明 |
|------|---------|------|
| 实时对话 | HTTP SSE | 流式输出 |
| 后台任务 | Redis + Celery | 成熟稳定 |
| 企业级 | RabbitMQ | 功能丰富 |
| 大数据量 | Kafka | 高吞吐 |

### 8.3 混合架构设计模板

```
┌─────────────────────────────────────────────────────────┐
│                       API Gateway                        │
│                   (限流/认证/路由)                        │
└─────────────────────────────────────────────────────────┘
                           │
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
    │ 同步服务    │ │ 异步服务    │ │ 混合服务    │
    │ ─────────── │ │ ─────────── │ │ ─────────── │
    │ Chat API    │ │ Task API    │ │ Search API  │
    │ (直接调用)  │ │ (投递队列)  │ │ (缓存优先)  │
    └─────────────┘ └─────────────┘ └─────────────┘
                           │
                    ┌──────┴──────┐
                    ▼             ▼
            ┌─────────────┐ ┌─────────────┐
            │ 任务队列    │ │ 结果存储    │
            │ (Redis)     │ │ (DB/Cache)  │
            └─────────────┘ └─────────────┘
```

## 八、总结

### 8.1 核心原则

```
┌─────────────────────────────────────────────────────────────┐
│                      架构设计原则                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   1. 实时交互不用队列，后台处理使用队列                       │
│                                                              │
│   2. 队列不能提高吞吐量，只能削峰填谷                        │
│                                                              │
│   3. 找到真正的瓶颈，再决定扩展方向                          │
│                                                              │
│   4. 用户体验优先，技术服务于业务                            │
│                                                              │
│   5. 混合架构是趋势，按需选择最适合的方案                     │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 8.2 从项目中学习

| 项目 | 学什么 | 怎么用 |
|------|--------|--------|
| **DB-GPT** | Controller-Worker 模式 | 自托管模型服务 |
| **RAGFlow** | 任务队列模式 | 文档处理管道 |
| **两者** | 混合架构 | 完整 AI 平台 |

### 8.3 一句话总结

> **队列是工具，不是目的。实时交互不用队列（不是因为不能扛高并发，而是因为资源消耗在 Worker，队列不能增加 GPU 算力）。后台处理使用队列。找到真正的瓶颈（Worker），选择最适合的方案（限流 + 扩容 Worker）。**
