# DB-GPT vs RAGFlow - 对话历史管理专项对比报告

> 深度对比两个项目的前后端对话历史管理机制

---

## 一、执行摘要

通过对 DB-GPT 和 RAGFlow 对话历史管理的深度分析，我们发现了两种截然不同的设计理念：

| 维度 | DB-GPT | RAGFlow |
|------|--------|---------|
| **架构风格** | 分层解耦，消息模型分离 | 集中存储，JSON 数组 |
| **前端状态** | ChatContext + useChat Hook | React Hooks 派生状态 |
| **后端存储** | 关系型 (独立消息表) | 关系型 + JSON 数组 |
| **Agent 历史** | GptsMemory 内存管理 | Canvas 全局变量 |
| **历史裁剪** | 智能溢出 (重要性排序) | Token 预算硬截断 |
| **引用溯源** |  无 |  索引对齐数组 |

## 四、后端存储设计对比

### 4.1 数据库表结构

**DB-GPT: 关系型存储 (独立消息表)**
```python
# packages/dbgpt-core/src/dbgpt/storage/chat_history/chat_history_db.py

class ChatHistoryEntity(Base):
    """对话主表"""
    __tablename__ = "chat_history"
    id = Column(Integer, primary_key=True)
    conv_uid = Column(String(255), unique=True)  # 对话唯一ID
    chat_mode = Column(String(255))  # 对话模式
    summary = Column(Text)  # 对话摘要
    # ... 其他元数据

class ChatHistoryMessageEntity(Base):
    """消息独立表 - 关键设计"""
    __tablename__ = "chat_history_message"
    id = Column(Integer, primary_key=True)
    conv_uid = Column(String(255), ForeignKey("chat_history.conv_uid"))
    index = Column(Integer)  # 消息顺序
    round_index = Column(Integer)  # 对话轮次
    
    # 消息内容 - 多态存储
    message = Column(Text)  # JSON 序列化的消息
    
    # 是否传递给模型
    pass_to_model = Column(Boolean, default=True)
```

**RAGFlow: 混合型存储 (JSON 数组)**
```python
# api/db/db_models.py

class Conversation(DataBaseModel):
    """对话表 - 消息存储为 JSON 数组"""
    __tablename__ = "conversation"
    
    dialog_id = Column(String(255), nullable=False)
    
    # 核心设计: 消息和引用分别存储为 JSON 数组
    message = Column(JSON, default=list)  
    # [{"role": "user", "content": "..."}, ...]
    
    reference = Column(JSON, default=list)
    # [{"chunks": [...], "doc_aggs": [...]}, ...]  # 与 message 索引对齐
    
    # 其他配置
    llm_setting = Column(JSON)
    prompt_config = Column(JSON)
```

### 4.2 存储策略对比

| 维度 | DB-GPT | RAGFlow |
|------|--------|---------|
| **消息粒度** | 独立记录 | JSON 数组元素 |
| **查询效率** | ⭐⭐⭐⭐⭐ SQL 查询 | ⭐⭐⭐ JSON 解析 |
| **分页支持** |  天然支持 | ⚠️ 需内存分页 |
| **引用关联** | 消息内部字段 | 索引对齐数组 |
| **扩展性** | ⭐⭐⭐⭐ 易于添加字段 | ⭐⭐ 需修改 JSON 结构 |

## 五、Agent 运行时历史管理对比

### 5.1 Agent 如何获取历史

**DB-GPT: GptsMemory 内存管理**
```python
# packages/dbgpt-core/src/dbgpt/agent/core/memory/gpts/gpts_memory.py:89-112

class GptsMemory:
    def __init__(self):
        self.messages_cache: List[GptsMessage] = []
    
    def get_history(self, n: int = 5) -> List[GptsMessage]:
        """获取最近 N 轮对话历史"""
        return self.messages_cache[-n:]
    
    async def _save(self, message: GptsMessage):
        """保存消息到内存和持久化"""
        self.messages_cache.append(message)
        # 同时保存到数据库...
```

**RAGFlow: Canvas 全局变量**
```python
# agent/canvas.py:718-742

class Canvas:
    def __init__(self):
        self.history: List[tuple] = []  # (role, content) 元组
        self.globals: Dict = {}
    
    def get_history(self, window_size: int = 13) -> List[dict]:
        """获取格式化后的历史"""
        history_msgs = []
        for role, content in self.history[-window_size:]:
            history_msgs.append({"role": role, "content": content})
        return history_msgs
    
    def add_turn(self, role: str, content: str):
        """添加一轮对话"""
        self.history.append((role, content))
        self.globals["sys.history"] = self._format_history()
```

## 八、优缺点总结

### 8.1 DB-GPT

**优点**:
-  消息模型分离，前端/Agent/存储各司其职
-  独立消息表，查询效率高
-  GptsMemory 支持智能溢出
-  多态消息存储 (BaseMessage 继承体系)

**缺点**:
-  架构复杂，学习成本高
-  消息转换开销大
-  无引用溯源机制
-  并发隔离依赖 conv_uid

---

### 8.2 RAGFlow

**优点**:
-  架构简单，易于理解
-  引用溯源设计优雅 (索引对齐)
-  Token 预算控制精确
-  Canvas 全局变量便于 Agent 访问

**缺点**:
-  JSON 数组查询效率低
-  消息和引用必须同步维护
-  历史裁剪简单粗暴
-  缺乏消息重要性评分

| RAGFlow | 引用同步问题 | 使用事务保证 message/reference 一致性 |

## 十、结论

DB-GPT 和 RAGFlow 代表了对话历史管理的两种设计哲学：

1. **DB-GPT** 追求**分层解耦**，通过独立的消息表和内存管理实现灵活的历史控制
2. **RAGFlow** 追求**简单直接**，通过 JSON 数组和索引对齐实现高效的引用溯源

对于我们自己的项目，建议：

- **存储层**: 借鉴 DB-GPT 的独立消息表设计
- **引用层**: 借鉴 RAGFlow 的索引对齐机制
- **裁剪层**: 结合 DB-GPT 的智能溢出和 RAGFlow 的 Token 预算
- **Agent 层**: 借鉴 Canvas 的全局变量设计，便于历史访问
