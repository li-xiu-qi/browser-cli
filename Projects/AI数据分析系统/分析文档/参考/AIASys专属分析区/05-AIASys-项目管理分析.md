## 二、AI 协作规范

AIASys 建立了一套完整的 AI 与人类协作体系，核心理念是"及时反馈，及时调整，快速迭代优化"。

### 2.1 AGENTS.md 管理规则

AGENTS.md 是 AI 和人类的第一视角入口，采用倒金字塔结构组织内容：

**内容优先级**: 快速导航 > AI 技能库 > 项目架构 > 文档索引 > 项目概述 > 技术栈 > 项目结构 > 快速命令 > 代码风格

**更新触发条件**:
- 新增技术栈 → 更新"技术栈"表格
- 新增文档目录 → 更新"完整文档索引"
- 新增 Skill → 更新"AI 技能库"
- 端口变更 → 更新"端口配置"
- 架构调整 → 更新"项目架构"图

### 2.2 AI 和人类的分工

| 角色 | 主要职责 | 工作方式 |
|------|----------|----------|
| **AI Agent** | 执行编码任务、遵循规范、发现问题 | 读取 Skill 后执行，遇到问题立即反馈 |
| **人类开发者** | 验证 AI 输出、提供反馈、更新规范 | 审查代码、调整 Skill、优化流程 |
| **维护者** | 定期整理经验、更新文档和 Skill | 每周/每月审查规范有效性 |

### 2.3 反馈循环机制

```
开始任务 → AI 执行 → 结果检查
              ↑         ↓
         即时反馈 ← 不满意
              ↓
         标记完成 → 记录经验 → 优化规范
```

**反馈原则**:
- **即时性**: 发现问题立即反馈
- **具体性**: 指出具体问题位置
- **建设性**: 提供改进方向
- **可追溯**: 记录反馈和修改

### 2.4 迭代优化策略

采用"小步快跑"模式：大目标拆分为小任务，快速验证，及时反馈调整。

**经验沉淀方式**:
- 任务执行问题 → 更新 Skill
- 代码规范调整 → 更新代码风格文档
- 流程优化 → 更新工作流文档
- 架构决策 → 更新架构文档

### 3.2 核心技能详解

#### aiasys-start: 需求分析与头脑风暴

**使用时机**: 接到新功能需求、需要分析可行性、拆解复杂任务

**核心流程**: 接收需求 → 分析上下文 → 识别约束 → 提出方案 → 输出结论

**必须读取的文档**:
- AGENTS.md - 项目概览
- docs/ai/01-start/ - 项目基本信息
- docs/ai/02-fe/design.md - 如涉及前端
- docs/ai/03-be/arch.md - 如涉及后端

**输出格式**: 分析结论包含目标、技术方案、涉及文件、风险评估、下一步建议

#### aiasys-plan: 编写实现计划

**使用时机**: 已完成需求分析、需要制定实现计划、将大任务拆分为小步骤

**计划格式要求**:
- 目标: 一句话描述
- 技术方案: 2-3 句话描述
- 任务列表: 每个任务包含文件、步骤、验证点
- 依赖关系: 使用 Mermaid 图示

**任务拆分原则**:
- 小步骤: 每个任务 2-5 分钟
- 可验证: 每个任务有明确验收标准
- 无依赖: 任务之间依赖清晰

**输出位置**: docs/plans/YYYY-MM-DD-[feature].md

#### aiasys-execute: 批量执行任务

**使用时机**: 已有书面计划、需要批量处理任务、需要子代理并行开发

**执行模式**:
- **顺序执行**: 适用于任务有依赖关系的场景
- **子代理并行**: 适用于无依赖的独立任务

**批次策略**: 默认每批次 3 个任务，执行后验证结果、报告进度、等待反馈

**停止条件**: 任务无法继续、验证反复失败、计划有明显缺陷时立即停止

#### aiasys-debug: 系统调试

**核心原则**: "没有根本原因调查，不得修复"

**四阶段调试法**:
1. **收集证据**: 阅读错误信息、一致重现、检查最近变更
2. **定位根因**: 跟踪数据流、分层诊断
3. **创建测试**: 修复前必须先有失败测试
4. **实施修复**: 修复根本原因、运行测试确认、运行回归测试

**红旗信号**: 立即停止返回阶段 1 的情况
- "先试试这个" - 未调查就修复
- 已尝试 2+ 次修复仍未解决
- 每次修复在不同地方引发新问题

#### aiasys-review: 代码审查

**审查检查清单**:
- 功能正确性: 实现符合需求、边界条件处理、错误处理完善
- 代码质量: 命名清晰、函数职责单一、无重复代码、复杂度合理
- 项目规范: 符合代码风格、类型注解完整、有必要注释、文档已更新
- 测试: 测试覆盖主要场景、测试可独立运行、无脆弱测试

### 3.3 技能调用方式

技能通过 AGENTS.md 中的"AI 技能库"表格进行索引，使用时读取对应的 SKILL.md 文件即可。

## 四、功能实现状态管理

AIASys 建立了系统化的功能实现追踪体系，使用状态图标和实现等级双重维度管理功能开发。

### 4.1 状态图标定义

| 图标 | 状态 | 说明 |
|------|------|------|
| ⬜ | 未开始 | 已规划但未实现 |
|  | 进行中 | 正在开发 |
| 🟡 | 部分完成 | 核心可用，待完善 |
|  | 已完成 | 已通过验收 |
|  | 已废弃 | 不再维护 |

### 4.2 实现等级定义 L0-L4

| 等级 | 名称 | 关键要求 | 适用场景 |
|------|------|----------|----------|
| L0 | 概念 | 设计文档、技术方案、风险评估 | 需求分析、技术预研 |
| L1 | 原型 | 核心流程可运行、基础代码实现 | 可行性验证 |
| L2 | 可用 | 功能完整、基础错误处理、通过类型检查 | 生产可用 |
| L3 | 完善 | 单元测试覆盖、完整文档、示例代码 | 功能完整 |
| L4 | 优化 | 性能测试、监控告警、故障恢复 | 高性能、高可用 |

### 4.3 文档职责分离

| 文档 | 内容重点 | 面向对象 | 示例 |
|------|----------|----------|------|
| **changelog/** | 变更摘要："做了什么" | 开发者、用户 | 新增中文字体支持，包含某某功能 |
| **completion-records/** | 验收质量："做得怎么样" | 项目管理 | 检查清单、L2 等级、已知问题 |

**规则**:
- Changelog 只记录变更摘要，按新增/修复/优化分类
- 验收报告必须包含：验收检查清单、实现等级 L0-L4、已知问题、后续建议
- 验收报告通过关联 Changelog 链接引用详细变更内容，避免重复描述

### 4.4 工作流程

**功能开发前**:
1. 在对应模块文档中添加功能条目，状态设为 ⬜
2. 定义验收标准 acceptance-criteria/

**功能开发中**:
1. 更新模块文档中的状态为 
2. 记录开发进度

**功能完成后**:
1. 创建验收报告 completion-records/
2. 更新模块状态为 
3. 记录变更摘要 changelog/
4. 标记任务完成 todo-lists/

## 五、Changelog 管理

AIASys 建立了严格的变更日志编写规范，确保变更记录清晰、可追溯。

### 5.1 记录范围

**不记录的内容**:
- 版本号更新
- Docker 部署操作
- 文档编写
- 配置变更
- 代码格式调整
- 文件移动/重命名

**需要记录的内容**:
- 功能新增: API 接口、组件模块、工具函数、配置项
- Bug 修复: 功能缺陷、接口错误、数据处理错误、安全问题
- 重构改动: 架构调整、核心逻辑重写、接口变更
- 性能优化: 响应速度、内存占用、并发处理

### 5.2 格式规范

```markdown
# Changelog vX.X.X - YYYY-MM-DD

## 贡献者GitHub用户名

### 功能标题

#### 问题描述（Bug 修复类需要）
- 现象：
- 原因：

#### 改动内容
- 具体修改点 1
- 具体修改点 2

#### 文件变更
- `path/to/file`: 变更说明
```

### 5.3 版本号管理

**正式版本格式**: 主版本.次版本.补丁版本，如 v0.1.4

**开发/调试细分版本格式**: x.y.z.a.b
- x.y.z: 正式版本号
- a: 前后端标识，1=前端，2=后端
- b: 最小更新计数

**版本更新规则**:
- 修改前端代码: 前端最小版本 +1
- 修改后端代码: 后端最小版本 +1
- 重启服务验证: 每次重启 +1
- 功能完成正式发布: 升级正式版本号，去掉最小版本号

**核心架构**:
```
API Layer (FastAPI)
    ↓
Logic Layer (DataAnalysisTeam / SessionManager)
    ↓
Engine Layer (smolagents: CodeAgent / JupyterKernelInterpreter)
```

**任务分类**:
- 简单任务: 独立函数、工具方法

## 七、代码风格规范

### 7.1 Python 规范

| 工具 | 用途 | 命令 |
|------|------|------|
| black | 格式化 | `python -m black --line-length 79 .` |
| isort | 导入排序 | `python -m isort --line-length 79 --profile black .` |
| mypy | 类型检查 | `python -m mypy --ignore-missing-imports app/` |
| pylint | Linter | `python -m pylint --disable=R,C app/` |

**通用规范**:
- 文件编码: UTF-8
- 注释: 优先使用中文
- 类型定义: 优先使用 TypedDict，不要使用 dict
- 异常处理: 不要过度 try except
- 废弃代码: 直接删除，不要标记废弃
- 字符串: 优先使用 `"""` 多行字符串语法
- 别名: 少用别名，尽量直接引用
- 参数处理: 少用 argparse，直接调用

### 7.2 TypeScript/React 规范

| 工具 | 用途 | 命令 |
|------|------|------|
| prettier | 格式化 | 自动运行 |
| eslint | Linter | `npm run lint` |
| tsc | 类型检查 | `npx tsc --noEmit` |

### 7.3 文档规范

- 严禁在文档内编写代码块
- 应使用详细的文字描述进行替代

## 八、Git 工作流

### 8.1 本地代码检查

项目使用 GitHub Actions 进行 CI 检查，本地使用脚本进行预检：

```bash
# Windows
.\scripts\check.ps1

# Linux/Mac
./scripts/check.sh

# 只检查前端
.\scripts\check.ps1 frontend

# 只检查后端
.\scripts\check.ps1 backend
```

### 8.2 CI 检查项

| 检查项 | 作用 | 目标文件 |
|--------|------|----------|
| Frontend Lint | ESLint + TypeScript 检查 | apps/web/**/*.{ts,tsx} |
| Frontend Format | Prettier 格式检查 | apps/web/**/*.{ts,tsx,js,jsx,json,css,scss} |
| Backend Black | Python 格式检查 | apps/backend/**/*.py |
| Backend isort | 导入排序检查 | apps/backend/**/*.py |
| Backend Pylint | 代码质量检查 | apps/backend/**/*.py |
| Docker Build | 镜像构建检查 | apps/web, apps/backend |

### 8.3 提交流程

1. 本地预检: 提交前运行检查脚本
2. 提交代码: git commit
3. 推送分支: git push
4. 创建 PR: GitHub 上创建 Pull Request
5. CI 检查: GitHub Actions 自动运行检查
6. 代码审查: 审查通过后合并

### 8.4 提交信息规范

格式: `type(scope): subject`

**类型说明**:
- feat: 新功能
- fix: Bug 修复
- docs: 文档更新
- style: 代码格式调整
- refactor: 代码重构
- test: 测试相关
- chore: 构建/工具相关

**示例**: `feat(agent): 添加 Worker 文档化汇报功能`

### 8.5 提交前检查清单

| 文档 | 检查项 |
|------|--------|
| changelog/ | 变更摘要是否已记录 |
| completion-records/ | 验收报告是否已创建 |
| implementation-status/ | 模块状态是否已更新 |

## 九、与 smolagents 社区对比

### 9.1 smolagents 文档体系特点

smolagents 是 Hugging Face 开发的智能体框架，其文档体系具有以下特点：

**官方文档结构**:
- 概念介绍: Agent 架构、工具系统、代码执行
- API 文档: 类和方法的详细说明
- 示例教程: 从入门到进阶的使用示例
- 最佳实践: 推荐的开发模式

**优势**:
- 文档与代码紧密集成
- 示例丰富且可运行
- 社区贡献指南清晰
- 版本更新说明详细

**不足**:
- 缺乏项目级的 AI 协作规范
- 没有系统化的任务管理指南
- 缺少实现状态追踪机制

### 9.2 AIASys 文档体系特点

**优势**:
- 三层文档架构清晰：AI 提示词层、开发指南层、项目管理层
- 完整的 AI 协作规范：Skill 体系、反馈机制、迭代优化
- 系统化的任务管理：前后端分别管理、难度分级、验收标准
