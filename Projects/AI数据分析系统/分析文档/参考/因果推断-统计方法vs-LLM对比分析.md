# 因果推断：统计方法 vs LLM 能力对比分析

**分析日期**: 2026-02-05  
**核心问题**: RATH的统计因果推断 vs LLM的因果推理，谁更强？  
**结论**: 两者完全不同，各有适用场景

---

## 二、LLM真的有因果推断能力吗？

### 2.1 LLM的"因果能力"是什么？

LLM表现出的因果能力是**语言层面的因果推理**，不是**统计因果推断**：

```python
# LLM能做到的：因果语言理解
prompt = """
小明没吃早饭就去上学了，课堂上他感到头晕。
问：小明头晕的可能原因是什么？
"""

LLM回答："小明头晕可能是因为没吃早饭导致低血糖。"
# ↑ 这是基于语言模式的因果解释

# LLM做不到的：统计因果推断
prompt = """
数据：
- 1000个没吃早饭的人，300人头晕（30%）
- 1000个吃了早饭的人，100人头晕（10%）

问：吃早饭对头晕的因果效应是多少？
置信区间是多少？
"""

LLM可能会回答："不吃早饭导致头晕的概率增加20%"
# ↑ 但：
# 1. 没有控制混杂因素（如熬夜、贫血等也导致头晕）
# 2. 没有给出置信区间
# 3. 没有验证统计显著性
```

### 2.2 LLM因果能力的来源

| 能力类型 | 来源 | 可靠性 |
|---------|------|--------|
| **常识因果** | 训练数据中的因果陈述 | 中等（可能有偏见） |
| **语言模式** | "因为...所以..."句型 | 高（语法正确） |
| **统计推断** | 无 | N/A |
| **反事实推理** | 基于语言生成 | 低（无法验证） |

**关键洞察**：
> LLM学到的是**人类描述因果的方式**，不是**因果本身**。

## 三、深度对比

### 3.1 能力维度对比

| 能力 | RATH(统计因果) | LLM | 说明 |
|------|---------------|-----|------|
| **因果发现** |  强（PC/GES算法） | ⚠️ 弱 | LLM无法从数据中发现新因果 |
| **因果效应量化** |  精确数值 |  无法 | LLM不能计算"增加10万预算→销售+25万" |
| **置信区间** |  自动计算 |  无 | 统计显著性只有统计方法能给 |
| **混杂控制** |  DoWhy自动处理 |  无 | LLM不理解混杂因素概念 |
| **反事实预测** |  数学推导 | ⚠️ 猜测 | LLM的"如果...会怎样"是生成不是计算 |
| **解释因果** | ⚠️ 技术术语 |  通俗语言 | LLM擅长用人话解释 |
| **处理新领域** |  需要数据 |  零样本 | LLM有常识迁移能力 |
| **验证因果** |  反驳检验 |  无法 | DoWhy可做稳健性检验 |

### 3.2 实际案例对比

#### 案例：营销预算 vs 销售额

**场景**: 你有6个月的营销预算和销售额数据，想知道增加预算能否提升销售。

```
数据：
月份    预算(万)   销售额(万)
1月     10        100
2月     15        140
3月     12        110
4月     20        180
5月     18        160
6月     25        220
```

---

**LLM的分析**:

```
输入给LLM：
"以下是营销预算和销售额数据，请问增加预算能提升销售吗？
[数据表格]"

LLM可能回答：
"从数据看，预算和销售额呈现正相关关系。
1月到6月，预算从10万增加到25万，销售额从100万增加到220万。
这表明增加营销预算可能有助于提升销售额。

但需要注意：
-  correlation does not imply causation
- 可能受其他因素影响，如季节性、市场竞争等"

问题：
 没有量化效应（增加1万预算→销售增加多少？）
 没有控制混杂因素（如6月是618大促）
 没有置信区间（这个结论可靠吗？）
```

---

**RATH的分析**:

```python
# Step 1: 因果发现
# 先发现因果图：预算 → 销售额
# 同时发现混杂因素：季节 → 预算
#                  季节 → 销售额

# Step 2: 因果推断（控制季节因素后）
result = dowhy.estimate_effect(
    treatment="budget",
    outcome="sales",
    method="backdoor.linear_regression"
)

输出：
{
  "causal_effect": 8.5,  # 每增加1万预算，销售增加8.5万
  "confidence_interval": [6.2, 10.8],  # 95%置信区间
  "p_value": 0.002,  # 统计显著
  "refutation": "通过随机混杂因子检验"  # 稳健性验证
}

解读：
 量化效应：增加10万预算 → 销售预计增加85万（置信区间62-108万）
 统计保证：p=0.002，结果可信
 已控制季节因素
 反驳检验通过
```

## 四、LLM在什么情况下会出错？

### 4.1 典型案例：Simpson悖论

```
数据：某医院两种疗法的治愈率

        轻症患者          重症患者         总计
        治愈/总数  率     治愈/总数  率     治愈/总数  率
疗法A    81/90    90%    192/320   60%    273/410   67%
疗法B    234/270  87%     55/80    69%    289/350   83%

问：哪种疗法更好？
```

**LLM可能回答**: "疗法B总体治愈率83% > 疗法A的67%，疗法B更好。"

**错误！这是Simpson悖论。**

**正确分析**（RATH/统计方法）：
```python
# 分层分析
# 轻症：疗法A 90% > 疗法B 87%
# 重症：疗法A 60% < 疗法B 69%

# 混杂因素：病情严重程度
# - 疗法A更多用于治疗重症（320/410=78%）
# - 疗法B更多用于治疗轻症（270/350=77%）

# 结论：
# - 轻症患者：选疗法A
# - 重症患者：选疗法B
# 不能简单比较总体治愈率！
```

### 4.2 LLM的其他局限

| 局限 | 例子 |
|------|------|
| **相关≠因果** | "冰淇淋销量和溺水人数相关，所以..." LLM可能错误归因 |
| **混杂因素** | 无法自动识别和控制混杂变量 |
| **样本偏差** | 不理解选择偏差对因果估计的影响 |
| **反事实** | 无法计算"如果当初..."的数学期望 |
| **不确定性** | 不能给出置信区间和P值 |

### 5.2 分工建议

| 任务 | 推荐方法 | 理由 |
|------|---------|------|
| **因果发现** | 统计方法 | LLM无法从数据中发现新因果 |
| **因果效应量化** | 统计方法 | 只有数学方法能给精确数值 |
| **结果解释** | LLM | LLM擅长用人话解释技术结果 |
| **业务建议** | LLM | LLM有商业常识和上下文理解 |
| **报告生成** | LLM | NLG能力强 |
| **假设生成** | LLM | 可以从知识库提出可能因果 |
| **假设验证** | 统计方法 | 必须用数据验证 |

## 六、总结

### 核心结论

1. **LLM没有真正的因果推断能力**
   - LLM有的是"因果语言理解"和"因果常识"
   - 无法替代统计因果推断的数学基础

2. **RATH的统计因果推断是必需的**
   - 量化因果效应（必需）
   - 控制混杂因素（必需）
   - 统计显著性检验（必需）

3. **最佳方案是结合**
   - 统计方法做"硬分析"（计算）
   - LLM做"软分析"（解释、建议）

### 一句话对比

> **LLM告诉你"可能为什么"，RATH告诉你"到底多少、有多确定"。**

### 使用建议

| 你的需求 | 推荐方案 |
|---------|---------|
| 快速理解数据关系 | LLM初步探索 |
| 做业务决策（预算分配） | RATH统计因果 |
| 写分析报告 | LLM基于RATH结果生成 |
| 发现未知因果关系 | RATH因果发现算法 |
| 向老板汇报 | RATH数据 + LLM润色 |
| 学术研究 | 统计因果方法（必须） |
