# DB-GPT 模型评测功能深度分析

---

## 一、功能定位

### 1.1 一句话定义
**模型评测是 DB-GPT 的"效果评估中心"，帮助用户量化模型能力、对比不同方案、持续优化应用效果。**

### 1.2 解决的核心问题

```
没有评测时的问题：                    有评测后的改进：
─────────────────                     ───────────────
"这个模型效果好吗？"                  "准确率达到85%，比基线高10%"
"换模型会不会更好？"                  "A模型准确率85%，B模型90%，建议切换"
"Prompt怎么优化？"                   "测试了5个版本，V3效果最好"
"应用上线后效果如何？"                "每周自动评测，效果波动监控"
    │                                      │
    ▼                                      ▼
凭感觉决策                            数据驱动决策
```

### 1.3 在系统中的角色
```
在线对话 ──收集反馈──► 模型评测 ──输出报告──► 应用优化
                              │
                              ├─ 模型选型
                              ├─ Prompt优化
                              ├─ 参数调优
                              └─ 效果监控
```

### 2.3 评测指标层

#### Text2SQL 评测指标

| 指标 | 定义 | 说明 |
|------|------|------|
| **SQL 准确率** | 生成的SQL与参考答案完全一致的比例 | 最严格的指标 |
| **执行准确率** | SQL执行结果与参考答案一致的比例 | 允许语法不同但结果相同 |
| **语法正确率** | 生成的SQL能成功执行的比例 | 衡量SQL合法性 |
| **Schema 链接准确率** | 正确识别表和字段的比例 | 衡量理解能力 |

#### RAG 评测指标

| 指标 | 定义 | 说明 |
|------|------|------|
| **召回率 (Recall)** | 检索到的相关文档 / 所有相关文档 | 衡量检索全面性 |
| **准确率 (Precision)** | 检索到的相关文档 / 检索到的所有文档 | 衡量检索精确性 |
| **F1 Score** | 召回率和准确率的调和平均 | 综合指标 |
| **MRR** | 第一个相关文档的排名倒数 | 衡量排序质量 |
| **NDCG** | 考虑排序位置的累积增益 | 衡量整体排序质量 |

#### 对话质量评测指标

| 指标 | 定义 | 说明 |
|------|------|------|
| **相关性** | 回答与问题的相关程度 | 人工打分或模型评估 |
| **准确性** | 回答内容的事实正确性 | 需要参考答案对比 |
| **完整性** | 回答是否覆盖了问题的所有方面 | 人工评估 |
| **流畅性** | 语言表达是否自然流畅 | 语言模型评估 |

## 三、评测场景详解

### 场景一：Text2SQL 模型选型

#### 背景
企业需要选择一个适合业务数据库的 Text2SQL 模型。

#### 评测流程
```
1. 准备数据集
   ├── 收集业务场景的真实问题
   ├── 编写对应的正确SQL
   └── 整理成评测数据集（100-500条）

2. 创建评测任务
   ├── 选择场景：Text2SQL
   ├── 选择候选模型：Model A, Model B, Model C
   ├── 选择指标：SQL准确率、执行准确率
   └── 配置数据库连接（用于执行验证）

3. 运行评测
   └── 系统自动对每个模型运行数据集

4. 对比结果
   ┌────────────────┬──────────┬──────────┐
   │     模型       │ SQL准确率 │ 执行准确率 │
   ├────────────────┼──────────┼──────────┤
   │   Model A      │   72%    │   85%    │
   │   Model B      │   68%    │   82%    │
   │   Model C      │   78%    │   88%    │
   └────────────────┴──────────┴──────────┘

5. 决策
   └── 选择 Model C，综合表现最好
```

#### 深度分析

**为什么需要执行准确率？**
```
示例：
问题："查询销售额最高的客户"

参考答案：
SELECT customer_id, SUM(amount) as total
FROM orders
GROUP BY customer_id
ORDER BY total DESC
LIMIT 1

模型A生成：
SELECT customer_id, MAX(amount) as total
FROM orders
GROUP BY customer_id
ORDER BY total DESC
LIMIT 1

分析：
- SQL准确率：0%（语法不同）
- 执行准确率：可能正确（如果恰好同一个客户）

结论：执行准确率更贴近实际使用效果
```

### 场景二：RAG 检索优化

#### 背景
知识库问答效果不佳，需要优化检索环节。

#### 评测流程
```
1. 准备评测集
   ├── 收集用户常见问题
   ├── 标注每个问题的相关文档片段
   └── 标注期望的答案

2. 对比不同配置
   ┌────────────────┬────────┬────────┬──────┐
   │    配置项      │ 召回率 │ 准确率 │  F1  │
   ├────────────────┼────────┼────────┼──────┤
   │  chunk_size=200│  85%   │  70%   │ 77%  │
   │  chunk_size=500│  78%   │  82%   │ 80%  │
   │  chunk_size=1000│ 65%   │  88%   │ 75%  │
   ├────────────────┼────────┼────────┼──────┤
   │  top_k=3       │  72%   │  85%   │ 78%  │
   │  top_k=5       │  85%   │  78%   │ 81%  │
   │  top_k=10      │  92%   │  65%   │ 76%  │
   └────────────────┴────────┴────────┴──────┘

3. 选择最佳配置
   └── chunk_size=500 + top_k=5，F1最高

4. 持续监控
   └── 每周运行评测，监控效果波动
```

#### 深度分析

**召回率 vs 准确率的权衡**
```
高召回率低准确率：
- 检索到很多内容，包含正确答案
- 但也带入了很多噪音
- 结果：LLM被干扰，回答质量下降

低召回率高准确率：
- 检索到的内容都很相关
- 但可能漏掉了关键信息
- 结果：LLM缺乏必要信息，回答不完整

最优策略：
- 根据业务场景选择平衡点
- 知识问答：优先保证召回率（不能漏掉关键信息）
- 客服场景：优先保证准确率（不能提供错误信息）
```

### 场景三：Prompt 版本迭代

#### 背景
优化 Prompt 提升 Text2SQL 准确率。

#### 评测流程
```
1. 准备多个Prompt版本
   V1: 基础提示词
   V2: 增加示例（Few-shot）
   V3: 增加Schema描述优化
   V4: 增加错误处理说明

2. 同一数据集评测
   ┌────────┬──────────┬──────────┐
   │ 版本   │ SQL准确率 │ 执行准确率 │
   ├────────┼──────────┼──────────┤
   │   V1   │   65%    │   78%    │
   │   V2   │   72%    │   83%    │
   │   V3   │   78%    │   87%    │
   │   V4   │   75%    │   85%    │
   └────────┴──────────┴──────────┘

3. 分析差异
   └── V3效果最好，Schema描述优化最关键

4. 应用到生产
   └── 所有Text2SQL应用使用V3 Prompt
```

## 五、评测使用最佳实践

### 5.1 构建高质量评测集

```
原则一：覆盖真实场景
 从实际业务中收集问题
 不要用人工编的简单问题

原则二：分布合理
├── 简单问题 40%：常规查询
├── 中等问题 40%：多条件、多表
└── 复杂问题 20%：嵌套、聚合、计算

原则三：标注准确
├── 参考答案必须经过验证
├── SQL要实际执行确认
└── 边界情况要充分考虑

原则四：持续更新
├── 定期补充新问题
├── 移除过时的问题
└── 保持评测集的时效性
```

## 六、设计亮点与不足

### 6.1 设计亮点

#### 亮点一：场景化评测
```
针对不同场景设计专门的评测指标
- Text2SQL：关注SQL正确性和执行结果
- RAG：关注检索质量和回答相关性
- 对话：关注流畅性和准确性

价值：评测结果更贴近实际使用效果
```

#### 亮点二：多维度对比
```
支持多模型、多版本、多配置的横向对比
一目了然看到差异和优劣

价值：辅助科学决策，避免主观判断
```

#### 亮点三：错误分析
```
不仅给出分数，还能查看具体错误案例
帮助理解模型的能力边界

价值：针对性优化，提升改进效率
```

### 6.2 设计不足

#### 不足一：评测集管理较弱
```
现状：简单的上传和选择
缺乏：
- 版本控制
- 数据统计
- 质量检查
- 协作编辑

改进方向：完善评测集生命周期管理
```

#### 不足二：人工评测支持不足
```
现状：自动化指标为主
缺乏：
- 人工标注界面
- 众包评测支持
- A/B测试框架

改进方向：增强人工评测和主观评价能力
```

#### 不足三：与生产环境脱节
```
现状：离线评测为主
缺乏：
- 在线效果监控
- 真实用户反馈收集
- 效果退化告警

改进方向：建立从离线评测到在线监控的闭环
```

## 七、总结

### 核心价值

模型评测的核心价值是**"让效果可量化、可对比、可优化"**。

它解决了AI应用开发中的关键问题：
1. **选型有据**：模型选择不再凭感觉，而是看数据
2. **优化有向**：知道哪里不好，才能针对性改进
3. **质量可控**：持续监控，确保效果不退化

### 设计哲学

```
从"黑盒试错"到"数据驱动"

传统方式：                    DB-GPT评测：
─────────                    ───────────
换模型试试效果                评测对比选择最优
    │                              │
调Prompt看感觉                量化评估找到最佳版本
    │                              │
上线后不知道效果如何           持续监控及时发现问题
    │                              │
被动响应问题                  主动优化迭代
```

### 使用场景总结

| 场景 | 使用方法 | 预期收益 |
|------|----------|----------|
| 模型选型 | 多模型对比评测 | 选择最适合的模型，节省成本 |
| Prompt优化 | 多版本效果对比 | 找到最优Prompt，提升效果 |
| 参数调优 | 网格搜索+评测 | 找到最佳参数组合 |
| 效果监控 | 定期自动评测 | 及时发现问题，保证质量 |
| 上线前验证 | 全量评测集测试 | 避免线上故障 |
