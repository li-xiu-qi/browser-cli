# 开源项目目录结构分析

对比 RAGFlow、DB-GPT、Vanna 三个 AI 数据分析项目的代码组织方式。

---

## 整体架构对比

| 项目 | 后端技术 | 前端技术 | 架构风格 |
|------|---------|---------|---------|
| RAGFlow | Python Flask | React + Ant Design | 模块化单体架构 |
| DB-GPT | Python FastAPI | Next.js | Monorepo + 插件化架构 |
| Vanna | Python 库 | Web Components | 核心库 + 多前端适配 |

---

### RAGFlow 结构特点

1. **垂直拆分**：按功能域划分目录，如 agent、rag、memory、deepdoc 等
2. **前后端分离**：api/ 和 web/ 完全独立
3. **模块化设计**：每个核心功能有独立目录，内部按职责分层
4. **丰富的集成层**：mcp、intergrations 支持多种外部协议

### DB-GPT 结构特点

1. **Monorepo 架构**：核心功能拆分为多个独立包，位于 packages/
2. **包命名规范**：dbgpt-{功能} 的命名方式，职责清晰
3. **混合路由**：web/ 同时包含 App Router 和 Pages Router
4. **插件化设计**：examples/ 展示丰富的扩展点

3. **轻量级结构**：目录扁平，没有复杂的分层
4. **以功能组织**：agents、tools、integrations 等按功能类型划分

## Agent 框架对比

### DB-GPT Agent 框架

DB-GPT 使用**自研 Agent 框架**，非 LangChain、非 AutoGen，而是独立实现的对话式 Agent 系统：

```
packages/dbgpt-core/src/dbgpt/agent/
├── core/
│   ├── agent.py              # Agent 抽象接口（send/receive/generate_reply）
│   ├── base_agent.py         # ConversableAgent 基类
│   ├── agent_manage.py       # AgentManager 管理器
│   ├── action/               # Action 动作系统
│   ├── memory/               # Agent 记忆系统
│   ├── plan/                 # 规划系统（支持 AWEL 编排）
│   └── profile/              # Agent 画像配置
├── expand/                   # 预置 Agent 实现
│   ├── tool_assistant_agent.py
│   ├── code_assistant_agent.py
│   ├── dashboard_assistant_agent.py
│   ├── data_scientist_agent.py
│   └── simple_assistant_agent.py
└── resource/                 # 资源管理
    ├── base.py               # 资源基类
    └── mcp.py                # MCP 资源接入
```

**核心特点**：
- 面向对话设计：Agent 通过 `send`、`receive`、`generate_reply` 进行消息传递
- 多 Agent 协作：支持 `UserProxyAgent` 与业务 Agent 对话
- 与 AWEL 深度集成：Agent 可以作为 AWEL 工作流的一个节点
- 资源驱动：通过 Resource 机制接入 MCP、数据库等外部能力

### RAGFlow Agent 框架

RAGFlow 也使用**自研 Agent 框架**，基于 JSON 模板配置驱动：

```
ragflow/agent/
├── component/
│   └── agent_with_tools.py   # Agent 组件定义
├── templates/                # 预置模板（JSON 配置）
│   ├── deep_research.json    # 深度研究模板
│   ├── customer_service.json # 客服系统模板
│   ├── knowledge_base_report.json
│   └── deep_search_r.json
└── ...
```

**核心特点**：
- 模板驱动：通过 JSON 文件定义多 Agent 协作流程
- 角色分工：支持 Lead Agent 分配任务给 Sub Agent
- MCP 原生支持：模板中直接调用 MCP 工具
- 代码执行：内置 sandbox 执行生成的代码

| 特性 | DB-GPT Agent | RAGFlow Agent |
|------|--------------|---------------|
| 框架类型 | 自研 | 自研 |
| 设计风格 | 面向对话 | 模板配置驱动 |
| 多 Agent | 支持 | 支持（JSON 定义） |
| 编排能力 | AWEL 深度集成 | 模板内置编排 |
| MCP 支持 | MCPToolPack | 原生模板支持 |
| 记忆系统 | GptsMemory | 内置会话管理 |

## 关键设计决策分析

### RAGFlow 的设计选择

1. **垂直模块化**：按业务功能垂直切分，每个模块内部自治
2. **独立管理后台**：admin/ 单独维护，与主应用解耦
3. **沙箱隔离**：sandbox/ 目录确保代码执行安全
4. **文档引擎独立**：deepdoc/ 作为独立模块，可单独迭代
5. **Agent 模板化**：通过 JSON 模板快速构建多 Agent 工作流

### DB-GPT 的设计选择

1. **Monorepo 治理**：复杂功能拆包，简化依赖管理
2. **核心与扩展分离**：dbgpt-core 保持精简，dbgpt-ext 承载扩展
3. **双文档策略**：docs/ 和 examples/ 分别服务不同场景
4. **混合部署**：docker/ 支持多种部署形态

### Vanna 的设计选择

1. **库化优先**：核心是一个可被集成的库，非独立应用
2. **前端解耦**：frontends/ 可选，用户可自建前端
3. **教学驱动**：notebooks/ 是重要的入门路径
4. **轻量治理**：目录扁平，降低理解成本

## 可借鉴的实践

### 推荐采用的组织方式

1. **模块化垂直切分**：RAGFlow 的 agent/、rag/、memory/ 方式
2. **Monorepo 包拆分**：DB-GPT 的 packages/dbgpt-* 方式，适合大型项目
3. **前后端独立版本**：Vanna 的 frontends/ 解耦方式
4. **配置集中管理**：RAGFlow 的 conf/ 或 DB-GPT 的 configs/

### 需根据场景选择的方式

| 场景 | 推荐参考 |
|------|---------|
| 大型企业级应用 | RAGFlow 的垂直模块化 |
| 多团队协作 | DB-GPT 的 Monorepo |
| 开源工具库 | Vanna 的库优先设计 |
| 快速迭代 | Vanna 的轻量结构 |
| 复杂部署场景 | RAGFlow 的 helm/ + docker/ |
