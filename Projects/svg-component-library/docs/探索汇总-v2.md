### 关键认知升级

**SVG 不是万能的**

经过6个场景的探索，发现 SVG 的适用边界：

| 场景 | 推荐工具 |
|------|----------|
| 技术架构图 |  SVG |
| PPT 配图 |  SVG |
| 文章配图 |  SVG |
| 时间线 |  SVG |
| 数据图表 | ⚠️ ECharts/D3 |
| **流程图** |  **Graphviz** |

**教训**：
1. 技术选型要考虑工具成熟度
2. 不要重复造轮子（Graphviz 已经做得很好）
3. SVG 适合静态视觉，不适合复杂逻辑图

## 本次新增问题（自查发现）

### 数据图表 (explore-data-chart.svg)

**问题1：Q4 数值标签 y 坐标为负数**
- 修复：`<text y="-10">` → `<text y="15">`
- 教训：避免使用负数坐标

**问题2：图例与图表区域重叠**
- 修复：图例位置从 x=550 移到 x=650，增大间距
- 教训：图例应放在图表区域外部，避免与数据元素重叠

**问题3：数值标签与柱形/其他元素重叠**
- 修复：调整所有数值标签的 y 坐标，确保在柱形上方清晰显示
- 教训：数值标签需要与柱形顶部保持适当间距

### 流程图 (explore-decision-flow.svg) - **重要结论**

**结论：SVG 不适合做复杂流程图**

**原因分析**：
1. **坐标计算复杂** - 返回线、分支线需要精确计算相对坐标，极易出错
2. **维护成本高** - 流程调整需要重新计算大量坐标
3. **可读性差** - SVG 代码难以直观理解流程逻辑
4. **已有更好工具** - Graphviz 专为此设计，自动布局、语法简洁

**建议**：
-  简单线性流程（3-5步）：可用 SVG
-  复杂分支流程（决策+循环）：使用 Graphviz
-  Graphviz 导出 SVG 后嵌入：最佳方案

**替代方案**：
```
Graphviz DOT → 自动布局 → 导出 SVG → 嵌入文档
```

**教训**：选择技术方案时要考虑工具成熟度，不要重复造轮子
