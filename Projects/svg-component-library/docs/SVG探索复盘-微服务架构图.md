## 项目背景

**项目**：微服务架构SVG配图探索  
**目标**：验证SVG方法论在文章配图场景的适用性  
**时间**：2026-02-06  
**结果**：发现4个严重问题，全部修复

## 一、成败认定

### 整体判断
**⚠️ 低于预期**（但学习价值高）

理由：
-  发布后才发现多个明显问题
-  连接线逻辑错误、图标画反、重叠等低级错误
-  但最终全部修复，沉淀了经验

### 分层判断

| 维度 | 目标 | 实际 | 判定 |
|------|------|------|------|
| **视觉完成度** | 无重叠、比例协调 | 多处重叠、透明度问题 | 低于预期 |
| **架构准确性** | 正确反映微服务关系 | 网关连接线逻辑错误 | 低于预期 |
| **代码质量** | 无低级错误 | 贝塞尔曲线方向错误 | 低于预期 |
| **问题修复速度** | 快速迭代 | 发现一处修复一处 | 符合预期 |

### 问题详情还原

#### 问题1：网关连接线逻辑错误
**用户反馈**："微服务这个图是有问题的，你看看能直接发现吗？"

**实际代码**：
```svg
<line x1="300" y1="240" x2="300" y2="300"/>
```

**问题**：只有一条垂直线，但网关需要路由到三个服务（x=200, 400, 600）

---

#### 问题2：用户图标身体画反
**发现过程**：用户提示"继续看看"后检查发现

**实际代码**：
```svg
<path d="M-20,25 Q-20,5 0,5 Q20,5 20,25"/>
<!-- 控制点在y=5，形成向上凸的山丘 -->
```

**问题**：画成了向上凸的弧线（像山丘），但人的肩膀应该是向下凹的

---

#### 问题3：标签透明度过低+间距太小
**用户反馈**："那三个绿色圆圈你看看吧，是有问题"

**实际代码**：
```svg
<rect fill="#10b981" opacity="0.2"/>  <!-- 太淡 -->
<g transform="translate(170, 520)">    <!-- 间距10px -->
```

**问题**：
- opacity="0.2" 在深色背景上几乎看不见
- 三个标签间距10px，视觉上连成一片

---

#### 问题4：图例和标签重叠
**用户反馈**："有重叠"

**实际代码**：
```svg
<!-- 图例 -->
<g transform="translate(650, 520)">
  <ellipse cy="35"/>  <!-- y=520+35=555 -->
  <text y="70"/>      <!-- y=520+70=590 -->

<!-- 标签 -->
<g transform="translate(60, 520)">
  <rect height="30"/> <!-- y=520~550 -->
```

**问题**：图例的第二个元素在y=555，标签在y=520~550，只差5px重叠

## 三、得失分析

### 做得好的 

#### 1. 快速修复能力
**发现**：发现问题后能快速定位并修复  
**原因**：代码结构清晰，组件化设计便于定位  
**假设**：良好的代码组织能提高修复效率

#### 2. 文档记录习惯
**发现**：每个问题都记录在文档中  
**原因**：建立了复盘意识和文档习惯  
**价值**：形成可复用的避坑指南

### 做得不好的 

#### 1. 缺乏自检清单
**问题**：四个明显问题都需要用户发现  
**根因分析**：
- 没有建立SVG发布前的自检流程
- 过于依赖视觉检查，没有系统方法
- 急于交付，没有冷静审视

**真问题**：缺乏"检查清单"的执行纪律

**深层追问**：
- 为什么连连接线逻辑都没检查？→ 只看了"有没有线"，没看"线连到哪"
- 为什么图标画反了没发现？→ 缺乏基本的美术常识（肩膀形状）
- 为什么重叠没发现？→ 没有计算边界框的习惯

#### 2. 对"完成"的定义不清晰
**问题**：我以为"画出来了"就是完成  
**真问题**：完成 = 视觉正确 + 逻辑正确 + 无重叠 + 符合场景

#### 3. 没有使用框架验证
**问题**：没有先用虚线框占位验证布局  
**根因**：觉得是小图，直接上手画  
**后果**：多个元素位置冲突

## 四、总结提炼

### 认知共识

| 类型 | 内容 |
|------|------|
| **完成标准** | SVG完成 = 视觉正确 + 逻辑正确 + 无重叠 + 符合场景 |
| **做事标准** | 发布前必须使用检查清单逐项验证 |
| **做对了** | 组件化设计、快速修复、文档记录 |
| **做错了** | 无自检清单、急于交付、跳过框架验证 |

### SVG配图检查清单（v1.0）

#### 发布前必须检查

**□ 逻辑正确性**
- [ ] 所有连接线指向正确的目标
- [ ] 架构关系符合实际（如网关需要分发线）
- [ ] 图标方向正确（人体曲线、箭头方向）

**□ 视觉正确性**
- [ ] 半透明元素在背景上可见（opacity≥0.3）
- [ ] 相邻元素间距≥15px
- [ ] 文字与背景对比度足够
- [ ] 图标比例协调（不过大/过小）

**□ 布局正确性**
- [ ] 无重叠元素（计算边界框）
- [ ] 层次关系清晰（前景/背景）
- [ ] 对齐统一（左对齐/居中对齐）

**□ 代码正确性**
- [ ] 无全角括号（）
- [ ] 所有引号闭合
- [ ] viewBox与width/height匹配

### 行动计划

#### 坚持做 
- [ ] 使用组件化设计（<defs>+<use>）
- [ ] 每个问题都记录到文档
- [ ] 快速迭代修复

#### 放弃做 
- [ ] 不经过检查清单就发布
- [ ] 跳过框架验证直接画
- [ ] 凭感觉判断"完成了"

#### 新开始 🆕
- [ ] 建立SVG配图检查清单（已完成v1.0）
- [ ] 每次发布前强制逐项检查
- [ ] 复杂图先用虚线框占位

### 过程资产

#### 1. SVG配图检查清单
位置：`docs/SVG配图检查清单.md`

#### 2. 常见错误速查表
| 错误 | 症状 | 检查方法 |
|------|------|----------|
| 连接线逻辑错误 | 线没有连到目标 | 追踪每条线的起点和终点 |
| 图标方向错误 | 曲线方向不对 | 对比常识（肩膀向下） |
| 透明度过低 | 元素看不见 | 在不同背景上测试 |
| 重叠 | 元素堆叠 | 计算每个元素的x,y,width,height |

#### 3. 贝塞尔曲线口诀
```
二次贝塞尔：Q控制点 终点
- 控制点在上方 = 向上凸 ⛰️
- 控制点在下方 = 向下凹 
```
