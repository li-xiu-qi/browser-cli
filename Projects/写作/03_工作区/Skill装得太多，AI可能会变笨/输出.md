# Skill 装得太多，AI 可能会变笨

## 一、 我想让它变强，结果它变慢了

前段时间，我特别沉迷于给 AI 加各种 Skill。

我想着多给它一些能力，它能做的事情就更多。

于是我把能找到的 Skill 全都塞进了配置文件里。联网搜索、网页爬取、代码运行、PDF 读取、绘图、数据分析。

我以为它会变得无所不能。

结果我发现，它**反而变得反应迟钝，经常出错**。

它开始选错工具，甚至经常无视我最关键的指令。

我开始反思，是不是我们对 Skill 的理解出了问题. 我们以为 Skill 越多越好，但其实装多了会有反效果。

## 二、 例子和指令冲突了，AI 会听例子的

这是我发现的第一个严重问题。

举个具体的例子。

我在 Skill 里写了一条明确的指令：写代码时，一定要优先使用 LiteLLM 库，不要使用 OpenAI SDK。

但是，我在同一个 Skill 的“示例代码”部分，为了图省事，贴了一段以前写的代码。那段代码里用的偏偏就是 OpenAI SDK。

结果你猜着？

AI 忽略了我的文字指令，直接照着示例代码，给我写了一个用 OpenAI SDK 的方案。

在大模型的眼里，**具体的例子往往比抽象的规则权重更高**。

它看到你给的例子是那样写的，它就默认那样写是对的。

如果你塞了太多 Skill，而这些 Skill 里的示例代码不够规范，或者和你的主指令有冲突。AI 就会陷入混乱。它会优先模仿例子，而不是遵守规则。

## 三、 输入的风格，会影响输出的风格

这是第二个问题，也是更隐蔽的一个问题。我把它叫做“内容趋同现象”。

有一次，我要求 AI：写文章要平实，不要使用比喻。

但是我的提示词写得很花哨，里面用了好几个比喻。

结果 AI 写出来的东西，依然满篇都是比喻。

为什么？因为**大模型本质上是一个“模仿者”**。它会通过分析你输入的文字，来判断当下的语境和风格。

如果你输入的文字是文绉绉的，它就会觉得现在的语境就是文绉绉的。即使你命令它“说人话”，它也会默认受到你输入风格的影响。

既然你输入的文字风格是复杂的，模型就会模仿这种复杂。它会觉得当下的语境就是需要这种风格，从而忽略你“不要用比喻”的指令。

Skill 也是一段文字。如果你的 Skill 写得啰嗦、浮夸，或者风格和你想要的结果不一致。这些文字都会进入模型的上下文，污染它的生成风格。

## 四、 选项太多，AI 往往会选错

Skill 越多，AI 变笨的第三个原因，是选择太多。

举个我自己的亲身经历。

我在用 gemini-cli 的时候，它自带了一个“网页获取”的功能，速度很快，但功能比较基础。

我自己又写了一个功能更强的 Skill，是用 Playwright 驱动的。这个工具虽然慢点，但专门用来处理那些有反爬虫机制的网站。

我本意是想让它灵活切换：遇到简单的网页用自带的，遇到难处理的网页用我写的。

结果呢？AI 经常直接调用自带的工具。

哪怕我给它一个明显有反爬机制的链接，它还是优先去调那个自带的简单工具。结果当然是报错、被拦截，啥也读不出来。

这时候它就报错了，告诉我读不到内容。它完全忘记了，我给它装的工具箱里，还躺着一个专门解决这个问题的工具。

**工具多了，AI 就容易选错**。它往往会选择路径最短、最常见的那个（通常是自带的），而不是效果最好的那个（你专门装的）。结果就是，本来能搞定的反爬任务，因为选错了工具而失败了。

## 五、 文件夹里的无关文件，占用了上下文

除了上面说的，我还发现了一个更隐蔽的技术坑。

Skill 不仅仅是一个文档，它往往是一个文件夹。

有一次，我写了一个基于 Node.js 的爬虫脚本作为 Skill。我觉得这个功能很强。

但加载进去之后，AI 完全处理不过来了。

后来我才意识到，Skill 加载的时候，**系统会把整个文件夹的路径结构都作为上下文扔给 AI**。

我的文件夹里有一个 node_modules 目录。懂代码的都知道，这里面可能有成千上万个小文件。

这一大堆文件名，瞬间就把上下文塞满了。

虽然系统有截断机制，不会真的把几万个文件都读进去。但问题就出在这个截断上。

因为前面的无效信息太多，系统为了省流，直接把后面真正重要的 SKILL.md 给切掉了。

这就导致真正重要的指令文件被挤到了一大堆无关的文件名后面。系统为了节省空间，可能会把后面的指令切掉，AI 根本就没读到你的核心要求。

所以，Skill 的文件夹必须干净。不要把无关的依赖库、临时文件都放进去。

## 六、 别用废话去指导 AI

还有一个坑，是我们自己造成的。

有时候我们懒。我们不想自己写 SOP，就让 AI 帮我们生成一个。

AI 生成的东西，乍一看很专业。排版整齐，用词考究。但你仔细读读，**全是正确的废话**。

本来一句话能说清楚的操作，AI 非要扩展成五句。

如果你把这种冗余的内容当成 Skill 喂给 AI，这就是一种污染。

高密度的信息能让模型变聪明，低密度的废话只会让模型变啰嗦。

你用废话去指导 AI，AI 自然也就回馈你一堆废话。

真正的 Skill，应该是经过你反复验证总结出来的干货，而不是另一堆 AI 生成的低质量内容。

## 七、 无法中途清除

还有一个麻烦事，很多人都不知道。

我之前以为，Skill 就像手机 App，不想用了关掉就行。

虽然 claude code 不开源，但我专门去查了查 gemini-cli 这些工具的底层逻辑，发现不是这么回事。

**一旦模型在对话里用过一次 Skill，那个 Skill 的具体指令就已经写进对话历史里了**。

对于模型来说，历史记录是只增不减的。

即使你在后台把那个 Skill 禁用了，模型依然记得刚才读过的内容。它没法把这段历史记录删掉。

你想让它彻底忘掉那个干扰你的 Skill，唯一的办法就是关掉窗口，全部重来。

这就意味着，在一个长对话里，你加载过的 Skill 越多，留在历史记录里的干扰信息就越多，你是没法半路给它减负的。

## 八、 少装点 Skill，效果反而更好

所以，我现在用 AI，开始做减法了。

我不再追求大而全的 Skill 库。我开始通过场景来拆分。

写代码的时候，我只加载代码相关的 Skill，把绘图和写作的 Skill 关掉。

写文章的时候，我把代码 Skill 关掉，只留下搜索和文档读取。

而且，我开始重新审查每一个 Skill 的内容。

检查例子：所有的示例代码，必须和我的核心指令完全一致。

检查风格：Skill 的描述文字，必须简单、直接、无歧义。

清理文件夹：把 node_modules 这种会对上下文造成比较大影响的内容清理出去，只保留核心脚本。

亲自写 SOP：不要偷懒用 AI 生成的套话，要写你自己验证过的经验。

在这个 AI 时代，给模型加功能谁都会。

但只有**懂得克制，懂得做减法，才能让 AI 保持清醒和高效**。
