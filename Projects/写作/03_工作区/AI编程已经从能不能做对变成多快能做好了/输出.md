## 一、被忽视的评测维度：任务吞吐量

### 1.1 AI模型评测的盲区

当前大模型评测存在一个普遍盲区：**大家都在比能不能做对，却很少认真比多快能做对**。

打开各大模型的官方宣传页，满屏都是 benchmark 分数。这些数据固然重要，但它们回答的是一个能力问题：这个模型够不够聪明？

![Step-3.5-Flash 与顶级模型 benchmark 对比：在 AIME、IMO、LiveCodeBench、SWE-bench 等多个 benchmark 上与 DeepSeek-V3.2、Gemini、Claude、GPT 等顶级模型对比，展现第一梯队实力](assets/step-3.5-flash-figure1-frontier-intelligence.jpg)

可对于每天和AI协作的开发者来说，更紧迫的是一个效率问题：我提交一个需求，要等多久才能看到结果？

这背后其实是一个更深层的理念分歧：我究竟需要什么样的AI模型？

当前大模型正处于从单纯追求逻辑深度的“L2 Reasoner”向强调任务交付的“L3 Agent”转型的关键期。在 Agent 工作流中，速度不再仅仅关乎体感，更是核心生产力——毕竟用户只想要结果，没人愿意盯着屏幕等 token 逐字蹦出。

这种对“快”的极致需求，Step-3.5-Flash 正是一个专门在速度上有做优化的模型。但我们的疑问也随之而来：这种速度的提升，究竟是架构优化的红利，还是以牺牲质量为代价的妥协？这正是我在实测中要验证的核心问题：Step-3.5-Flash 的速度优势，是否建立在妥协质量的基础上？

### 1.2 真实开发场景 ≠ 考试

考试是一锤定音的：题目发下来，你写答案，交卷，得分。答案对了，任务完成。

但真实的开发工作从来不是这样。尤其是前端开发的场景，颜色，样式，交互的效果，都需要不断多次修改。

在迭代式开发中，迭代次数决定最终质量，而不是单次生成质量。

> 只要有偏好，就有对齐，再聪明的模型也不可能一次性猜对你的需求。
>
> 这意味着迭代不是例外，而是常态。选择一款能让我快速迭代的模型，就是选择了更高效的工作方式。

### 1.3 多数人都忽略了任务吞吐量

根据我的观察，在众多的AI coding群里面，大家很喜欢聊，哪个哪个最强的模型太慢了，**但是从来没有考虑过任务吞吐量指标，只是一味的说更强的模型的任务完成率会更高**，但是我想要强调的是任务完成率不代表任务完成的数量就多。

> 如果 Step-3.5-Flash 用 25 秒生成一个 85 分的版本，Kimi-K2.5 用 490 秒生成一个 90 分的版本，
> 在同样 1 小时内，谁能帮你探索更多可能性？谁能让你保持思路连贯？

这个问题的答案，指向一个被严重低估的指标：任务吞吐量。

定义：单位时间内完成的有效任务迭代次数。任务吞吐量 = 3600秒 / (模型生成时间 + 人工检查/调整时间)


假设每次迭代中，人工检查输出、提出修改意见需要 30 秒，以数据仪表盘任务的实测平均时间计算：

| 模型             | 平均生成时间 | 单轮总时间 | 每小时可完成迭代 |
| -------------- | ------ | ----- | -------- |
| Step-3.5-Flash | ~25s   | 55s   | ~66 轮    |
| GLM-4.7        | ~110s  | 140s  | ~26 轮    |
| DeepSeek-V3.2  | ~230s  | 260s  | ~14 轮    |
| Kimi-K2.5      | ~490s  | 520s  | ~7 轮     |

*数据来源：数据仪表盘任务 Think 与 NoThink 双模式实测平均值，每小时可完成迭代采用四舍五入的方式计算。*

64 轮 vs 7 轮，相差 9.1 倍，更多迭代意味着更多试错机会，最终产物质量会显著更高。

> 任务成功率是一种成本，时间成本也是成本。
>
> 传统评测只关注成功率，即能不能做对，但忽略了时间成本，即多快能做对。在真实工作场景中，这两个成本同样重要。

行业趋势也在印证这一点。2026年2月，Anthropic 为 Opus 4.6 推出了 Fast Mode——同样的模型质量，仅通过优化推理基础设施来提升速度，但定价从标准的 $5/$25 per MTok 飙升到 $30/$150 per MTok，足足贵了 6 倍；当上下文超过 200K tokens 时，价格更是从 $10/$37.5 涨到 $60/$225。Anthropic 明确建议用户在"快速迭代"场景下开启它，哪怕成本显著更高。

**速度，正在被重新定价**。但前提是推理质量能得到保证——否则再快也没有意义。Step-3.5-Flash 的独特之处在于，它不需要你支付这笔溢价，就能在质量有保障的前提下实现速度优势。

![Anthropic Fast Mode 官方文档：说明 Fast Mode 适用于快速迭代代码修改、实时调试会话等场景，代价是更高的成本](assets/anthropic-fast-mode-documentation.png)

此外，速度还影响心流。当等待时间超过 30-60 秒，人的注意力就会自然分散。20 多秒等待可以保持专注，490 秒等待则会让人忍不住刷手机，所以快不只是省时间，而是保持创造力。

## 二、前端实战：数据仪表盘

### 2.1 任务设定

这是一个典型的图表可视化任务，也不算特别复杂，因为没有复杂的交互逻辑。我使用的提示词如下：

> 请用HTML/CSS/JavaScript实现一个美观的数据仪表盘。要求：1. 包含至少3种图表类型（折线图、饼图、柱状图）；2. 使用Chart.js或类似库；3. 响应式布局，适配不同屏幕；4. 包含顶部导航栏和侧边栏；5. 配色专业、美观。【重要】请只输出完整的HTML代码，从<!DOCTYPE html>到</html>，不要包含任何解释、说明或markdown代码块标记。


### 2.2 实测：各模型生成速度相差多少倍？

不同模型的 Thinking 偏好不同，导致 Think 模式下的耗时差异巨大。为了公平对比，我同时测试了 NoThink（非思考模式）和 Think（思考模式）两种模式：

| 模型             | NoThink (时间/Tokens/速度)    | Think (时间/Tokens/速度)       | 相对速度* | 每小时迭代 |
| -------------- | ------------------------- | -------------------------- | ----- | ----- |
| Step-3.5-Flash | 26.2s / 9,236 / 352.8 t/s | 25.5s / 9,353 / 367.3 t/s  | 1x    | ~138次 |
| GLM-4.7        | 119.6s / 7,337 / 61.3 t/s | 101.2s / 7,303 / 72.2 t/s  | 4.6x  | ~30次  |
| DeepSeek-V3.2  | 212.6s / 6,236 / 29.3 t/s | 251.3s / 7,300 / 29.0 t/s  | 8.1x  | ~17次  |
| Kimi-K2.5      | 235.0s / 6,219 / 26.5 t/s | 761.5s / 16,661 / 21.9 t/s | 9.0x  | ~15次  |

这里的测试是以 NoThink 时间为基准，因为 NoThink 模式下可以直接生成代码，产出即有效结果，无需等待推理过程，还有就是不同的模型 think 的程度我无法有效控制所以主要还是以 think 产出作为基准。
从表格的数据对比可以看出，Step-3.5-Flash 速度领先明显，352.8 tokens/s 是 Kimi-K2.5 的十多倍，think、no think生成时间平均快 9 倍。

### 2.3 效果展示：你更愿意用哪一版？

![4个模型生成的数据仪表盘对比：上排为 Thinking 模式，下排为 NoThinking 模式](assets/interact-gifs/comparison_data_dashboard_final.gif)

从实际测试结果来看，各模型在数据仪表盘这个任务上都实现了功能完整、布局合理的版本。Step-3.5-Flash 的配色协调、三个图表正常显示数据，整体效果完全可用，在单模态代码生成能力上不弱于 Kimi-K2.5。

值得注意的是，GLM-4.7（总参数 355B，约为 Step-3.5-Flash 的 1.8 倍）在本次测试中出现了数据渲染问题，生成的图表框架完整，但图表数据为空无法正常显示。这说明 Step-3.5-Flash 的水平至少在这一批测试里面是要胜过 GLM-4.7 的。

一个值得思考的问题是：虽然 Kimi-K2.5 的版本在渐变、阴影等视觉细节上可能略精致一些，但为了这 10-15% 的视觉差异，值得等 9 倍的时间吗？

想象这个场景：周五下午，老板临时说："帮我做个数据看板，周一给客户演示"。你需要快速出 3 版不同风格的方案让他选。用 Step-3.5-Flash，**你可以在 2 分钟内把 3 个版本摆在老板面前，然后悠悠地喝杯咖啡。用 Kimi-K2.5，你可能还在等第二个版本生成，期间忍不住刷了 5 分钟手机，甚至 AI 运行完了你还在刷，写代码的思路早就断了。**

## 三、前端实战：生命游戏

### 3.1 任务设定

生命游戏是一个经典的细胞自动机，算法逻辑标准：每个细胞根据周围 8 个细胞的状态决定下一代生死。

但实际实现时，如果提示词没有详细描述需求，很多细节需要模型自行决定：动画流畅度、颜色选择、按钮布局等。我使用的提示词如下：

> 请用HTML/CSS/JavaScript实现一个完整的康威生命游戏（Conway's Game of Life）。要求：1. 使用HTML5 Canvas渲染细胞网格；2. 实现开始/暂停/重置按钮；3. 支持随机生成初始状态；4. 网格大小：50x50；5. 动画流畅，可以调节速度。【重要】请只输出完整的HTML代码，从<!DOCTYPE html>到</html>，不要包含任何解释、说明或markdown代码块标记。

下面是本次测试单次生成的时间消耗：

| 模型             | NoThink (时间/Tokens/速度)    | Think (时间/Tokens/速度)      | 相对速度* | 每小时迭代 |
| -------------- | ------------------------- | ------------------------- | ----- | ----- |
| Step-3.5-Flash | 21.2s / 6,738 / 317.1 t/s | 17.4s / 5,774 / 331.6 t/s | 1x    | ~166次 |
| GLM-4.7        | 79.3s / 5,449 / 68.8 t/s  | 98.8s / 5,553 / 56.2 t/s  | 4.7x  | ~35次  |
| DeepSeek-V3.2  | 193.5s / 5,657 / 29.2 t/s | 173.2s / 5,451 / 31.5 t/s | 9.3x  | ~19次  |
| Kimi-K2.5      | 151.9s / 4,486 / 29.5 t/s | 243.1s / 4,207 / 17.3 t/s | 10.0x | ~17次  |

Step-3.5-Flash 再次以大概 20 秒的成绩大幅领先。值得注意的是，生命游戏的代码量比数据仪表盘更少，所以整体生成时间比数据仪表盘任务略短，但各模型之间的相对差距保持一致。

## 四、逻辑推理测试

除了前端编程任务，我还测试了逻辑推理题来验证模型的能力边界。最初我选了5道不同的逻辑题：

1. 过河问题：经典路径规划
2. 烧绳计时：反直觉思维
3. 囚徒帽子：编码策略与信息论
4. 海盗分金：逆向归纳推理
5. 现代控制综合题：状态转移矩阵与系统响应，自动化大三下学生水平

经过预测试，我发现前几道题几乎所有模型都能答对，难以区分能力差异。最终我选择了海盗分金和现代控制综合题作为正式测试题目。**前者能难倒 DeepSeek-V3.2 非 think 模式，后者是我从网上找到的一道大学期末考试真题**，用来验证大模型是否能达到大学生真实考试实战水平，而非只会做榜单上的"博士级"难题。
> 没有选博士级的题目是因为大多数人可能也看不懂，也难以验证是否正确。

### 4.1 这两道逻辑题在考什么？

题目一：海盗分金（逆向归纳推理）

> 5个海盗分100金币，编号1-5，按编号顺序轮流提案。规则：1. 提案需要至少半数同意才能通过（含提案者本人）；2. 提案者被扔下海后，由下一个编号的海盗提案；3. 所有海盗优先级：保命 > 多拿钱 > 看别人死。问：1号海盗应该如何分配金币才能既保命又拿最多？请详细说明推理过程。

正确答案：1号分配 (98, 0, 1, 0, 1)
- 自己98枚，2号0枚，3号1枚，4号0枚，5号1枚
- 获得1号、3号、5号共3票通过，超过半数2.5

推理过程（逆向归纳）：
- 剩5号一人：5号得100
- 剩4、5号：4号需1票（自己），分配(100, 0)
- 剩3、4、5号：3号需2票，争取5号（5号在上一轮得0），分配(99, 0, 1)
- 剩2、3、4、5号：2号需2票，争取4号（4号在上一轮得0），分配(99, 0, 1, 0)
- 剩1-5号：1号需3票，争取3号和5号（他们在上一轮都得0），分配(98, 0, 1, 0, 1)

题目二：现代控制原理综合应用题

![现代控制原理综合应用题：包含题目描述、特征根、状态转移矩阵和系统输出答案](assets/modern_control_system_problem.png)

这道题综合考察了特征值计算、拉普拉斯变换、状态转移矩阵求解、零输入/零状态响应分解、卷积积分等多个知识点，是一道典型的现代控制理论综合应用题。

### 4.2 各模型答题表现对比

海盗分金测试结果：

| 模型 | NoThink 耗时 | Think 耗时 | 答案 | 是否正确 |

### 4.3 快且准：Step-3.5-Flash的逻辑推理表现

Step-3.5-Flash 在海盗分金和现代控制综合题上表现突出。think 和 no think 模式基本都在60秒左右完成海盗分金的逆向归纳推理，think 模式下25.8秒算出控制题的特征根、状态转移矩阵和系统输出，答案完整准确。更重要的是，即使在 NoThink 模式下，Step-3.5-Flash 也能直接给出正确答案，不需要依赖深度思考模式。

相比之下，Kimi-K2.5 虽然也能答对，但耗时是 Step-3.5-Flash 的 3-4 倍。DeepSeek-V3.2 则出现了分化：NoThink 模式下海盗分金给出次优解 (97,0,1,0,2)，必须开启 Thinking 模式才能答对。

这说明 196B/11B 的小模型通过充分的后训练，在逻辑推理上完全可以媲美 1T 参数的大模型，而且不需要在快和对之间做权衡。Step-3.5-Flash 的 NoThink 模式本身就是可靠的，不需要被迫等待 Thinking 模式的冗长推理。

## 五、结论与建议：快且好，才是真的好

经过上面的测试，我的发现是：Step-3.5-Flash 用 1/8 的时间达到了与千亿参数模型同等的效果。

![OpenRouter 平台上 Kimi-K2.5、DeepSeek-V3.2、GLM-4.7 的定价与性能对比：Step-3.5-Flash 在 OpenRouter 上免费提供](assets/openrouter_pricing_comparison.png)

| 维度         | Step-3.5-Flash vs Kimi-K2.5 | 评论                              |
| ---------- | --------------------------- | ------------------------------- |
| 速度         | 快 4-20 倍                    | 前端 25秒 vs 490秒，逻辑题 60秒 vs 263秒  |
| 质量         | 基本等价                        | 前端功能相同，逻辑题答案相同                  |
| 吞吐量        | 每小时 66 轮 vs 7 轮             | 迭代次数决定最终产物质量                    |
| 输出 token成本 | 免费 vs $2.25/百万token         | 隐性成本（时间/注意力/机会）差距更大             |
| 部署         | 196B vs 1000B               | 4-bit 量化仅需 ~98GB 显存，降低 80% 硬件成本 |
*上述表，前端时间数据来自数据仪表盘任务的实测平均时间计算，逻辑题时间来自海盗分金测试题 think 部分*

选对模型比选最聪明的模型更重要。Step-3.5-Flash 在单模态代码生成上能力不弱于 Kimi-K2.5，速度还快 9 倍左右。既然效果相当，何必等待更长时间？

我觉得单模态的场景，**Step-3.5-Flash 无论是什么任务上基本是完全能够平替各家的顶级模型的**，Step-3.5-Flash本身就是顶级模型的第一梯队，除了多模态场景可能不如各家的多模态模型，但是单模态场景，Step-3.5-Flash 的速度这么快，效果还如此优秀，我没有理由拒绝。

我觉得编程的时候不能忽略时间的成本，所以我仍然推荐不同场景使用不同的模型，这不仅仅是经济上的考虑，还有时间上的成本考虑。关注隐性成本：时间、注意力、机会成本往往比 API 费用更贵。


> 快且好优于慢而完美。当你的任务可以通过快速迭代完成时，选择速度。因为时间成本，也是成本！
