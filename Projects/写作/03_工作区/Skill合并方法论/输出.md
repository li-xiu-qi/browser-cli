# Skill 合并方法论：从 34 个混乱到 11 个清晰的实战复盘

> 状态：正文写作中 | 预计字数：3000-4000 字 | 目标平台：公众号/技术博客

---

## 钩子：AI 怎么突然变"笨"了

三天前，我遇到了一个怪事。

我的 AI 助手用了大半年，本来挺顺手的。我说"帮我整理这篇笔记"，它能准确调用对应的工具，格式也处理得漂漂亮亮。

但最近它开始犯糊涂了。

我说"帮我整理一下"，它问我："您是想用笔记排版工具，还是公众号排版工具，还是文本润色工具？"

我愣了一下。这三个工具明明干的是差不多的事，只是应用场景略有不同。怎么现在 AI 反而不知道选哪个了？

我打开技能目录一看，发现问题所在。

我的 Skill 从最初 5 个，不知不觉膨胀到了 34 个。

---

## 第一部分：Skill 太多，是个陷阱

### 症状 1：AI 得了选择困难症

 Skill 是怎么变多的呢？

一开始，我只是想让 AI 帮我整理笔记。于是做了一个 Skill，专门处理 Obsidian 的 Markdown 格式。

后来我发现，把文章发到公众号需要不同的排版规则。又做了一个 Skill。

再后来，语音转写的文字需要润色。又做了一个。

就这样，工具越来越多。到我发现问题的时候，已经有 34 个 Skill 了。

这就像你抽屉里的充电线。一开始只有一条，后来手机、平板、耳机、充电宝各来一条，最后 34 条线缠在一起。你想找根线，得翻半天。

AI 也一样。当你说"帮我整理一下"的时候，它面对 3 个功能重叠的 Skill，不知道该选哪个。

### 症状 2：改个规则要改三处

更头疼的是维护。

我发现标题的格式规范有问题，想统一改一下。结果发现这个规则散落在 3 个 Skill 里。

改完一处，忘了另一处。AI 输出的格式开始不一致，有时候这样，有时候那样。

这就好比你家三个房间的灯，开关分别在三个不同的地方。你想关灯，得跑三个房间。

### 症状 3：AI 的"脑子"快炸了

还有一个隐藏问题。

每个 Skill 都有自己的依赖。有的是 Node 项目，带着 node_modules。有的是 Python 项目，带着 venv。

AI 在帮你干活之前，得先加载所有 Skill 的信息。34 个 Skill，每个都带着一大堆依赖文件，AI 的上下文被这些无关信息塞满了。

这就像你让助手找一份文件，但他必须先读完整个图书馆的书才能开始找。

## 第二部分：什么时候该合并

### 四个信号

我总结了四个判断标准。如果你遇到这些情况，就该考虑合并了。

**第一个信号：总是同时使用**

如果你调用 A 的时候，90% 的概率也会调用 B，那它们就该在一起。

比如你几乎每次都是"先排版，再润色"，那这两个功能就该在同一个 Skill 里。

**第二个信号：概念重叠**

看看两个 Skill 的描述。如果它们有 50% 以上的内容在说同一件事，就该合并。

"笔记排版工具"和"公众号排版工具"，核心都是排版，只是输出格式不同。

**第三个信号：维护负担**

改一个规则要改 N 处，这就是负担。

**第四个信号：用户困惑**

如果用户经常问你"我该用哪个工具"，说明边界不清晰，需要整合。

### 这次合并的成果

我用这四个标准审视了我的 34 个 Skill，然后动手合并。

合并之前，我的 Skill 目录是这样的状态。

结果是这样的。

![Skill 合并前后对比](assets/comparison_table.png)

合并逻辑如下图所示。

![合并逻辑](assets/merge_logic.svg)

34 个 Skill，变成了 11 个 Skill 加 2 个工作流。

具体来说。

内容排版类的三个 Skill，合并成了一个。现在叫 `content-formatter`，里面分了三个场景：笔记排版、公众号适配、转写润色。

内容转换类的七个工具，合并成了一个。现在叫 `content-converter`，里面包括 EPUB 转笔记、笔记转 Word、OCR 识别、PDF 翻译、音视频转文字、PDF 文件整理、视频转 GIF 这七个场景。

学习类的两个 Skill，合并成了 `learning-notes`。

资源获取类的两个工具，合并成了 `resource-crawler`。包括网页内容采集和视频资源下载。

文本可视化类的三个 Skill，合并成了 `text-visualization`。

知识库管理类的三个 Skill，变成了 `agent-collaboration` 里的参考文档，不再作为可调用的 Skill。

还有两个写作流程类的 Skill，从 Skill 降级成了 Workflow。因为它们不是"工具"，而是"告诉用户怎么做事的方法论"。

## 第三部分：合并的三大原则

### 原则 1：按场景组织，不要按工具组织

合并之前，我是这样分类的。

```
skills/
├── epub 转笔记工具/
├── 笔记转 Word 工具/
├── PDF 翻译工具/
├── OCR 识别工具/
├── PDF 文件整理工具/
...
```

这是按"工具类型"分类。用户想转 EPUB，就找 EPUB 工具。想转 Word，就找 Word 工具。

但用户真正想的是什么呢？

用户想的是"我要转换内容"。

至于是从 EPUB 转，还是从 PDF 转，还是从图片转，那是具体场景，不是用户的根本意图。

所以合并后，我是这样组织的。

```
skills/
└── content-converter/     # 统一入口：内容转换
    ├── tools/
    │   ├── epub/          # EPUB 场景
    │   ├── md2docx/       # Word 场景
    │   ├── ocr/           # 图片识别场景
    │   ├── pdf-translator/# PDF 翻译场景
    │   ├── audio/         # 音视频转文字
    │   ├── ebook/         # PDF 文件整理
    │   └── video2gif/     # 视频转 GIF
```

一个入口，七个场景。用户说"我要转换内容"，AI 知道去 `content-converter` 里找具体场景。

SKILL.md 里的描述也变了。以前是每个 Skill 一份文档，现在是统一用表格展示。

| 需求 | 场景 | 工具 |
|------|------|------|
| EPUB 转笔记 | 场景 1 | tools/epub/ |
| 笔记转 Word | 场景 2 | tools/md2docx/ |
| 图片识别 | 场景 3 | tools/ocr/ |
| PDF 翻译 | 场景 4 | tools/pdf-translator/ |
| 音视频转文字 | 场景 5 | tools/audio/ |
| PDF 文件整理 | 场景 6 | tools/ebook/ |
| 视频转 GIF | 场景 7 | tools/video2gif/ |

一目了然。

### 原则 2：目录结构要保留原貌

合并的时候，我犯过一个错误。一开始想把所有工具文件摊平放在一起。

```
# 这样不对
tools/
├── epub2md.js
├── md2docx.py
├── ocr.py
```

看起来简洁，但问题来了。

每个工具原来的 README、配置文件、依赖说明，都找不着北了。以后想拆分或者修改，也不知道该改哪里。

后来我改成了子目录结构。

```
# 这样才对
tools/
├── epub/
│   ├── epub2md.js
│   └── package.json
├── md2docx/
│   ├── md2docx.py
│   └── requirements.txt
└── ocr/
    ├── ocr.py
    └── README.md
```

每个工具保持原来的完整结构，只是嵌套在统一的 tools/ 目录下。

这样做有三个好处。

第一，原项目的文档、配置都能对上号。我想改 md2docx，去 tools/md2docx/ 里找就行了。

第二，依赖关系清晰。每个子目录里有自己的 package.json 或 requirements.txt，但不会真的安装私有依赖，只是说明它需要哪些包。

第三，未来想拆分容易。如果以后发现某个工具确实应该独立，直接从子目录拎出来就行。

### 原则 3：环境必须统一

这是最关键的一条。

合并之前，每个 Skill 都有自己的依赖环境。

```
skills/
├── skill-a/node_modules/  # 重复的依赖
├── skill-b/venv/          # 重复的 Python 环境
├── skill-c/node_modules/  # 又一套重复依赖
...
```

AI 加载的时候，会把所有这些目录都读进上下文。node_modules 动辄几千个文件，AI 的脑子被这些无关信息塞满了。

合并之后，我规定了一条规则：**tools/ 子目录里禁止出现私有依赖**。

所有依赖都集中到项目根目录。

```
项目根目录/
├── .venv/              # 共享 Python 环境
├── node_modules/       # 共享 Node 环境
└── .agents/skills/
    └── content-converter/
        └── tools/
            ├── epub/       # 没有 node_modules
            ├── md2docx/    # 没有 venv
            └── ocr/        # 没有私有依赖
```

每个工具的 package.json 或 requirements.txt 只是声明依赖，实际安装在根目录。这样既保留了工具的独立性，又避免了上下文污染。

## 第四部分：Skill、Workflow 和 Doc 的边界

合并过程中，我还解决了一个分类问题。

有些"工具"其实不是工具，而是"告诉用户怎么做"的流程指南。

比如写作流程。我原来有两个 Skill，一个叫 AI 写作协作，一个叫模型评测写作。它们告诉用户怎么写文章：先揉开观点，再找素材，再搭骨架，再填充，最后优化。

这不是"工具"，是"方法论"。用户不能"调用"它来完成某个任务，只能参考它来学习做事的方法。

这类内容，我把它从 Skill 降级成了 Workflow，放在 `.agents/workflows/` 目录里。

现在我的分类是这样的。

| 类型 | 定位 | 例子 | 存放位置 |
|------|------|------|----------|
| Skill | 可调用能力 | content-converter | `.agents/skills/` |
| Workflow | 流程指南 | AI协作写作工作流 | `.agents/workflows/` |
| Doc | 参考规范 | agent-collaboration | `.agents/skills/agent-collaboration/` |

这样分类之后，目录清晰多了。11 个 Skill，2 个 Workflow，一目了然。

## 第五部分：给你的建议

![Skill 合并自检清单](assets/checklist_table.png)

### 什么时候该合并

如果你遇到以下情况，就该考虑合并了。

- **总是同时使用**：调用 A 时 90% 也会调用 B
- **概念重叠**：两个工具的描述有 50% 以上重叠
- **维护负担**：改一个规则要改多处
- **用户困惑**：用户经常问"该用哪个"
- **目录混乱**：skills/ 目录超过 20 个条目

### 什么时候该保持独立

但如果出现这些情况，保持独立更好。

- **独立价值**：单独使用也有意义
- **不同受众**：面向完全不同的用户群
- **组合灵活性**：用户可能需要 A 但不要 B
- **技术隔离**：依赖冲突严重无法共存

### 快速自检清单

你可以问自己三个问题。

第一个问题：你的 skills/ 目录有超过 20 个条目吗？

有的话，可能需要合并。

第二个问题：用户经常问"该用哪个工具"吗？

是的话，说明概念重叠，边界不清晰。

第三个问题：改一个规则要改多处吗？

是的话，维护负担太重，需要整合。

## 结语

回头看这次整理，最大的收获不是从 34 变成了 12 这个数字。

而是我终于想明白了一件事：AI 协作工具的管理，不是越多越好，而是越整越好。

Skill 就像工具箱里的工具。你可以有 34 把零散的工具，也可以有 12 套成套的工具。后者拿起来更顺手。

希望我的经验对你有帮助。
