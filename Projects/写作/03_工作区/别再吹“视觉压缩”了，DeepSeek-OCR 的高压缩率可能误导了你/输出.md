## 视觉压缩的理念与可行性

论文里面把这个概念叫做"光学压缩"，从理论上看是由于一个图片的 token 可以包含多个文本的 token，但实际上是否真的可以在单个图片的 token 内包含多个文本呢？

先给出答案：我觉得压缩文本 token 也能实现一样的效果，视觉压缩还存在不少的局限性，视觉作为输入方式是一个值得期待的方式，目前主流还会是文本的方式作为输入。

在后面的部分我们会具体展开。

## 文本分词流程：从字符到离散 ID 序列

与我们使用的 LLM 类似 VLM 在处理图片的时候也有对应的 tokenizer，不过与文本的 tokenizer 之间有区别，文本的 tokenizer 一般是基于子词的 BPE、WordPiece 等，图片的 tokenizer，比如 VIT 架构使用的是一种 patching 的方式。有个特点就是图片的 tokenizer 不需要训练，而文本的 tokenizer 是需要预先进行训练的，不过训练 tokenizer，需要的时间一般不会太长，如果你想要体验下训练一个文本的 tokenizer，可以尝试了解下 github 上的 nanochat 项目，我当时在大概是在 170 GB 的文本数据上训练，复现 nanochat 的分词器就花了几分钟，速度非常快。

我们先看看文本的 token 产生的流程。

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao3ick7wmEuiaCAa8T8KB5TOibka2ibK9cHOiajpKy7icq1mBWyoibwiboI1r23SxxC84ticfWvuP2TZswiaPUCNg/640?wx_fmt=png&from=appmsg&tp=webp&wxfrom=5&wx_lazy=1#imgIndex=1)

文本 token 的产生过程，我们一般称为分词，或者叫 Tokenization。它依赖一个叫分词器 Tokenizer 的工具来完成。

这个工具的核心是一个提前训练好的词汇表 Vocabulary。

词汇表是通过在海量的文本数据上，使用特定算法，比如 BPE 或 WordPiece，统计训练出来的。算法会不断地把最高频一起出现的字符或字符组进行合并，形成一个新单元。

比如，算法最开始只认识单个字母，它发现"S"和"e"总是一起出现，就合并成"Se"。接着又发现"Se"和"ek"总是一起出现，就合并成"Seek"。这个过程不断重复，直到词汇表达到一个固定的大小，比如五万或十万, 以 nanochat 为例，他默认的词汇表大小是 6 万五千多。

这个词汇表最终会包含各种单元，有单个的字符，有常见的词根，也有完整的单词。

当我们输入一段新文本，比如"DeepSeek-OCR"，分词器就会对照它的词汇表，把这段文本"切"成它认识的最长单元组合。

比如，它可能会把"DeepSeek-OCR"切分成"DeepSeek"、"-"和"OCR"这三个单元。在词汇表里，"DeepSeek"可能对应 ID 15000，"-"对应 ID 28，"OCR"对应 ID 987。

所以，"DeepSeek-OCR"这段文本，最后就变成了一串整数 ID 序列[15000, 28, 987]。

## 图片分词流程：从 patch token 到视觉 token

我们下面看看图片的 token 产生的流程。

首先，输入一张高分辨率图片。

这张图片会被送入 DeepEncoder ，这个组件被视为图像的 Tokenizer 。

第一步，图像经过一个基于 SAM 的组件，这个组件主要使用窗口注意力。它会把图像分割成大量的初始 patch token，需要注意的是这里的 patch token 并不是我们说的视觉 token 。例如，一张 1024 x 1024 的图片在这里会被分成 4096 个 patch token，即 。

第二步，这几千个 token 并不会直接交给语言模型，而是先经过一个 16 倍的卷积压缩模块。这个模块会进行 16 倍的下采样，把 token 数量大幅减少，比如从 4096 个压缩到 256 个。

第三步，这些被压缩过的、数量大大减少的 token，会再进入一个基于 CLIP 的组件。这个组件使用全局注意力，充当嵌入层，最终生成的就是视觉 token 。

> 关于 clip 的部分我觉得可以去看看我之前的一篇关于 clip 模型微调的文章，里面有很详细的介绍 clip 的背景，以及如何实现跨模态内容对齐。 [多模态检索模型微调：以CLIP为例的工程实践](https://mp.weixin.qq.com/s?__biz=MzkwNjcxNTc2Ng==&mid=2247488832&idx=1&sn=1bac0d8cc5ba3926f2479d669af1c51f&scene=21#wechat_redirect)

这些最终产出的视觉 token，最后才会被送入解码器，也就是语言模型部分。

## 无需训练的图片分块器

我们回顾下 VIT 的论文。

ViT 处理图像的第一步，是将图片分割成固定大小的图块，也就是我们所说的 patches。例如，一个  的图像被切成  像素的图块，得到  个图块。

这个切分动作本身是一个固定的、确定性的操作，就像在 NLP 中按照空格分割单词一样，它不需要训练。

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao38z7Gd8iacOZRG1qZ8Ckm6BUb4otnMMb5lXFL57LBZwa2ibqTPDFv0B44LgskNKysc3iaTw8D90wo90w/640?wx_fmt=png&from=appmsg&tp=webp&wxfrom=5&wx_lazy=1#imgIndex=2)

在我回顾 VIT 的论文之时，我发现这些图块（tokens）随后需要被转换成 Transformer 能处理的向量。

ViT 使用一个"可训练的线性投射"（trainable linear projection）层来实现这一点。每个图块被展平，然后通过这个线性层，从原始的像素维度（例如 ）映射到模型的工作维度 （例如 ViT-Base 的 768 维）。

这个线性投射层，是需要和模型主体一起训练的。它在功能上等同于 NLP 中的词嵌入层（embedding layer），负责学习如何将原始的 token（在这里是图块）表示为有意义的向量。

## DS-OCR 的视觉 token 压缩模块

如果你悟性高的话，我觉得应该能发现，deepseek-ocr 的环节有一个压缩 token 的环节，他本来已经使用了视觉的图片分割方式，进一步又使用了一个模块进行 token 压缩。

具体来说是按照不同的分别率输入内容，统一进行 16 X 倍率压缩。

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao38z7Gd8iacOZRG1qZ8Ckm6BUzicFZYTMUPLA3LoHD4EnHGjicl4YDEuqs5Jc5HRtLnACXjVuqIuR5TQA/640?wx_fmt=png&from=appmsg&tp=webp&wxfrom=5&wx_lazy=1#imgIndex=4)

> 图源：[7]

不同的分辨率分别是：

-   Tiny 模式：输入  图像，最终输出 64 个 vision tokens 。
    
-   Small 模式：输入  图像，最终输出 100 个 vision tokens 。
    
-   Base 模式：输入  图像，输出 256 个 vision tokens 。
    
-   Large 模式：输入  图像，输出 400 个 vision tokens 。
    

而我们回顾下 deepseek-ocr 论文开始就出现的那张图的内容。

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao38z7Gd8iacOZRG1qZ8Ckm6BUiaudzIktQOX2LoSq8rMDVR8l3pxzrC4SicpMCDb9Ww5f5LbTXgibO2ocw/640?wx_fmt=png&from=appmsg&tp=webp&wxfrom=5&wx_lazy=1#imgIndex=5)

> 图源：[7]

当压缩率在 10 倍以内时，也就是 1 个 vision token 对应 10 个 text token，模型的 OCR 解码精度可以达到 97% 左右。

当压缩率达到 20 倍时，精度仍然能保持在 60% 左右。

假设他没有中间的图片 token 压缩环节，也就是 **去掉 16 X 的压缩模块会怎么样？**

这个问题容易让人迷糊，我们先从文本-视觉压缩率的评估指标开始，它的计算方式是： 。

比如论文表格里，1200-1300 个文本词元，用 Tiny 模式的 64 个视觉词元去解码，压缩率大概是  倍，也就是我们说的 20 倍左右。

第二个是内部模块压缩，也就是压缩模块 16 倍压缩。这是一个模型组件。

它是一个 16 x 的卷积压缩模块，是 DeepEncoder 架构的一部分。它的作用是把内部的 patch 数量减少 16 倍，然后再交给后续的全局注意力模块处理。

我们拿 Base 模式举个例子。输入  图像。

1.  最开始的 SAM 编码器把它切成  的图块，这时内部有  个内部 patch tokens。
    
2.  这 4096 个 tokens 经过 16 倍压缩模块，变成了  个 vision tokens。
    
3.  这 256 个 vision tokens，才是最终喂给解码器、并用来计算 20 倍压缩率的那个分母。
    

现在我们回答问题：如果去掉这个 16 倍的压缩模块会怎么样？

那模型在 Base 模式下输出的 Final Vision Tokens 数量就不是 256，而是 4096。

假设原来 1000 个文本词元可以用 Small 模式的 100 个视觉词元解码，压缩率是 10 倍。

如果去掉 16 倍压缩，模型输出的视觉词元会变成  个。

那文本-视觉压缩率就变成了  倍。

从我们刚才的计算结果可以看出如果去掉了视觉 token 的压缩环节之后，对比文本的方式作为输入并不能压缩内容。

这个问题是我在看到孔老师的一篇 [简评DeepSeek-OCR](https://mp.weixin.qq.com/s?__biz=Mzk0MDU2OTk1Ng==&mid=2247485390&idx=1&sn=71f8e969b2d31e867d834e50be46deff&scene=21#wechat_redirect) 文章提到一个文本的内容压缩 baseline 得到启发想到的。

> 孔老师的文章提到：
> 
> 把连续的 16 个 token 的文本作为一个 chunk，再向前 padding 之前的 8 个 token 做 overlap，使用一个 LSTM 结构进行处理，聚合输出为一个 embedding 表示。这个方式也可以把文本 token 压缩为一个更高密度的表示，压缩率 16：1。

## 视觉压缩的是哪里进行的？

有的小伙伴可能看过 [Karpathy 评 DeepSeek-OCR：分词器必须消失！马斯克：光子才是 AI 的终极语言](https://mp.weixin.qq.com/s?__biz=MzA4NzgzMjA4MQ==&mid=2453478604&idx=1&sn=df6f46525167aa26391712e5d49ed87d&scene=21#wechat_redirect) 这篇文章里面说的一个测试：

**纯文字输入** ：796 tokens，花费

**截图输入（低分辨率）** ：192 tokens，花费

为什么刚才说了文本转图片没办法压缩 token ，这里为什么实际测试出来是明显减少了 token？

原因是大多数 VLM 都会有一个压缩环节，我们以 qwen 作为例子。

初代 Qwen-VL 使用了 **Position-aware Vision-Language Adapter** ，适配器包含一个 **单层交叉注意力 (cross-attention) 模块** 。它的作用是将视觉编码器 (ViT) 输出的可变长度图像特征序列压缩到一个固定的长度 (256 个 token)，然后才送入 LLM 。

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao38z7Gd8iacOZRG1qZ8Ckm6BUMkRyzpTnlVRicKOc7tTK23EQFicw5gXqhM9Lhf6u0SlmRn9lNPsLNOOg/640?wx_fmt=png&from=appmsg#imgIndex=6)

> 图源：[15]

Qwen 2-VL 和 Qwen 2.5-VL 采用了 **不同的压缩方法** 。它们使用了一个 **MLP-based Vision-Language Merger** (或称为简单的 MLP 层) 。

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao38z7Gd8iacOZRG1qZ8Ckm6BUoIjHYqQXxpxg3NmrriaNTfOA2gdTaqju8SSgGAryiaa7HYcQr9DbiaMQA/640?wx_fmt=png&from=appmsg#imgIndex=7)

> 图源：[14]

这个合并器/层位于 ViT 之后、LLM 之前，它将空间上相邻的 2 x 2 (共 4 个) patch 特征分组。

这些分组后的特征被拼接，然后通过一个两层 MLP 进行处理，投影到 LLM 所需的维度。

这种方法有效地将 ViT 输出的 token 数量减少了 4 倍 (因为每 4 个合并为 1 个)，从而降低了计算成本。例如，对于 224 x 224 的输入图像，ViT 产生 256 个 patch token，经过这个 2 x 2 压缩后变成 64 个，再加上特殊 token 变为 66 个输入 LLM 。

> 数据来源：[14]

所以我们如果要压缩 LLM 的 token，实际上通过添加一个压缩模块，一样是可以实现类似 token 压缩效果的。 [Karpathy 评 DeepSeek-OCR：分词器必须消失！马斯克：光子才是 AI 的终极语言](https://mp.weixin.qq.com/s?__biz=MzA4NzgzMjA4MQ==&mid=2453478604&idx=1&sn=df6f46525167aa26391712e5d49ed87d&scene=21#wechat_redirect) 里面也有提到 Mahaoo 作为视觉领域的人，但他表示 **将一万字的文本压缩成约 100 个 token 也非常容易** 。

**Glyph 论文也展示了另一种视觉压缩思路** 。它不是在模型内部加压缩模块，而是在预处理阶段就把长文本渲染成紧凑的图片 。然后 VLM 直接处理这些图片。他们通过精心设计的渲染参数（字体、布局、分辨率等），可以达到 **3-4 倍的 Token 压缩率** ，同时性能与同等规模的文本 LLM 相当 。

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao3ick7wmEuiaCAa8T8KB5TOibkaicphicteCXXlC8D3JgTNBKSnLiaTUkkGVDDv7h1cbNqPBtf1sA8m2kKkg/640?wx_fmt=png&from=appmsg#imgIndex=8)

**但这篇 Glyph 论文也指出了局限性** ： **模型性能对渲染参数非常敏感** ，需要通过复杂的搜索（LLM 驱动的遗传算法）来找到最佳配置，这本身就带来了额外的复杂度和计算开销。

**对于一些精细的文本识别（比如 UUID 或罕见字符序列）仍然存在挑战** 。

**跨任务泛化能力可能不如纯文本模型** ，表明这种视觉表示可能损失了一部分通用性，之前杨植麟也说到 多模态不损伤"脑子"已经很好了。 [和杨植麟时隔一年的独家对话："站在无限的开端"](https://mp.weixin.qq.com/s?__biz=MzE5ODg1MTY4Mw==&mid=2247484596&idx=1&sn=b18d9ee6b10fb2e10637dd6626fc811c&scene=21#wechat_redirect)

这些局限性恰恰印证了我们的观点，无论是 DS-OCR 的内部压缩，还是 Glyph 的外部渲染压缩，这种视觉压缩都不是图像模态与生俱来的、可以轻松获得的优势，仍然存在诸多问题。

从文本、图片对内容的表征能力思考，DeepSeek-OCR 论文中它明确提出了一个观点：利用视觉模态作为文本信息的一种高效压缩介质，我觉得倒是有一个前提，就是他刚开始分割出来的 patch token 里面的 token 是否就能压缩到多个文本 token 呢，至少要达到两个 token 以上吧。

目前的压缩只是因为他在对 token 进行 embedding 前加入了压缩的环节，如果我们给 LLM 加上压缩的环节，一样是能达到压缩的效果的，所以我觉得一个 patch token 不能包含多个字的时候，我觉得他的观点并不能成立，即使是 Glyph 能够基于非模型结构输入的中间压缩实现，也是需要各种复杂的条件限制输入内容。

## 跨模态对齐的鸿沟与挑战

有的小伙伴问图像的本质是分辨率，语言的本质是字符串、是词，且词本身有歧义，两个存在最根本的对齐问题。会不会导致文转图转文本身难度极大?

其实这主要是跨模态对齐的问题，可以参考类似 SD 扩散模型为什么能使用文本引导图片生成的逻辑，他内部使用了一个 clip 的模型进行内容对齐。

跨模态的内容对齐存在信息差是必然的，但是可以在一定程度上削减，比如 jina clip v 2 模型之前我在很早之前开发 SmartImager 的时候就开始使用了，这个模型就是存在跨模态导致的向量相似度极低，举例来说。

文本"海象"向量化后和"海象"图片向量化进行 cos 计算，结果他们之间的相似度只有 0.2 左右，我们从直觉来看相似度太低了，不过实际使用的过程中可以发现文本各个图片之间还是能够看出来他们之间的排序关系的，也就是相似度搜索是可以实现的，所以我当时的时候就将 jina clip v 2 作为 MVP 的方式应用到了我的一个图片搜索项目里面，那个项目的名字叫做 SmartImager，地址在：https://github.com/li-xiu-qi/Smartlmager

> GitHub - li-xiu-qi/Smartlmager: 一个基于多模态向量模型及视觉多模态模型构建的图片搜索引擎&管理系统，实现精准的以文搜文，文搜图、以图搜图多种智能检索方式。
> 
> 我觉得还是挺好用的，你可以理解成一个图片搜索管理系统，或者本地的图片搜索引擎，如果觉得对你有用的话，可以给我点个 star。

但是文本和文本之间，他们一点都不像, 相似度也能达到 0.5 左右的值，也就导致了使用 jina clip v 2 模型进行多模态混合检索的路子不能暴力的混合图片和文本的方式，否则仅仅基于文本查询向量进行检索的时候文本内容一定是每次高于查询向量与图片之间的相似度的，不管内容是否相关。

虽然我们可以独立开文本和图片实现，但是我们这里主要想要讨论的是跨模态之间的向量语义差。

我从论文里看到他们重点讨论了这个问题。他们称之为"模态鸿沟" (modality gap) 。我倒是觉得，不同模态之间的检索可以分类为非对称检索类型，不过可能不够细致，用跨模态形容确实更贴切。

这个鸿沟指的是，在像 CLIP 这样的双编码器模型中，文本和图像的嵌入向量倾向于聚集在嵌入空间的不同区域。

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao38z7Gd8iacOZRG1qZ8Ckm6BUMeAPYVichSJaG3j2SSvfXZ7oTqw6KwZdzQCSCkVD6Z0Qxql6NWQUJFA/640?wx_fmt=png&from=appmsg#imgIndex=9)

> 图源：[10]

jina-clip-v 2 明确包含两个独立的模块 ：

1.  一个文本编码器，用的是 Jina XLM-ROBERTa 。
    
2.  一个视觉编码器，用的是 EVA 02-L 14 。
    

这导致一个现象，一个文本和它语义匹配的图像之间的相似度，可能还不如它和一个语义不相关的其他文本之间的相似度高。由于两个编码器各自进行工作导致他们之间生成的内容存在语义差，有点像是两个不懂对方语言的人在对话，说出来的内容总是有点差异，即使他们都试图表达一个意思，英国人说中文或者中国人说英文我觉得在他们不熟悉的前提下进行表达就是一个很好的例子。

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao38z7Gd8iacOZRG1qZ8Ckm6BUrHMSsic1ulR4LkTCibMywWCtcEozYRzeiblCfXicC4N3q75DLON3Hu84og/640?wx_fmt=png&from=appmsg#imgIndex=10)

> 图源：[9]

论文提到，v 4 模型因为采用了统一的架构，即文本和图像通过共享的语言模型解码器处理，从而显著减少了这种模态鸿沟，虽然论文提到显著减少，但是我没有实际测试，我只是更新了这个项目使用的默认模型是 jina embedding v 4，但是没有看改进能达到什么程度。

感兴趣的小伙伴可以去看原论文，或者进行实际测试文本和图片之间的模态鸿沟情况，本文重点不是 jina embedding v 4 就不细聊了。

## 纯图片输入对比文本输入如何？

Karpathy 之前公开提到他对分词器的厌恶，觉得 LLM 的 tokenizer 是丑陋了，破坏了模型的端到端过程，且分词器带来的不一致性和安全隐患也是大问题，比如一个词在不同的上下文里面可能会被分成不同的 token。还有就是，分词器对数字、代码或罕见词的处理非常低效，并且引入了 Unicode 和字节编码的很多历史包袱，导致了续字节攻击这类风险。

基于我们上面提到的观点：视觉 token 如果不加入一个压缩的模块，实际上效率是不如文本的。且文本输入的 token 最输入值一定是比视觉最小输入的 token 要小的，因为图片先有一个分割环节后面还有一个压缩的环节，但最小单位仍然是一个图片，从我们对 qwen 系列的模型分割图片的设计来看，输入的 token 最小值也是要几十 token 起。另外一个需要思考的，即超长上下文不可能统一放到一个图片里，如果分成了多张图之后，各个图之间的联系模型是否能理解。

从文本的角度理解，token 输入的时候会添加位置编码，所以我们思考图片的 token 他们需要添加一个图片之间的位置编码。类似 qwen vl 初代就采用了类似的处理。

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao38z7Gd8iacOZRG1qZ8Ckm6BUibs0IuuvNYjmtAicEWSDdUrhwOT3HeqlJqxWjflWmY8NiahmRiba3xabiag/640?wx_fmt=png&from=appmsg#imgIndex=11)

> 图源：[13]

论文中提到了"位置感知的视觉-语言适配器" (Position-aware Vision-Language Adapter) 。为了缓解压缩过程中可能丢失的位置细节，它将二维绝对位置编码 (2 D absolute positional encodings) 融入了交叉注意力机制的查询-键对中。

在 Qwen-VL 系列论文中我看到，它们处理多张图片输入的方式主要是将图片 **穿插在文本序列中，按顺序输入** 给模型。

模型在训练时就支持任意的图文穿插数据输入。这意味着你可以像在对话中那样，先说一段话，插入一张图片，再说一段话，再插入另一张图片。

为了区分文本和图像特征，图像特征序列的开始和结束会用特殊的标记（比如 `<img>` 和 `</img>` ，或者 `<vision_start>` 和 `<vision_end>` ）包围起来。

当有多张图片时，为了区分它们，模型在处理时（特别是在对话格式中）会给不同的图片加上标识，例如 "Picture 1: ", "Picture 2: " 这样的前缀。

比如下面的模板：

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao38z7Gd8iacOZRG1qZ8Ckm6BU18CB1TM5Aic7dfvK2uIvsnj5N8JynDSW5DBMRmMyvU0MoniaFiaxOMf5A/640?wx_fmt=png&from=appmsg#imgIndex=12)

> 图源：[13]

如果统一，都使用图片，而不使用文本，似乎我们就可以全部使用图片，而不需要考虑文本和图片混合的一个 chat 模板了，对于大一统输入格式倒是有帮助。

## Gemini 输入机制猜想

之前有人在文章中提到 gemini 就是使用图片的方式对文本内容进行压缩，我当时也有一点怀疑，但是经过上面的一个计算介绍，大家应该能知道图片的压缩文本实际上并不能比直接压缩文本要优秀，而我在一次无意中的测试居然发现了 gemini 的一个输入内容机制。

gemini 居然能接受图文格式的 markdown 内容。仔细观察下面三个图你就能发现。

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao38z7Gd8iacOZRG1qZ8Ckm6BUFjhsl2iaYdiawvuUq8Diaw1EZGBhRv7GoB49sDwPPYLyuic4Z1NzxSTm9g/640?wx_fmt=png&from=appmsg#imgIndex=13)

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao38z7Gd8iacOZRG1qZ8Ckm6BUjBA3LAxVuicicFu57vDCjVqPbaiaTEFDRkp0nJO5mhSfEulYfUGulI6mA/640?wx_fmt=png&from=appmsg#imgIndex=14)

![图片](https://mmbiz.qpic.cn/mmbiz_png/OpQ5uW8ao38z7Gd8iacOZRG1qZ8Ckm6BUcKSiaGoVReyWafVeoLT08MZOxicets9X0FRakBDST0h3kial4bsBrcayw/640?wx_fmt=png&from=appmsg#imgIndex=15)

第一个图是一个标准的 markdown 图片语法内容, 第二个图的 markdown 图片语法是错误的，第三个图的 markdown 语法是正确的，不过是另外一种表达方式。

我们可以发现的是，只要我们的 markdown 语法正确，那么 gemini 就会回答图片的内容，即使我们的图片是无效的。

从这里我们可以猜出来，gemini 绝对不会是使用了图片对文本进行压缩，因为如果直接使用图片的方式压缩了内容就不会误解了我们这次输入的内容，而是使用类似正则的方式提取图片，并使用类似 Qwen VL 的聊天模板设置的方式填入图片和文本的内容，并生成回答，真相大白了！

那么我们还可以进一步发现 markdown 格式的大模型的输入会导致又有一个新的注入攻击的出现，就像以前防 js xss 一样。

## 总结

我认为视觉压缩这个理念，或者说用图片 token 作为文本的高效压缩介质，目前是有不少局限性的。

DeepSeek-OCR 论文里展示的高压缩率，比如 1 个视觉 token 对应 10 个文本 token，并不是因为图像模态本身具有天然的压缩能力。

这种效果的实现，关键在于模型架构中加入了一个显式的压缩模块。比如 DS-OCR 里的 16 倍卷积压缩，或者 Qwen-VL 里的 MLP 合并层。它们是在图像被初步分割成大量 patch token 之后，再强行把 token 数量减少。

如果我们去掉这个压缩环节，会发现视觉 token 的数量远超它试图表示的文本 token，压缩率反而会低于 1。

同样的压缩逻辑完全可以应用在纯文本 token 上，比如使用 LSTM 结构同样能实现高倍率的文本压缩。

所以，目前观察到的压缩效果来自压缩模块，而不是来自视觉模态。把文本转成图片再输入，并不能带来优于直接压缩文本的效率。

所以，目前观察到的压缩效果主要来自 **显式的压缩模块** （如 DS-OCR、Qwen-VL）或 **预处理渲染策略** （如 Glyph ），而不是来自视觉模态本身。把文本转成图片再输入，虽然在 Glyph 的实践中显示出了一定的压缩效果和效率提升，但面临的渲染敏感性、OCR 挑战等局限性也表明，它并非一种绝对优秀的方案。

最后，视觉作为一种输入方式还是很值得期待与探索的，就看后面如何继续优化了。
