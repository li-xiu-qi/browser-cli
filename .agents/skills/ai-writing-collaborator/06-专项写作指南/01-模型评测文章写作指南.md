# 模型评测文章写作指南

**来源**: 从 30+ 篇评测文章（GLM/Kimi/豆包/Step 系列）提炼的方法论  
**适用**: AI 模型评测类文章（编程模型、多模态模型等）

**配套资源**: [[../07-模型评测资源库/_INDEX|模型评测资源库]] - 提供标准化评测case集合

---

## 快速开始

写模型评测文章时，可直接使用 [[../07-模型评测资源库/使用指南|评测资源库]] 中的标准化case：

1. 选择要测试的能力维度（如：视觉理解、代码生成）
2. 从资源库中选取对应评测集（如：[[../07-模型评测资源库/视觉理解评测集/评测集说明|视觉理解评测集]]）
3. 使用评测集中的Prompt进行测试
4. 按照本指南的结构撰写文章

## 使用评测资源库的优势

- **标准化**: 使用经过验证的评测case，保证测试质量
- **可比性**: 与已有范文使用相同case，便于对比
- **效率**: 无需从零设计测试，直接使用现成case
- **全面**: 覆盖9大维度、25+测试case

## 一、评测风格类型谱系

基于 30+ 篇文章分析，推荐组合：

| 风格类型 | 特点 | 适用场景 |
|----------|------|----------|
| **专业横评型** | 多模型对比、数据说话 | 基准评测 |
| **关卡挑战型** | 递进式测试、难度升级 | 能力边界探索 |
| **视觉编程型** | 视频→代码全流程 | 多模态评测 |
| **痛点替代型** | 直击替代价值、成本对比 | 竞品对比 |

**推荐组合**: 专业横评型 + 关卡挑战型 + 痛点替代型

## 三、核心卖点前置（6点式）

开篇或目录后快速呈现：

```markdown
- **能力定位**: 与标杆模型的对比
- **技术参数**: 上下文、速度、架构
- **多模态能力**: 视频理解、视觉编辑
- **Agent能力**: 集群协同、效率提升
- **成本数据**: 具体价格（精确到毛）
- **国际认可**: 海外反响、平台接入
```

## 六、写作风格规范

### 人称使用
- 使用第一人称"我"而非"我们"
- 示例："我测试了4个模型"

### 语气态度
- 客观中立，不贬低任何模型
- 不说"A不如B"，而是"A在XX场景下表现..."
- 突出被测模型的独特价值

### 数据呈现
- 表格：模型作为列（横向对比）
- 截图：每个模型独立展示
- 关键数据：加粗突出

### 禁用元素
- 不要使用表情符号（⚠️ 等）
- 用"正确/错误/不知道"文字替代
- 避免标签式引出语："关键发现："

### 基于复盘经验的特别禁止（重要！）
- ❌ **禁止开篇暴露利益相关**：不要在开头100字内出现产品链接、推广信息
  - 错误：《...Step-3.5-Flash目前可在OpenRouter免费使用...》（第38行就暴露）
  - 正确：开篇专注讲问题/发现，产品信息移到文末附录

- ❌ **禁止时效性知识测试灌水**：不要堆砌截图没有分析
  - 错误：4个模型各一张截图，每段就一句"正确"/"不知道"
  - 正确：要么深入分析，要么直接删除这部分

- ❌ **禁止结论自相矛盾**：前文承认的局限，后文不能否认
  - 错误：前文说"知识时效性不如GLM"，后文说"完全平替各家"
  - 正确：明确适用场景，承认局限性

- ❌ **禁止单向吹捧**：评测要客观，必须有反面声音
  - 必须包含：什么时候不要用？实际使用中的坑？测试局限性？

## 七、图片与视频规范

### 图片命名
```
{模型}_{任务}_{状态}.png
示例：step_dashboard_correct.png
```

### 表格中不嵌套图片
❌ 错误：表格内放 `<img>`
✅ 正确：散开格式

```markdown
DeepSeek-V3.2：

![截图](deepseek.png)

回答正确

Kimi-K2.5：

![截图](kimi.png)

回答正确
```

### 视频转 GIF
微信公众号不支持视频，需转 GIF：

```bash
ffmpeg -i input.mp4 -vf "fps=4,scale=720:-1:flags=lanczos,split[s0][s1];[s0]palettegen=max_colors=128[p];[s1][p]paletteuse" -loop 0 output.gif
```

**限制**:
- 帧数 ≤ 300 帧
- 宽度 640-800px
- 颜色 64-128 色

## 八、结论写法

- ❌ 不要：汇总表格 + 文字总结（重复）
- ✅ 要：保留其一，推荐带截图的表格
- 金句收尾，强调核心观点

**示例金句**:
```markdown
"快且好，才是真的好"
"在 AI 编程时代，速度本身就是竞争力"
```

## 九、评测文章自查清单（发布前必检）

### 结构检查
- [ ] 开篇100字内没有产品链接/推广信息
- [ ] 测试部分有实质分析，不是截图堆砌
- [ ] 结论包含反面声音（局限性、不适用场景）
- [ ] 前后逻辑一致，没有自相矛盾

### 内容检查
- [ ] 有时效性知识测试的，要么深入分析要么删除
- [ ] 案例前后一致，数据准确
- [ ] 结论明确，不说"视情况而定"的模糊话

### 传播检查
- [ ] 标题没有术语堆砌
- [ ] 有数字或对比
- [ ] 有悬念或反直觉

## 十、失败案例复盘（Step-3.5-Flash评测）

### 数据表现（最差案例）
- 阅读量：202（极低）
- 点赞率：3%（很低）
- 转发率：3.5%（很低）
- 推荐：~0（算法不认可）

### 致命错误
1. **开篇即广告**（第38行产品链接）→ 信任崩塌
2. **时效性测试灌水**（18%篇幅无实质内容）→ 读者失望
3. **结论自相矛盾** → 可信度归零
4. **缺少反面声音** → 像公关稿

### 改进方案
1. 删除时效性测试整段
2. 重写结论（承认局限性）
3. 开篇移除产品链接（移到文末）
4. 增加"什么时候不要用Step"章节
