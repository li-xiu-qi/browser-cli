# 技术博客风格模板 V1.0

**定位**: 技术分享、教程、经验总结  
**语气**: 专业但不晦涩，像技术同事间的交流  
**目标读者**: 有一定基础的技术从业者

---

## 角色与读者

- **角色**: 有实践经验的技术从业者
- **读者**: 同行、想学习该技术的开发者
- **语气**: 专业、清晰、有帮助

---

## 风格要点

### 1. 问题导向
- 从具体问题出发，不是从概念出发
- 说明"我为什么要写这篇文章"
- 让读者快速判断这篇文章是否适合他

### 2. 实用优先
- 提供可执行的代码/步骤
- 标注版本信息和环境要求
- 说明可能遇到的坑

### 3. 深度思考
- 不止于"怎么做"，还要解释"为什么"
- 分享决策过程和权衡
- 给出个人见解（而非照搬文档）

---

## 文章结构

### 开头（100-200字）
1. 场景引入：我在什么情况下遇到这个问题
2. 问题描述：具体是什么问题
3. 方案预告：我是怎么解决的/本文要讲什么

### 主体
1. **背景说明**（可选）- 必要的上下文
2. **方案探索** - 尝试过什么，为什么选这个方案
3. **具体实现** - 代码、配置、步骤
4. **踩坑记录** - 遇到的问题和解决方法
5. **效果验证** - 如何验证方案有效

### 结尾（50-100字）
1. 方案总结
2. 适用场景说明
3. 下一步/相关阅读

---

## 用词偏好

### 推荐
- "我们"、"你" - 直接与读者对话
- 具体技术术语 - 准确表达
- 具体数字 - "3 个步骤"而不是"几个步骤"

### 避免
- "首先...其次...最后..." - 过于正式
- "综上所述"、"总之" - 套话
- 模糊形容词 - "非常"、"很"、"特别"

---

## 句式习惯

- 短句为主，技术解释可用长句
- 一个段落一个观点
- 代码和说明交替，不要大段代码后大段说明
- 用列表呈现步骤

---

## 禁止清单

- ❌ 禁止"赋能"、"抓手"、"闭环"等商业黑话
- ❌ 禁止"简单"、"容易"、"只要"等主观判断
- ❌ 禁止大段无注释的代码
- ❌ 禁止不说明版本的环境配置

## 示例
```python
# Python 3.9+
import asyncio

async def fetch_data(url):
    # 使用 aiohttp 进行异步请求
    async with aiohttp.ClientSession() as session:
        async with session.get(url) as response:
            return await response.json()
```
