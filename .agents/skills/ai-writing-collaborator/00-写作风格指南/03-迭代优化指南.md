# 迭代优化指南

**目标**: 通过持续迭代，让风格 Skill 越来越精准  
**核心原则**: 每一次修改都是训练数据

---

## 迭代的价值

### 为什么必须迭代

初版 Skill 就像厨子第一次尝你做的菜：
-  能分辨出大致口味偏好
-  细节还差得远

**迭代次数与效果**：

| 阶段 | 修改量 | Skill 状态 |
|------|--------|-----------|
| 第1次 | 50%+ | 只有大方向 |
| 第3次 | 30% | 核心风格成型 |
| 第5次 | 10% | 基本稳定 |
| 第10次 | 5% | AI 比你更一致 |

### 复利效应

```
第1轮：发现 5 个规则 → Skill 准确度 30%
第2轮：发现 3 个规则 → Skill 准确度 50%
第3轮：发现 2 个规则 → Skill 准确度 70%
...
第10轮：微调 1 个细节 → Skill 准确度 95%
```

## 迭代流程

```
实战写作 → 手动修改 → 分析差异 → 更新 Skill → 重复
```

### 步骤1：实战写作

**频率**: 每次用 AI 协作写作时  
**前提**: 加载最新版风格 Skill

**提示词模板**：
```markdown
请根据以下【写作风格 Skill】完成写作任务。

[粘贴 Skill 全文]

任务：[具体写作任务]
```

### 步骤2：手动修改（最关键）

**原则**: 必须亲自动手改，不能只在聊天里指挥 AI

**修改范围**：
- 个别用词
- 句式调整
- 段落重组
- 结构优化

**修改时注意记录**：
- 这里为什么要改？
- 原句哪里不符合我的风格？
- 我的习惯表达是什么？

### 步骤3：分析差异

使用对比提示词：

```markdown
请对比以下两段文字，分析我做了哪些修改：

【AI 原文】
[粘贴]

【我的修改版】
[粘贴]

请输出：
1. 逐条列出具体修改
2. 总结修改规律（3-5条）
3. 建议添加到 Skill 的新规则
```

**示例输出**：

| 修改项 | 规律 | Skill 更新建议 |
|--------|------|----------------|
| "然而" → 删除 | 少用转折词，直接陈述 | 禁止清单：少用"然而"、"但是" |
| 长句拆短 | 短句为主，信息密度高 | 句式习惯：单句不超过 25 字 |
| "我们可以看到" → "你看" | 第二人称直接对话 | 用词偏好：用"你看"代替"我们可以看到" |

### 步骤4：更新 Skill

**更新原则**：
1. **具体优于抽象** - 写"禁止'赋能'"而不是"避免商业词汇"
2. **正反示例** - 每条规则配一个修改示例
3. **归纳规律** - 禁止清单太长时，让 AI 帮你总结规律

**更新位置**：

```markdown
## 禁止清单

-  禁止"赋能"、"闭环"、"抓手"  ← 已有
-  禁止"首先...其次...最后..."  ← 已有
-  少用转折词（然而、但是），直接陈述  ← 新增

## 句式习惯

- 短句为主，长短句交替  ← 已有
- 单句不超过 25 字  ← 新增
- 一个段落一个观点  ← 已有
```

**版本号更新**：
```
V1.1 → V1.2 → V1.3...
```

## 迭代技巧

### 技巧1：Git Diff 对比

如果你用 Git 管理：

```bash
# 修改前保存为 ai-version.md
git diff ai-version.md my-version.md
```

一目了然看到所有改动。

### 技巧2：批量积累再更新

不用每写一句就更新 Skill：

| 策略 | 适用场景 |
|------|----------|
| 即时更新 | 发现了很明确的规则 |
| 批量更新 | 写完一篇文章后统一分析 |
| 定期更新 | 每周/每月统一回顾 |

### 技巧3：规则归纳

当禁止清单太长时，让 AI 帮你找规律：

```markdown
我的禁止清单已经有 30 条了，请帮我总结归纳成 5-8 条核心规律。

当前禁止清单：
[粘贴]
```

**示例归纳**：

| 原清单（30条） | 归纳后（5条） |
|----------------|---------------|
| 禁止"赋能"、禁止"闭环"、禁止"抓手"... | 禁止商业黑话，用日常词汇替代 |
| 禁止"然而"、禁止"但是"... | 少用转折词，直接陈述观点 |
| 禁止"首先"、禁止"其次"... | 避免序列词，用自然过渡 |

### 技巧4：正反示例

每条规则配一个示例，AI 学得更快：

```markdown
## 用词偏好

### 说人话
-  "用了十几年" 
-  "深耕多年"

### 直接对话
-  "你看"
-  "我们可以看到"
```

## 常见问题

### Q: 迭代多少次才够？

没有固定次数，看收敛程度：

| 信号 | 说明 |
|------|------|
| 主要改个别词 | 可以放慢迭代频率 |
| 还要改结构 | 继续迭代 |
| AI 写的比你还像你 | 成功了 |

宝玉老师的 Skill 迭代了好几个月，150+ 行。

### Q: 规则越多越好吗？

不是。规则太多会：
- 影响 AI 生成效果
- 规则之间可能矛盾
- 难以维护

**建议**: 
- 禁止清单控制在 20 条以内
- 定期归纳总结
- 删除已内化的规则

### Q: 要写多个版本的 Skill 吗？

看你需求：

| 场景 | 建议 |
|------|------|
| 只有一种写作风格 | 一个 Skill 足够 |
| 公众号 + 技术博客风格不同 | 分开两个 Skill |
| 工作/个人/学术 | 见 [[03-管理维护/01-多风格切换指南|多风格切换指南]] |

### Q: 原文变了，Skill 要重写了？

不需要。Skill 是你写作 DNA 的记录，不是原文的副本：
- 新增的文章验证/微调 Skill
- 风格转变时，渐进式更新 Skill
- 重大转变时，标注 V2.0 重新开始

## 迭代检查清单

每次迭代后检查：

- [ ] 记录了具体的修改点
- [ ] 总结出了可复用的规律
- [ ] 更新了 Skill 文档
- [ ] 版本号已更新
- [ ] 规则有正反示例（可选但推荐）
- [ ] 禁止清单是否需要归纳总结
