# 多风格切换指南

**场景**: 拥有多个风格 Skill，需要在不同场景间切换  
**典型用例**: 工作文档 vs 个人博客 vs 学术论文

---

## 为什么需要多风格

### 不同场景的要求

| 场景 | 风格特点 | 示例 |
|------|----------|------|
| 工作汇报 | 简洁、数据驱动、客观 | 周报、项目总结 |
| 个人博客 | 口语化、有观点、个人色彩 | 技术随笔、思考 |
| 公众号 | 易读、抓眼球、适合移动端 | 热点评论、干货分享 |
| 学术论文 | 严谨、规范、引用完整 | 研究报告、论文 |

### 单风格的局限

硬用一个风格处理所有场景：
- 工作文档太随意
- 个人博客太拘谨
- 学术论文不够严谨

---

## 多风格架构

### 风格层次

```
基础风格（个人风格）
    ├── 工作文档风格（继承 + 调整）
    ├── 公众号风格（继承 + 调整）
    ├── 技术博客风格（继承 + 调整）
    └── 学术论文风格（继承 + 调整）
```

### 继承原则

子风格继承父风格的：
- 核心用词偏好
- 基础句式习惯
- 个人特色表达

子风格自定义的：
- 场景特有的规则
- 不同的语气强度
- 特定的结构要求

## 切换策略

### 策略1：手动选择（推荐）

**每次任务前明确指定**：

```markdown
本次任务使用【筱可-公众号风格】：

[粘贴公众号风格 Skill]

任务：写一篇关于 AI 工具的公众号文章
```

**优点**: 精确控制，不会混淆  
**缺点**: 每次都要手动粘贴

### 策略2：任务前缀

**在任务描述中声明风格**：

```markdown
【风格：工作文档】

请帮我写一份项目周报...

【我的风格说明】
- 简洁明了
- 数据驱动
- 客观陈述
```

**优点**: 快速指定  
**缺点**: 需要 AI 记住或重新加载风格

### 策略3：场景映射表

**建立任务类型到风格的映射**：

```markdown
## 风格映射表

| 任务关键词 | 使用风格 |
|------------|----------|
| 周报/汇报/总结 | 工作文档 |
| 公众号/推文/发布 | 公众号 |
| 博客/文章/随笔 | 技术博客 |
| 论文/研究/报告 | 学术论文 |
```

使用时：
```markdown
我要写一篇【公众号】文章...
（AI 自动匹配对应风格）
```

## 风格切换检查清单

切换风格前确认：

- [ ] 当前任务适合什么风格？
- [ ] 该风格的 Skill 是否最新版本？
- [ ] 是否需要临时调整风格规则？
- [ ] 输出后是否需要回退到主风格？

## 实际示例

### 示例1：同一主题，不同风格

**主题**: AI 辅助写作

**个人博客风格**：
> 聊聊我用 AI 写作这三个月...
> （口语化、个人经历、观点鲜明）

**工作汇报风格**：
> 关于 AI 辅助写作工具的效果评估...
> （数据、客观、结构清晰）

**公众号风格**：
> 3 个月亲测：AI 写作到底靠不靠谱？
> （抓眼球、易读、互动引导）

### 示例2：多风格协作

**场景**: 翻译一篇外文技术文章，发公众号

**流程**：
1. 使用【翻译解读风格】翻译
2. 使用【公众号风格】重写
3. 使用【content-formatter]] 排版

## 管理多风格的建议

### 命名规范

```
筱可-个人风格-V3.2.md      # 主风格
筱可-公众号风格-V2.1.md     # 公众号（继承个人）
筱可-技术博客-V1.5.md       # 技术博客（继承个人）
```

### 版本同步

父风格重大更新时，子风格需检查：
- 继承的规则是否需要调整
- 特有规则是否与父风格冲突
- 版本号是否需要更新

### 定期审查

每季度审查：
- 哪些风格使用频率低？（考虑合并或删除）
- 哪些风格差异太小？（考虑合并）
- 是否需要新增风格？
