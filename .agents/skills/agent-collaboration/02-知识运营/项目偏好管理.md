# 项目偏好管理

**定位**: 项目级规范的定义、维护与冲突解决  
**作用**: 确保每个项目可以有自己的管理规则，且优先级高于全局规范

---

## 核心原则

### 1. 项目自治原则

每个项目是一个独立的自治单元：
- 可以定义自己的目录结构
- 可以覆盖全局的数据路径规则
- 可以规定特定的执行顺序
- 可以禁用某些全局默认行为

### 2. 优先级规则

当规范冲突时，优先级从高到低：

```
项目 README.md 中的 Project-Specific Rules
    ↓ 覆盖
项目级 Skill (Projects/xxx/.agents/skills/)
    ↓ 覆盖
Areas/My_Persona/ 中的个人偏好
    ↓ 覆盖
AGENTS.md 全局规范
    ↓ 覆盖
全局 Skill (.agents/skills/)
    ↓ 覆盖
Skill 默认行为
```

**说明**:
- 项目级 Skill 可以覆盖全局 Skill 的行为
- 适用于项目特有的处理方式（如特定的数据格式、特定的输出规范）
- 项目级 Skill 不改变全局 Skill 的定义，仅在当前项目生效

### 3. 显式声明原则

项目规范必须**显式声明**才生效：
-  隐含的、未写入文档的"约定"不算数
-  必须写在 `Projects/xxx/README.md` 中
-  必须标明优先级（使用  警告标识）

## 项目特定规范 (Project-Specific Rules)

 **重要**: 本项目的以下规范优先级高于全局 AGENTS.md 规范。

### 1. 数据规范

- 数据位置: `data/`
- 禁止重复下载: 使用本地缓存
- 输出目录: `outputs/`

### 2. 代码规范

- Python 版本: 3.10+
- 必须包含类型提示
- 测试文件位置: `tests/`

### 3. 执行顺序

必须按以下顺序执行：
1. 数据预处理
2. 模型训练
3. 结果评估

### 4. 禁用规则

- 禁用自动格式化（与项目现有风格冲突）

## 项目级 Skill（可选）

如果项目有特定的 AI 协助需求，可以在项目内部创建 Skill：

```
项目名/
├── .agents/
│   └── skills/
│       └── xxx-skill/
│           └── SKILL.md
```

**说明**:
- 项目级 Skill 优先级高于全局 Skill
- 适用于项目特有的操作规范（如特定类型的数据处理、特定的输出格式）
- 应在项目 README 中声明项目级 Skill 的存在

**示例**:
```markdown
## 项目级 Skill

本项目包含以下专用 Skill：
- `.agents/skills/table-visualization/` - 表格可视化规范
  - 定义了本项目特有的配色方案
  - 规定了 PNG 输出尺寸和命名规则
```

**Skill 归档示例**:

当项目探索完成，可以将项目级 Skill 提升为全局 Skill：

1. 原项目: `Projects/2026-02-彩色表格生成方式探索/`
2. 项目级 Skill: `Projects/2026-02-彩色表格生成方式探索/.agents/skills/table-visualization/`
3. 归档项目 → 提升 Skill 到 `.agents/skills/html-table-templates/`
4. 原项目归档到 `Archives/`

这样探索成果得以保留，同时成为可复用的全局 Skill。

## 规范类型模板

### 类型 A: 数据路径覆盖

适用于数据科学项目：

```markdown
### 数据使用规范

- **原始数据位置**: `data/raw/`（不是 `data/`）
- **处理后的数据**: `data/processed/`
- **模型输出**: `models/`
- **实验结果**: `experiments/YYYY-MM-DD/`
- **禁止操作**: 不要删除 `data/raw/` 中的文件
```

### 类型 B: 执行顺序强制

适用于有依赖关系的工作流：

```markdown
### 执行顺序规范

必须严格按以下顺序执行，否则会产生错误：

1. `01_preprocess.py` - 数据预处理
2. `02_train.py` - 模型训练
3. `03_evaluate.py` - 模型评估
4. `04_export.py` - 导出结果

**禁止**: 跳过步骤 2 直接运行步骤 3
```

### 类型 C: 代码修改约束

适用于需要保持兼容性的项目：

```markdown
### 代码修改规范

修改以下文件时必须满足：

- `train.py`: 
  - 必须支持三种模式: `--mode sft/plan_a/plan_b`
  - 数值必须使用 `float()` 转换
  - 不要修改默认超参数

- `config.yaml`:
  - 新增配置项必须提供默认值
  - 不要删除现有配置项
```

### 类型 D: 禁用全局规则

适用于与全局规范冲突的场景：

```markdown
### 禁用规范

以下全局规范在本项目不适用：

-  禁用 "自动格式化 Markdown"（项目使用特定格式）
-  禁用 "强制使用 wikilink"（项目使用标准 Markdown 链接）
-  禁用 "附件必须放在 assets/"（项目使用 img/）
```

## AI 执行检查清单

当 AI 在项目中执行任务时，必须：

1. **首先读取** `Projects/xxx/README.md`
2. **检查是否存在** "项目特定规范" 章节
3. **如有冲突** → 严格遵循项目规范
4. **如无冲突** → 遵循全局 AGENTS.md
5. **不确定时** → 询问用户确认

### 示例对话

**用户**: "处理一下 ERNIE-SQL 项目里的数据"

**AI 内部流程**:
1. 读取 `Projects/ERNIE-SQL/README.md`
2. 发现规范: "禁止重复下载数据，使用 `data/alpaca_zh/train.json`"
3. 检查本地文件存在 → 直接使用
4. 执行处理

**AI 输出**:
> 根据项目 README 中的规范，使用本地已有数据 `data/alpaca_zh/train.json`，跳过下载步骤。

## 项目规范维护

### 何时更新项目规范

| 场景 | 操作 |
|------|------|
| 发现全局规范不适用 | 在 README 中添加禁用声明 |
| 新增数据路径 | 更新 "数据规范" 章节 |
| 调整执行流程 | 更新 "执行顺序" 章节 |
| 发现重复问题 | 添加约束规则 |

### 规范演进流程

```
发现问题 → 记录到 README → 执行验证 → 固化规则
```

## 冲突解决决策树

```
开始任务
    ↓
项目有 README.md?
    ├── 否 → 遵循全局 AGENTS.md
    ↓
    是
    ↓
README 有 Project-Specific Rules?
    ├── 否 → 遵循全局 AGENTS.md
    ↓
    是
    ↓
规范覆盖当前场景?
    ├── 否 → 遵循全局 AGENTS.md
    ↓
    是
    ↓
遵循项目规范 ✓
```

## 示例项目规范

### 示例 1: 数据科学项目

```markdown
## 项目特定规范

### 数据规范
- 原始数据: `data/raw/`（已存在，禁止下载）
- 特征工程输出: `data/features/`
- 模型保存: `models/YYYYMMDD/`
- 实验记录: `experiments/YYYYMMDD_HHMMSS/`

### 执行规范
- 必须按: preprocess → feature → train → evaluate 顺序
- 每个步骤产生的时间戳目录必须保留
```

### 示例 2: Web 开发项目

```markdown
## 项目特定规范

### 代码规范
- 前端代码在 `frontend/`，后端在 `backend/`
- API 路由前缀: `/api/v1/`
- 数据库迁移文件不要手动修改

### 部署规范
- 生产环境使用 `docker-compose.prod.yml`
- 环境变量从 `.env.production` 读取
```
