# 规范索引质量管理

**定位**: AI 协作规范体系索引的维护、评估与优化指南  
**适用场景**: 维护规范导航、优化查找效率、确保索引完整性  
**关联文档**: [[规范冲突检测与处理]]

---

## 概述

规范索引（如 `_入口导航.md`、`SKILL.md` 中的表格）是 AI 和用户查找规范的核心入口。索引质量直接影响：

- **查找效率**: 能否快速定位所需规范
- **执行准确性**: AI 是否能找到正确的规范执行
- **维护成本**: 规范更新时索引同步的难度

---

## 索引质量的四个维度

```
┌─────────────────────────────────────────────────────────────┐
│                    规范索引质量四维度                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 完整性 (Completeness)                                    │
│     ├── 所有规范文档都有入口                                 │
│     ├── 无孤岛规范（不被引用的文档）                         │
│     └── 分类覆盖全面                                         │
│                                                             │
│  2. 准确性 (Accuracy)                                        │
│     ├── 所有链接有效（无 404）                               │
│     ├── 路径正确（无遗留路径）                               │
│     └── 描述与内容一致                                       │
│                                                             │
│  3. 一致性 (Consistency)                                     │
│     ├── 格式统一（表格、列表、wikilink 风格）               │
│     ├── 分类逻辑一致                                         │
│     └── 优先级定义一致                                       │
│                                                             │
│  4. 可用性 (Usability)                                       │
│     ├── 层级清晰（不过深）                                   │
│     ├── 决策路径明确（何时查阅什么）                         │
│     └── 快速入口设计（常用规范优先展示）                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 索引质量检测方法

### 检测 1：完整性检测

**检测目标**: 发现孤岛规范和遗漏入口

**检测步骤**:
```
1. 遍历目录，列出所有 .md 文件
2. 构建引用图，找出入度为 0 的节点（无文档引用它）
3. 检查这些节点是否应该被索引
4. 分类处理：
   ├── 应该被索引 → 添加到导航
   ├── 不应该被索引（如模板、样例）→ 标记为例外
   └── 过时文档 → 归档或删除
```

**检测频率**: 每月一次

---

### 检测 2：准确性检测

**检测目标**: 发现无效链接和路径错误

**检测步骤**:
```
1. 提取所有 wikilink：[[...]]
2. 解析链接目标
3. 验证目标文件是否存在
4. 分类处理：
   ├── 内部链接无效 → 修复路径
   ├── 外部链接无效 → 检查是否应为占位符
   └── 锚点链接 → 验证锚点是否存在
```

**检测频率**: 每次规范重构后

---

### 检测 3：一致性检测

**检测目标**: 确保格式和逻辑统一

**检测项**:

| 检测项 | 标准 | 示例 |
|--------|------|------|
| wikilink 格式 | 统一使用别名 | `[[路径\|别名]]` |
| 表格列头 | 统一命名 | 文档/内容/何时查阅 |
| 分类层级 | 不超过 3 层 | 基础/操作/工具 |
| 描述风格 | 统一风格 | 动名词短语 |

**检测频率**: 每季度一次

---

### 检测 4：可用性检测

**检测目标**: 评估实际使用体验

**检测方法**:
```
1. 模拟查找场景
   - "我要写笔记，从入口开始需要几步找到写作规范？"
   - "我要修改规范，能否快速找到规范体系管理？"

2. 统计关键指标
   - 平均查找步数（应 < 3 步）
   - 核心规范直达率（应 > 80%）
   - 决策路径清晰度（是否有明确的"何时查阅"）

3. 收集使用反馈
   - AI 执行时是否找不到规范
   - 用户是否经常询问"xxx 在哪里"
```

**检测频率**: 按需，或发现查找困难时

## 索引质量问题分级

| 级别 | 问题 | 影响 | 处理时限 |
|------|------|------|----------|
| **致命** | 核心规范无法通过索引找到 | AI 无法执行关键任务 | 立即 |
| **严重** | 大量链接失效（>20%） | 查找效率大幅下降 | 24小时 |
| **中等** | 孤岛规范、分类混乱 | 部分规范难找 | 本周 |
| **轻微** | 格式不统一、描述不准确 | 美观度下降 | 下次重构 |

## 索引修复流程

### 流程图

```
发现索引问题
    ↓
判定维度（完整性/准确性/一致性/可用性）
    ↓
判定级别（致命/严重/中等/轻微）
    ↓
执行修复
    ├── 完整性问题 → 添加遗漏入口
    ├── 准确性问题 → 修复链接路径
    ├── 一致性问题 → 统一格式标准
    └── 可用性问题 → 优化层级结构
    ↓
验证修复
    ├── 重新运行检测
    ├── 模拟查找场景
    └── 确认无新问题引入
    ↓
更新变更日志
```

## 具体修复方法

### 方法 1：添加遗漏入口（完整性）

**场景**: 发现孤岛规范

**处理步骤**:
1. **评估规范价值**
   ```
   这个规范是否仍需要？
   ├── 是 → 继续
   └── 否 → 归档或删除，结束
   ```

2. **确定分类位置**
   ```
   根据规范内容确定层级：
   ├── 基础规范（所有 AI 必读）→ 第一层
   ├── 开发指南（开发者专用）→ 第二层
   ├── 知识运营（管理维护）→ 第三层
   └── 其他 → 根据关联度决定
   ```

3. **添加索引条目**
   ```markdown
   | [[路径|别名]] | 简短描述（15字内） | 何时查阅 |
   ```

4. **建立反向链接**
   - 在规范文档底部添加"关联文档"章节
   - 链接回索引文档

**示例**:
```markdown
<!-- 在 _入口导航.md 中添加 -->
| [[Git 管理规范/_入口导航|Git 管理规范]] | Git 分支策略、提交规范 | 进行 Git 版本控制时 |
```

---

### 方法 2：修复失效链接（准确性）

**场景**: 链接指向不存在或错误路径

**处理步骤**:
1. **定位所有失效链接**
   ```bash
   # 使用脚本扫描
   python detect_invalid_links.py --skill agent-collaboration
   ```

2. **分类处理**
   ```
   失效类型
   ├── 旧路径引用 → 批量替换为新路径
   ├── 文件已删除 → 移除链接或指向替代文档
   ├── 拼写错误 → 修正拼写
   └── 外部链接 → 验证可用性，必要时标注
   ```

3. **批量修复**
   ```python
   # 示例：批量替换旧路径
   replacements = {
       "07-规范体系管理": "01-开发指南/规范体系管理",
       "00-入口导航": "00-基础规范/_入口导航"
   }
   ```

4. **验证修复**
   - 重新运行检测脚本
   - 确认无遗漏

---

### 方法 3：统一格式标准（一致性）

**场景**: 索引格式混乱

**标准规范**:

**wikilink 格式**:
```markdown
<!-- 推荐 -->
[[路径/文件名|显示别名]]

<!-- 不推荐 -->
[[文件名]]（无别名，路径不清晰）
[[路径/文件名]]（无别名，可读性差）
```

**表格格式**:
```markdown
| 文档 | 内容 | 何时查阅 |
|------|------|----------|
| [[路径|别名]] | 简短描述 | 使用场景 |
```

**分类层级**:
```markdown
## 第一层：基础规范（必读）
## 第二层：操作手册（按需查阅）
## 第三层：工具参考（偶尔查阅）
```

---

### 方法 4：优化层级结构（可用性）

**场景**: 查找路径过长或逻辑混乱

**优化原则**:

1. **3 步原则**: 从入口到任何规范不超过 3 次点击
   ```
   _入口导航 → 第一层 → 目标规范 ✓（2步）
   _入口导航 → 第一层 → 第二层 → 目标规范 ✓（3步）
   _入口导航 → ... → ... → ... → 目标规范 ✗（超过3步）
   ```

2. **决策支持**: 每个条目都有"何时查阅"
   ```markdown
   | [[写作规范]] | 笔记写作标准 | 每次创建或编辑笔记时 |
   ```

3. **常用优先**: 高频使用的规范放在前面
   ```markdown
   <!-- 基础规范按使用频率排序 -->
   1. 写作规范（最常用）
   2. PARA方法论（常用）
   3. 附件管理规范（偶尔）
   ```

4. **快速决策树**: 复杂场景提供决策路径
   ```markdown
   不确定从哪里开始？
   ├── 创建新笔记？→ 查看 PARA方法论
   ├── 修改规范？→ 查看 规范体系管理
   └── 处理问题？→ 查看 规范冲突检测与处理
   ```

## 索引维护机制

### 定期审查

| 审查类型 | 频率 | 负责人 | 内容 |
|----------|------|--------|------|
| 完整性审查 | 每月 | AI / 用户 | 检查孤岛规范 |
| 准确性审查 | 重构后 | AI / 用户 | 检查链接有效性 |
| 一致性审查 | 每季度 | 用户 | 检查格式统一 |
| 可用性审查 | 按需 | 用户 | 评估查找体验 |

### 变更同步流程

当新增、修改、删除规范时：

```
规范变更
    ↓
是否需要更新索引？
    ├── 是 → 更新所有相关索引文档
    │       ├── _入口导航.md
    │       ├── SKILL.md
    │       └── 上层索引（如有）
    ↓
运行检测验证
    ↓
更新变更日志
```

### 质量标准清单

索引文档发布前检查：

- [ ] 所有规范都有入口（无孤岛）
- [ ] 所有链接有效（无 404）
- [ ] 格式统一（wikilink 带别名）
- [ ] 分类逻辑清晰（不超过 3 层）
- [ ] 每个条目有"何时查阅"
- [ ] 核心规范 3 步内可达

## 使用示例

**场景**: 新增一个规范文档，如何更新索引？

**操作步骤**:
1. 创建规范文档 `01-开发指南/新规范.md`
2. 评估分类：属于开发指南，应放在第二层
3. 更新 `_入口导航.md` 第二层表格
   ```markdown
   | [[../01-开发指南/新规范|新规范]] | 简短描述 | 使用场景 |
   ```
4. 更新 `SKILL.md` 对应章节
5. 在新规范底部添加关联文档
   ```markdown
   ## 关联文档
   - [[../00-基础规范/_入口导航|入口导航]]: 规范总目录
   ```
6. 运行完整性检测，确认无孤岛
7. 创建变更日志记录
