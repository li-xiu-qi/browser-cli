# 长时间任务防超时技巧

**定位**: Windows 环境下长时间运行任务的防超时解决方案
**适用场景**: 浏览器自动化、Docker 构建、批量数据处理等耗时操作

---

## 问题背景

Windows PowerShell/Shell 工具默认有 **60-300 秒**的执行超时限制。长时间任务会被强制中断，报错类似：

```
Command failed with exit code: 1
The operation has timed out.
```

### 超时类型辨析

| 类型 | 原因 | 解决方案 |
|------|------|----------|
| **工具超时** | Shell 命令有默认 timeout 限制 | 增加 `timeout` 参数 |
| **终端断开** | 关闭窗口/SSH 后进程被杀 | 使用后台运行 |
| **页面加载** | 网页需要长时间加载或人工干预 | 延长等待 + 暂停机制 |

---

## 解决方案

### 1. 增加 Timeout 参数（Shell 工具）

适用于**单次长时间命令**：

```javascript
// 在 Kimi 的 Shell 工具调用时设置（单位：秒）
Shell({
    command: "node web_clipper.js <url>",
    timeout: 300  // 5 分钟，默认是 60 秒
})
```

**常用 timeout 设置参考**:

| 任务类型 | 建议超时 |
|---------|---------|
| 简单网页抓取 | 60 秒 |
| 复杂页面 + 图片加载 | 180-300 秒 |
| Docker 构建 | 600 秒 |
| 需要人工干预的浏览器任务 | 无限制 / 手动暂停 |

---

### 2. Windows 后台运行（替代 Linux nohup）

Linux 的 `nohup` 在 Windows 下无直接对应，使用以下替代方案：

#### 方案 A: Start-Process（推荐）

后台运行并隐藏窗口，输出到日志文件：

```powershell
# 后台运行，输出到日志
Start-Process -FilePath "node" -ArgumentList "web_clipper.js", "<url>" -RedirectStandardOutput "clip.log" -RedirectStandardError "error.log" -WindowStyle Hidden

# 监控进度：查看日志
Get-Content clip.log -Tail 20

# 检查进程
Get-Process | Where-Object { $_.ProcessName -like "*node*" }
```

#### 方案 B: Start-Job

```powershell
# 启动后台任务
Start-Job -ScriptBlock { node web_clipper.js <url> } -Name web-clip

# 查看任务状态
Get-Job -Name web-clip

# 获取输出
Receive-Job -Name web-clip -Keep
```

#### 方案 C: CMD 的 start /B（最简洁）

```powershell
# 后台执行，输出到日志
cmd /c "start /B node web_clipper.js <url> > clip.log 2>&1"

# 实时查看进度
tail -f clip.log  # 或使用 Get-Content -Wait
```

### 3. 浏览器自动化专用技巧

#### 延长页面加载等待

```javascript
// 从默认的 30 秒延长到 2 分钟
await page.goto(url, { 
    waitUntil: 'networkidle',  // 等待网络空闲
    timeout: 120000  // 120 秒
});

// 等待页面稳定后再操作
await page.waitForTimeout(10000);  // 额外等待 10 秒
```

#### 人工干预暂停（遇到验证时）

```javascript
// 检测验证页面，暂停让用户处理
if (fullHtml.includes("环境异常") || fullHtml.includes("请完成验证")) {
    console.log("检测到验证，请在浏览器中完成验证...");
    console.log("验证完成后按回车继续...");
    
    // 暂停等待用户输入
    const readline = require('readline');
    const rl = readline.createInterface({
        input: process.stdin,
        output: process.stdout
    });
    
    await new Promise(resolve => rl.question("按回车继续...", () => {
        rl.close();
        resolve();
    }));
}
```

#### Playwright 的 page.pause()

```javascript
// 暂停，保持浏览器打开供用户操作
await page.pause();

// 或者使用 Promise 等待信号
await new Promise(resolve => {
    console.log("请在浏览器中完成操作，然后按 Ctrl+C 继续...");
    process.once('SIGINT', resolve);
});
```

### 4. 脚本文件方式（最稳定）

创建 `.ps1` 脚本文件，然后后台执行：

```powershell
# clip_article.ps1 内容
$ProgressPreference = 'SilentlyContinue'
$url = $args[0]
$outputDir = $args[1] ?? "cliper_datas"

Write-Host "开始抓取: $url"
node web_clipper.js $url $outputDir *>&1 | Out-File -FilePath "clip.log" -Append
Write-Host "抓取完成！"
```

执行：
```powershell
Start-Process powershell -ArgumentList "-File", "clip_article.ps1", "<url>", "output" -WindowStyle Hidden
```

## 实践建议

| 场景 | 推荐方案 | 说明 |
|------|---------|------|
| 微信文章抓取（需验证） | `timeout: 300` + 人工暂停 | 微信有反爬，可能需要人工过验证 |
| Docker 构建 | `timeout: 600` + `Start-Process` | 构建时间长，需要日志 |
| 批量数据处理 | `Start-Job` | 多任务并行 |
| 简单后台任务 | `cmd /c "start /B"` | 快速简洁 |
| 复杂工作流 | `.ps1` 脚本文件 | 可维护性强 |

---

## 注意事项

1. **后台任务无法直接交互** - 需要输入时不能用后台方式
2. **日志文件会增长** - 长时间任务注意磁盘空间
3. **检查任务状态** - 使用 `Get-Job` 或 `Get-Process` 确认是否完成
4. **资源占用** - 后台任务仍占用 CPU/内存，注意系统负载
5. **浏览器排他性** - Chromium 同一时间只能有一个进程访问同一 user-data-dir
