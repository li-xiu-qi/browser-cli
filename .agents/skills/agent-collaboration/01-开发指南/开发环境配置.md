# 开发环境配置

**定位**: 项目开发环境的统一配置规范  
**适用范围**: Python 和 Node.js 环境

---

## Node.js 环境

本项目使用 **npm** 管理统一的 Node.js 环境，位于项目根目录 `node_modules/`。

### 环境信息

| 项目 | 值 |
|------|-----|
| Node.js 版本 | 18+ (推荐 20+) |
| 管理工具 | npm |
| 依赖文件 | `package.json` |
| 已安装依赖 | `playwright` (用于 ECharts/Draw.io 渲染) |

### 使用方法

**安装依赖**（在项目根目录）:
```powershell
npm install
```

**运行脚本**:
```powershell
# 使用 text-diagram skill 统一渲染
npm run diagram:render diagram.dot
npm run diagram:batch src/

# 传统方式（保留兼容）
npm run echarts:svg Areas/文本制图/guides/examples/src/chart.json
npm run drawio:convert Areas/文本制图/guides/examples/src/diagram.drawio
```

**添加新依赖**:
```powershell
npm install package-name
```

### 统一使用原则

- **所有 Node.js 脚本必须使用根目录的 `node_modules`**
- 不要在子目录创建独立的 `package.json` 和 `node_modules`
- 各 skill 中的 Node.js 工具应使用根目录的依赖
- 统一通过根目录的 `package.json` 管理依赖

## Python 环境

本项目使用 **uv** 管理统一的 Python 环境，位于项目根目录 `.venv/`。

### 环境信息

| 项目 | 值 |
|------|-----|
| Python 版本 | 3.12 |
| 管理工具 | uv |
| 环境路径 | `.venv/` |
| 激活命令 | `.venv\Scripts\activate` (Windows) |
| 依赖文件 | `pyproject.toml` |

### 使用方法

**运行 Python 脚本**:
```powershell
uv run python script.py
```

**安装依赖**:
```powershell
uv add package-name
```

**安装开发依赖**:
```powershell
uv add --dev package-name
```

**激活环境**（需要时）:
```powershell
.venv\Scripts\activate
```

### 统一使用原则

- 所有 Python 脚本**必须**使用此环境运行
- 新依赖**必须**通过 `uv add` 安装到 `pyproject.toml`
- 不要在项目内创建其他虚拟环境
- 各 skill 中的 Python 工具应检测并使用此环境

### 工具集成

项目内 Python 工具（如 OCR、数据处理等）应优先使用：
```powershell
uv run python .agents/tools/xxx/main.py
```

或在代码中检测环境：
```python
import sys
if sys.prefix.endswith('.venv'):
    # 在 uv 环境中
    pass
```

## 浏览器用户数据目录规范

本项目使用 Playwright 进行浏览器自动化时，统一使用以下目录保存浏览器用户数据：

### 目录位置

```
.agents/browser_user_data/
```

### 规范说明

| 项目 | 说明 |
|------|------|
| **用途** | 存储 Playwright 浏览器持久化数据 |
| **包含** | Cookies、LocalStorage、IndexedDB、缓存、登录状态 |
| **共享** | 所有需要浏览器登录的 Skill 共用同一目录 |
| **安全** | 包含敏感登录凭证，已加入 `.gitignore` |

### 使用方法

在 Playwright 中使用 `launchPersistentContext`：

```python
from playwright.sync_api import sync_playwright
from pathlib import Path

# 统一用户数据目录
user_data_dir = Path(".agents/browser_user_data")

with sync_playwright() as p:
    browser = p.chromium.launch_persistent_context(
        user_data_dir=str(user_data_dir),  # 所有 Skill 共享同一浏览器数据目录
        headless=False,
        args=['--disable-blink-features=AutomationControlled']
    )
    # ... 后续操作
```

### 设计优势

1. **登录一次，多处使用** - 登录状态被所有 Skill 共享
2. **环境一致性** - 避免多个浏览器实例的冲突
3. **持久化** - 关闭浏览器后登录状态不会丢失
4. **排他性** - Chromium 同一时间只允许一个进程访问，天然防止并发问题

### 注意事项

- **不要上传** - 该目录包含敏感信息，已加入 `.gitignore`
- **定期清理** - 缓存会随时间增长，可定期清理释放空间
- **排他访问** - 同一时间只能有一个脚本使用该目录

### 配置位置

```
.vscode/settings.json

### 目录显示规则

| 分类 | 目录 | 说明 |
|------|------|------|
| **显示** | `.agents` | AI 协作规范和 Skill 目录 |
| | `.github` | GitHub 工作流和配置 |
| | `.vscode` | VS Code 本身配置 |
| **隐藏** | `.obsidian` | Obsidian 配置 |
| | `.pandoc` | Pandoc 转换工具目录 |
| | `.venv` | Python 虚拟环境 |

        ".qoder": true,
        ".qwen": true,
        ".roo": true,
        ".smart-env": true,
        ".trae": true,
        ".windsurf": true,

        // 其他配置目录 - 隐藏
        ".obsidian": true,
        ".pandoc": true,
        ".venv": true
    }
}
```

### 重要说明

**VS Code 隐藏 ≠ 命令行隐藏**
- `files.exclude` 只影响 VS Code 文件资源管理器
- `dir`、`ls` 等命令仍能看到所有文件
- 需要查看隐藏目录时，直接命令行操作即可

### 维护流程

1. **新增 AI 工具**：将新工具的配置目录加入 `files.exclude`
2. **临时显示**：将对应目录值改为 `false` 或删除该行

## Windows 命令超时处理

### 问题背景

Windows PowerShell/Shell 工具默认有 **60-300 秒**的执行超时限制。长时间任务（如 Docker 构建、大数据处理）会被强制中断。

### 超时类型辨析

| 类型 | 原因 | 解决方案 |
|------|------|----------|
| **工具超时** | Shell 命令有默认 timeout 限制 | 增加 `timeout` 参数 |
| **终端断开** | 关闭窗口/SSH 后进程被杀 | 使用后台运行 |

### 解决方案

#### 1. 增加 Timeout 参数（推荐）

适用于**单次长时间命令**，在 Shell 工具调用时设置：

```powershell
# Shell 工具调用时设置 timeout（单位：秒）
Shell({
    command: "docker build -t myimage .",
    timeout: 600  # 10 分钟
})
```

#### 2. Windows 后台运行（替代 nohup）

Linux 的 `nohup` 在 Windows 下无直接对应，使用以下替代方案：

**方案 A: Start-Process（推荐）**
```powershell
# 后台运行并隐藏窗口，输出到日志文件
Start-Process -FilePath "docker" -ArgumentList "build", "-t", "myimage", "." -RedirectStandardOutput "build.log" -RedirectStandardError "error.log" -WindowStyle Hidden

# 监控进度：查看日志
Get-Content build.log -Tail 20

# 检查进程
Get-Process | Where-Object { $_.ProcessName -like "*docker*" }
```

**方案 B: Start-Job**
```powershell
# 启动后台任务
Start-Job -ScriptBlock { docker build -t myimage . } -Name docker-build

# 查看任务状态
Get-Job -Name docker-build

# 获取输出
Receive-Job -Name docker-build -Keep
```

**方案 C: CMD 的 start /B**
```powershell
# 最简洁的后台执行方式
cmd /c "start /B docker build -t myimage . > build.log 2>&1"

# 查看进度
tail -f build.log  # 或使用 Get-Content -Wait
```

#### 3. 脚本文件方式（最稳定）

创建 `.ps1` 脚本文件，然后后台执行：

```powershell
# build.ps1 内容
$ProgressPreference = 'SilentlyContinue'
docker build -t myimage . *>&1 | Out-File -FilePath build.log -Append
Write-Host "构建完成！"
```

执行：
```powershell
Start-Process powershell -ArgumentList "-File", "build.ps1" -WindowStyle Hidden
```

### 实践建议

| 场景 | 推荐方案 | 说明 |
|------|---------|------|
| Docker 构建 | `timeout: 600` + `Start-Process` | 构建时间长，需要日志 |
| 批量数据处理 | `Start-Job` | 多任务并行 |
| 简单后台任务 | `cmd /c "start /B"` | 快速简洁 |
| 复杂工作流 | `.ps1` 脚本文件 | 可维护性强 |

### 注意事项

1. **后台任务无法直接交互** - 需要输入时不能用后台方式
2. **日志文件会增长** - 长时间任务注意磁盘空间
3. **检查任务状态** - 使用 `Get-Job` 或 `Get-Process` 确认是否完成
4. **资源占用** - 后台任务仍占用 CPU/内存，注意系统负载
