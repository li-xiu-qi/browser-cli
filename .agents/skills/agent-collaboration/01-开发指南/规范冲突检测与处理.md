# 规范冲突检测与处理

**定位**: AI 协作规范体系的冲突识别、检测与解决指南  
**适用场景**: 维护规范体系、发现规范问题、执行规范修复  
**状态**: 已验证（基于 agent-collaboration skill 实践）

---

## 概述

随着规范体系持续演进，规范数量增加必然带来**内在张力**。本指南提供系统化的冲突检测与解决方法论，确保 AI 协作规范体系保持健康、一致、可维护。

---

| 场景 | 操作 |
|------|------|

## 冲突类型（五种）

### 类型 1：重复定义冲突（内容冗余）

**特征**: 同一规则在多处定义，内容可能不一致

**示例**: 
- SKILL.md 语言风格速查 vs 写作规范.md 完整定义
- 知识库管理.md 中的写作规范章节 vs 写作规范.md 全文

**处理原则**: 
- SKILL.md 保留**速查表**（极简版），链接到完整规范
- 详细内容只在**单一文档**中定义
- 其他文档只放**链接**，不放重复内容

---

### 类型 2：递归引用（知识库导航特性）

**特征**: 规范 A 引用 B，B 引用 C，C 又引用 A

**关键认知**: 知识库中的**双向链接是 Feature，不是 Bug**

- **程序循环引用** → 导致死循环（必须打破）
- **知识库双向链接** → 提升导航效率（保持现状）

**判断标准**: 只要用户每次跳转都能获得新信息，就不是问题。

---

### 类型 3：优先级表述不一致

**特征**: 不同文档中优先级层级定义不一致

**示例**:
- 文档 A：项目 README > AGENTS.md > SKILL.md（3层）
- 文档 B：项目 README > 项目级 Skill > Areas/My_Persona/ > AGENTS.md > 全局 Skill > 默认（6层）

**标准优先级架构**（6层）:

| 优先级 | 规范来源 |
|--------|----------|
| 1 (最高) | 项目 README.md |
| 2 | 项目级 Skill (Projects/xxx/.agents/skills/) |
| 3 | Areas/My_Persona/ (个人偏好) |
| 4 | AGENTS.md (全局规范) |
| 5 | 全局 Skill (.agents/skills/) |
| 6 | Skill 默认行为 |

---

### 类型 4：元规则与执行规则冲突

**特征**: 元规则被实际规范违反

**示例**:
- 元规则："每类信息只应存在于一个位置"
- 实际：写作规范同时存在于 SKILL.md、写作规范.md、知识库管理.md

**处理方案**:
- 明确区分**速查**与**完整**的定位差异
- SKILL.md 的速查表不算"重复定义"，是**导航辅助**
- 但完整复制需要移除，改为链接

---

### 类型 5：遗留路径引用

**特征**: 规范重构后，旧路径引用未更新

**示例**:
- 旧：`07-规范体系管理.md`
- 新：`01-开发指南/规范体系管理.md`

**修复方法**:
1. 全局搜索旧路径引用
2. 批量替换为新路径
3. 使用带别名的 wikilink：`[[路径|别名]]`

---

## 检测方法（四步法）

### Step 1: 静态扫描

| 检测项 | 方法 |
|--------|------|
| 提取所有 wikilink | 遍历 .md 文件，解析 `[[...]]` 格式 |
| 路径有效性检查 | 验证链接目标文件是否存在 |
| 重复内容检测 | 文本相似度分析（>90% 视为重复） |
| 优先级一致性 | 提取各文档中的优先级定义，对比差异 |

### Step 2: 引用图分析

| 检测项 | 方法 |
|--------|------|
| 构建引用有向图 | 节点=文档，边=引用关系 |
| 识别核心枢纽 | 连接度最高的节点（如 SKILL.md、_入口导航） |
| 发现孤岛规范 | 无入边也无出边的节点 |
| ⚠️ 循环引用 | 在此不是错误，是知识库导航特性 |

### Step 3: 语义一致性检查

| 检测项 | 方法 |
|--------|------|
| 概念表述一致性 | "附件" vs "assets" vs "附件目录" |
| 矛盾指令检测 | "必须" vs "禁止" 的冲突 |
| 管辖范围界定 | 多个规范声称管辖同一领域 |

### Step 4: 执行验证

| 检测项 | 方法 |
|--------|------|
| 模拟 AI 执行流程 | 从入口开始，验证能否找到目标规范 |
| 检查元规则遵守 | 如"单一事实来源"是否被违反 |
| 验证优先级执行 | 冲突场景下是否能正确选择 |

## 处理流程

```
检测发现冲突
    ↓
判定级别
    ├── 致命（导致无法执行）→ 立即修复
    ├── 严重（可能导致错误）→ 本周修复
    └── 轻微/中等 → 记录并计划修复
    ↓
执行修复（见下节「具体处理方法」）
    ↓
验证修复效果
    ↓
更新变更日志（创建 YYYYMMDD-简述.md）
```

## 具体处理方法

### 1. 缺失文件处理

**场景**: 规范 A 引用规范 B，但规范 B 不存在

**处理决策树**:
```
评估缺失文件的重要性
    ├── 核心规范（被多处引用）→ 创建文件
    ├── 非核心但有价值 → 创建文件或移除引用
    └── 过时引用 → 移除引用
```

---

### 2. 路径引用修复

**场景**: 规范合并/重构后，旧路径失效

**处理步骤**:
1. 全局搜索旧路径引用
2. 批量替换为新路径
3. 统一格式：`[[路径|别名]]`
4. 更新变更日志记录重构历史

**示例**:
```
旧：[[../07-规范体系管理]]
新：[[../01-开发指南/规范体系管理|规范体系管理]]
```

---

### 3. 优先级统一

**场景**: 不同文档中优先级层级定义不一致

**处理步骤**:
1. 确定最完整的定义（通常是 6 层架构）
2. 以之为基准，同步到其他所有文档
3. 使用表格形式清晰展示

---

### 4. 孤岛规范连接

**场景**: 规范存在但未被任何其他规范引用

**处理步骤**:
1. 评估价值：该规范是否仍需要？
2. 添加入口：在核心导航文档（如 `_入口导航.md`）中添加链接
3. 建立关联：确保规范之间形成连通图

---

### 5. 重复内容处理

**场景**: 同一内容在多个文档中重复定义

**处理决策树**:
```
判定重复类型
    ├── SKILL.md 速查表 + 详细规范 → 保留，明确分工
    ├── 无意识复制粘贴 → 移除重复，改为链接
    └── 内容相似但目的不同 → 重写，明确差异化定位
```

---

## 规范重复的处理原则

### 三种重复场景

| 场景 | 说明 | 处理方式 |
|------|------|----------|
| **速查与详查** | SKILL.md 极简版 + 专门文档完整版 | ✅ 保留，明确分工 |
| **无意识复制** | 编辑时复制粘贴未清理 | ❌ 移除，改为链接 |
| **边界模糊** | 两个规范都涉及同一领域 | 🔧 重构，明确边界 |

### 重复检测标准

- **文本相似度 > 90%** → 视为重复，需要处理
- **文本相似度 50-90%** → 可能是合理分工，需人工判定
- **文本相似度 < 50%** → 通常不是重复

---

## 冲突分级

| 级别 | 定义 | 处理时限 | 示例 |
|------|------|----------|------|
| **致命** | 导致执行失败或严重错误 | 立即 | 关键文件缺失、无效核心链接 |
| **严重** | 可能导致执行不一致 | 24小时 | 优先级定义矛盾、路径错误 |
| **中等** | 造成困惑但可执行 | 本周 | 表述不一致、孤岛规范 |
| **轻微** | 影响美观或冗余 | 下次重构 | 格式不统一、轻微重复 |

---

## 解决原则

| 原则 | 定义 | 实施示例 |
|------|------|----------|
| **单一事实来源** | 每类信息只存一处，其他都是链接 | 写作规范只在 `写作规范.md`，SKILL.md 只做链接引用 |
| **显性覆盖声明** | 覆盖全局规范必须显式声明 | 项目 README 必须写"本规范优先级高于..." |
| **版本锚定** | 引用规范时注明版本 | "遵循 [[写作规范]] (v2.1)" |
| **向下兼容** | 修改规范时，旧引用仍然有效 | 重命名文件时保留别名或重定向 |
| **修复留痕** | 所有修复必须记录变更日志 | 创建 `YYYYMMDD-简述.md` 记录修复详情 |
